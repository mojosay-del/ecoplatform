## 1. Назначение раздела

Настоящий документ фиксирует полную продуктовую спецификацию файлового хранилища ЭкоПлатформы. Документ является единым источником истины по составу хранимых файлов, по правилам загрузки, доступа, замены и удаления файлов, по уровням доступа, по лимитам на размер и количество файлов в каждом продуктовом контексте, по интеграции файлового хранилища с подсистемами платформы.

Документ определяет:
- продуктовую модель файлового хранилища как единого сервиса для всех типов файлов платформы;
- сущности файлового хранилища, общие для всех потребителей;
- уровни доступа к файлам и правила формирования ссылок;
- каталог продуктовых потребителей с указанием уровней доступа и характера контента;
- глобальные лимиты по типам файлов и сводную таблицу лимитов по потребителям;
- жизненный цикл файла — загрузка, антивирусная проверка, хранение, замена, удаление;
- правила удаления файлов при удалении компании и при удалении сущностей;
- правила журналирования операций с файлами;
- открытые вопросы и ограничения первой версии.

Документ описывает только продуктовый слой. Архитектурные вопросы (выбор конкретного S3-совместимого провайдера, конкретный антивирусный движок, организация bucket-ов, IAM-роли, лайфсайкл-политики, пресеты обработки изображений, бэкапы, мониторинг) фиксируются в `08-architecture/` и в архитектурном файле, посвящённом storage. Настоящий документ задаёт продуктовый контракт, на который опирается архитектура.

При расхождении между настоящим документом и архитектурными документами приоритет имеет настоящий документ в части продуктового поведения и архитектурный документ в части технических деталей реализации.

## 2. Принципы

### 2.1. Единый сервис файлового хранилища
Все файлы платформы хранятся в едином сервисе файлового хранилища. Отдельные хранилища для разных категорий файлов (отдельное для биллинга, отдельное для торговой площадки и так далее) не предусматриваются. Различия между категориями файлов выражаются через атрибуты файла: уровень доступа, привязку к сущности, политику удаления, лимиты.

Единство сервиса обеспечивает единые правила загрузки, единый антивирусный контур, единый интерфейс доступа, единое журналирование.

### 2.2. Продуктовый слой и архитектурный слой
Настоящий документ — продуктовый слой. Он фиксирует, какие файлы существуют в платформе, кто их загружает, кто видит, какие лимиты применяются, как файл живёт в продуктовом смысле.

Архитектурный слой фиксирует, какой именно S3-совместимый провайдер используется, как организованы bucket-ы, как настроены лайфсайкл-политики, как генерируются подписанные ссылки на уровне инфраструктуры, какой антивирусный движок применяется. Архитектурный слой опирается на продуктовый контракт настоящего документа.

При наличии конфликта приоритет имеет продуктовый слой в вопросах поведения, архитектурный слой — в вопросах реализации.

### 2.3. Уровни доступа к файлам
Каждый файл в платформе имеет один из пяти уровней доступа:

- `public` — доступен по прямому URL без авторизации;
- `authenticated` — требует авторизации на платформе, доступен любому авторизованному пользователю;
- `company-private` — доступен только пользователям конкретной компании в соответствии с ролевой моделью этой компании;
- `conversation-private` — доступен только сторонам конкретного канала переписки;
- `platform-private` — доступен только платформенным сотрудникам в соответствии с ролевой моделью админ-панели.

Уровень доступа фиксируется в момент загрузки файла и не меняется в течение жизни файла. При изменении контекста использования (например, при принятии предложения покупателем на торговой площадке точный адрес партии раскрывается, но фотографии партии становились публичными при публикации объявления и не меняют уровень доступа) изменение видимости информации обеспечивается на уровне продуктовых сценариев, а не на уровне файла.

Подробно — раздел 4.

### 2.4. Подписанные временные ссылки для приватных файлов
Файлы уровней `authenticated`, `company-private`, `conversation-private`, `platform-private` доступны только через временные подписанные ссылки. Подписанная ссылка имеет ограниченный срок жизни и выдаётся бэкендом после проверки прав на конкретный файл.

В первой версии срок жизни подписанной ссылки — пятнадцать минут с момента выдачи. Срок является параметром платформенной конфигурации (`07-platform/admin-panel.md`, раздел 18.1).

Прямой публичный URL для приватных файлов невозможен по определению — приватные файлы хранятся в bucket-ах с ограниченным доступом и доступны только через подписанную ссылку. Архитектурные детали — в архитектурном документе storage.

Подробно — раздел 8.

### 2.5. Антивирусная проверка обязательна для всех загружаемых файлов
Все файлы, загружаемые в платформу, проходят антивирусную проверку до того, как становятся доступны конечным потребителям. Файл, не прошедший проверку, в платформе не сохраняется и в каталоге не появляется. Файл с неоднозначным результатом проверки рассматривается как не прошедший.

Антивирусная проверка распространяется на все категории файлов без исключения, включая файлы, загружаемые платформенными сотрудниками. Файлы, генерируемые платформой автоматически (финансовые документы — счета, акты, чеки), антивирусной проверке не подлежат, поскольку формируются доверенным процессом платформы.

Подробно — раздел 7.

### 2.6. S3-совместимое хранилище без CDN в первой версии
Платформа использует S3-совместимое хранилище для всех типов файлов (`08-architecture/tech-stack.md`, раздел 1.6). Конкретный провайдер фиксируется в архитектурном документе.

В первой версии CDN-кеширование публичных файлов не используется. Публичные файлы отдаются напрямую с S3-эндпоинта. Подключение CDN — открытый вопрос для последующих версий, рассматривается при росте нагрузки на публичные файлы (раздел 12.1).

## 3. Сущности

### 3.1. Файл (StoredFile)
Сущность хранимого файла. Один файл — одна запись `StoredFile`.

Ключевые атрибуты:
- идентификатор;
- имя файла, заданное при загрузке (отображается пользователю при скачивании);
- тип содержимого (MIME-тип);
- размер в байтах;
- хеш содержимого (для контроля целостности и дедупликации);
- ссылка на физическое расположение в S3-совместимом хранилище;
- уровень доступа (`public`, `authenticated`, `company-private`, `conversation-private`, `platform-private`);
- категория потребителя (раздел 5);
- идентификатор владельца — компании или платформенного сотрудника (для приватных файлов);
- идентификатор автора загрузки — пользователя или платформенного сотрудника;
- дата загрузки;
- результат антивирусной проверки (`clean`, `infected`, `error`);
- дата антивирусной проверки;
- состояние (`uploading`, `scanning`, `available`, `failed`, `deleted`);
- дата перевода в `deleted` (если применимо).

Состав полей подлежит окончательной фиксации в `08-architecture/data-model.md`.

### 3.2. Связь файла с сущностью (FileReference)
Связь хранимого файла с конкретной сущностью платформы (объявлением, обращением в поддержку, сделкой CRM, реквизитами компании, пользователем, материалом контентного модуля и так далее).

Один файл может иметь несколько связей; одна сущность может иметь несколько файлов. Связь фиксирует контекст использования и определяет, в рамках какого продуктового сценария файл должен быть удалён или сохранён.

Ключевые атрибуты:
- идентификатор;
- идентификатор файла (`StoredFile`);
- тип сущности-получателя (компания, пользователь, объявление, ценовое предложение, сделка CRM, поступление сырья, отгрузка, контрагент CRM, обращение в поддержку, сообщение в обращении, материал контента, иллюстрация номенклатуры, шаблон документа CRM, финансовый документ);
- идентификатор сущности-получателя;
- порядок отображения в рамках сущности (для упорядоченных коллекций — например, фотографии объявления);
- роль файла в рамках сущности (например, для логотипа — «логотип»; для медиа объявления — «фото» либо «видео»; для аватара — «аватар»; для документа CRM — «договор», «ТТН» и иные);
- дата создания связи.

### 3.3. Иммутабельность контента
Содержимое существующего файла не изменяется. Если пользователь хочет «изменить» файл (заменить логотип, заменить фотографию объявления, заменить вложение в обращении поддержки до его отправки), фактически выполняется загрузка нового файла и удаление старого.

Это упрощает кеширование, журналирование и удаление файлов: каждое физическое изменение контента — это новая запись `StoredFile` с собственными датой загрузки, автором, хешем.

Версионирование (хранение нескольких редакций одного логического файла) в первой версии не поддерживается (раздел 12.2).

## 4. Уровни доступа

### 4.1. public — публичный доступ
Файлы уровня `public` доступны по прямому URL без авторизации. Прямой URL не подписывается и не имеет срока жизни.

Этот уровень применяется к файлам, которые по продуктовому замыслу видимы любому посетителю платформы или внешнему потребителю — поисковому индексатору, ссылке в шаринге и так далее.

Категории файлов уровня `public` в первой версии:
- логотипы компаний;
- медиафайлы опубликованных объявлений торговой площадки (фотографии, видео);
- иллюстрации видов сырья в справочнике номенклатуры;
- медиафайлы опубликованных материалов контентных модулей (новости, база знаний, обучение, индексы цен).

Прямой URL публичного файла стабилен на весь срок существования файла. URL не меняется при изменении контента (контент иммутабелен — раздел 3.3).

Защита публичных файлов от хотлинкинга (вставки на сторонние сайты с использованием пропускной способности платформы) в первой версии не реализуется (раздел 12.7).

### 4.2. authenticated — авторизованный пользователь платформы
Файлы уровня `authenticated` доступны через подписанную ссылку любому авторизованному пользователю платформы. Бэкенд проверяет факт авторизации и выдаёт подписанную ссылку без дополнительной проверки прав на конкретный файл.

В первой версии единственная категория файлов уровня `authenticated` — аватары пользователей. Аватар пользователя должен быть видим тому, кто видит самого пользователя в любом контексте: в кабинете компании (для коллег), в обращениях поддержки (для администраторов), в комментариях контентных модулей (для иных авторизованных пользователей).

Аватар пользователя не делается публичным по соображениям приватности — платформа не предоставляет в открытый доступ изображения сотрудников компаний-клиентов.

### 4.3. company-private — внутри компании по ролевой модели
Файлы уровня `company-private` доступны только пользователям конкретной компании в соответствии с ролевой моделью этой компании и в соответствии с правилами доступа той сущности, к которой файл привязан.

Бэкенд при выдаче подписанной ссылки проверяет:
- принадлежность пользователя к компании-владельцу файла;
- наличие у пользователя роли, дающей доступ к сущности, к которой файл привязан;
- состояние компании (доступ к приватным файлам компании сохраняется в состояниях `demo`, `active`, `past_due`; в состоянии `suspended` доступ к приватным файлам кабинета закрыт за исключением раздела биллинга и финансовых документов; в состоянии `archived` доступ закрыт за исключением раздела биллинга у собственника; в состоянии `blocked` доступ закрыт полностью).

Категории файлов уровня `company-private` в первой версии:
- документы CRM (договоры, ТТН, УПД, спецификации и иные документы, перечисленные в `03-crm/documents.md`, раздел 1.2);
- фотографии и иные файлы поступления сырья (`03-crm/intake.md`, раздел 1.3);
- прикреплённые документы и фотографии при отгрузке (`03-crm/shipment.md`, раздел 1.2 и 1.4);
- шаблоны документов в настройках CRM (`03-crm/ui.md`, раздел 1.10).

### 4.4. conversation-private — стороны переписки
Файлы уровня `conversation-private` доступны только сторонам конкретного канала переписки. В первой версии единственный такой канал — обращения в поддержку.

Бэкенд при выдаче подписанной ссылки проверяет, что запрашивающий пользователь является:
- автором обращения в поддержку, к которому файл привязан;
- сотрудником компании-автора обращения, имеющим право видеть это обращение в кабинете (в первой версии — только автор обращения; иные сотрудники компании обращения не видят, см. `07-platform/admin-panel.md`, раздел 14.6);
- ведущим администратором обращения;
- любым активным администратором платформы (поскольку любой администратор может взять обращение в работу или участвовать в его обработке).

Категория файлов уровня `conversation-private` в первой версии:
- вложения в сообщениях обращений поддержки (`07-platform/admin-panel.md`, раздел 14.9).

### 4.5. platform-private — платформенные сотрудники
Файлы уровня `platform-private` доступны только платформенным сотрудникам в соответствии с ролевой моделью админ-панели.

Бэкенд при выдаче подписанной ссылки проверяет наличие у пользователя платформенной роли (`admin`, `moderator`, `content_manager`) и применимость роли к контексту файла. Контекст определяется по сущности, к которой файл привязан.

В первой версии файлы уровня `platform-private` — это исключительно служебные файлы платформы, генерируемые автоматически: финансовые документы платформы, направляемые компаниям-клиентам.

Категория файлов уровня `platform-private` в первой версии:
- финансовые документы платформы — счета, акты оказания услуг, кассовые чеки, счета-фактуры, УПД, корректирующие документы (`07-platform/subscriptions-and-billing.md`, раздел 8). Доступ к скачиванию документа компанией-клиентом — отдельный сценарий через подписанную ссылку с проверкой принадлежности пользователя к компании-плательщику; на уровне хранилища документ доступен платформе и платформенному сотруднику. Сотрудники компании-клиента получают временный доступ к документу через специальный сценарий выдачи подписанной ссылки, который проверяет роль пользователя в компании (собственник, активный бухгалтер).

## 5. Каталог потребителей

Раздел фиксирует полный перечень продуктовых потребителей файлового хранилища в первой версии. Для каждого потребителя указаны: уровень доступа, кто загружает, кто видит, лимиты, привязка к сущности, политика удаления.

### 5.1. Логотип компании
**Применимо к:** все типы компаний (заготовитель, трейдер, переработчик).

**Уровень доступа:** `public`.

**Кто загружает:** собственник компании. Иные роли в первой версии логотип не загружают.

**Кто видит:** любой посетитель платформы — логотип отображается в публичных контекстах: на странице компании в торговой площадке, в карточке объявления (после принятия предложения), на странице компании в публичном профиле и иных местах, где появляется компания.

**Лимиты:**
- одно изображение на компанию;
- максимальный размер — 5 МБ;
- допустимые форматы — JPG, PNG, WEBP.

**Привязка к сущности:** компания.

**Политика удаления:**
- собственник может в любой момент заменить логотип (новый файл загружается, старый удаляется);
- при удалении компании — файл удаляется (раздел 9.2).

### 5.2. Аватар пользователя
**Применимо к:** все пользователи платформы — сотрудники компаний и платформенные сотрудники.

**Уровень доступа:** `authenticated`.

**Кто загружает:** сам пользователь — в настройках своего профиля.

**Кто видит:** любой авторизованный пользователь платформы. Аватар отображается рядом с именем пользователя в местах, где имя видно в первой версии:
- в списке сотрудников кабинета компании;
- в карточке задач CRM (`03-crm/tasks-and-comms.md`);
- в обращениях поддержки рядом с автором сообщения;
- в комментариях к материалам контентных модулей (когда модули будут утверждены).

**Лимиты:**
- одно изображение на пользователя;
- максимальный размер — 5 МБ;
- допустимые форматы — JPG, PNG, WEBP.

**Привязка к сущности:** пользователь.

**Политика удаления:**
- пользователь может в любой момент заменить аватар (новый файл загружается, старый удаляется);
- пользователь может удалить аватар без замены — в этом случае в местах отображения используется стандартное изображение «нет аватара»;
- при удалении пользователя из компании файл аватара сохраняется до фактического удаления учётной записи (учётная запись пользователя сохраняется в служебном виде после удаления из компании; см. `07-platform/auth-and-onboarding.md`);
- при жёстком удалении учётной записи пользователя файл удаляется.

Аватары не унаследованы при изменении компании пользователя — каждый пользователь привязан к одной компании в первой версии, и при удалении из компании учётная запись становится служебной.

## 7. Жизненный цикл файла

### 7.1. Загрузка
Загрузка файла инициируется пользователем в продуктовом сценарии (создание объявления, оформление поступления сырья, создание сообщения в обращении поддержки и так далее). Конкретный сценарий загрузки фиксируется в продуктовом документе соответствующего модуля; настоящий документ задаёт общие правила.

При инициировании загрузки клиент:
- проверяет, что выбранный файл соответствует допустимым форматам и лимитам категории (раздел 10.1);
- передаёт файл на бэкенд;
- получает идентификатор записи `StoredFile`.

На бэкенде запись `StoredFile` создаётся в состоянии `uploading`. После успешной передачи содержимого запись переводится в состояние `scanning` и отправляется на антивирусную проверку (раздел 7.2).

Файл в состоянии `uploading` или `scanning` не доступен потребителям — не отдаётся по прямой ссылке для публичных файлов и не доступен по подписанной ссылке для приватных. Попытка обращения возвращает служебный ответ «файл ещё не готов».

### 7.2. Антивирусная проверка
Каждый загруженный файл проходит обязательную антивирусную проверку. Конкретный антивирусный движок и параметры проверки фиксируются в архитектурном документе. Настоящий документ фиксирует продуктовое требование о факте проверки и поведение по её результатам.

Возможные результаты проверки:
- `clean` — файл чист, переводится в состояние `available` и становится доступен потребителям;
- `infected` — обнаружена угроза, поведение раздела 7.3;
- `error` — антивирусная проверка не завершилась с однозначным результатом, поведение раздела 7.4.

Срок выполнения проверки в первой версии не нормируется. Целевое значение — несколько секунд для типового файла; конкретный SLA фиксируется в архитектурном документе.

### 7.3. Поведение при детекте угрозы
Файл, по которому антивирусная проверка вернула `infected`:
- переводится в состояние `failed`;
- физически удаляется из хранилища;
- запись `StoredFile` сохраняется с пометкой `infected` для журнала;
- автору загрузки немедленно отображается сообщение «загрузка не выполнена: файл не прошёл проверку безопасности»;
- продуктовый сценарий, инициировавший загрузку, не завершается этим файлом — пользователю предлагается выбрать другой файл;
- никаких связей `FileReference` для этого файла не создаётся;
- факт детекта фиксируется в журнале файлового хранилища (раздел 11.2).

Уведомление пользователя направляется только через интерфейс, без email и SMS (это операционное событие, не требующее канала за пределами клиента).

В первой версии повторная попытка проверки того же файла не предусматривается.

### 7.4. Поведение при ошибке проверки
Файл, по которому антивирусная проверка вернула `error` (проверка не завершилась за приемлемое время, провайдер недоступен, иные технические сбои):
- переводится в состояние `failed`;
- физически удаляется из хранилища;
- автору загрузки отображается сообщение «загрузка временно недоступна, попробуйте позже»;
- факт ошибки фиксируется в журнале файлового хранилища (раздел 11.2);
- если число ошибок проверки за короткий период превышает порог, событие отображается в служебной аналитике админ-панели и платформенные сотрудники обращают на него внимание; конкретные пороги — в архитектурном документе.

Файл с результатом `error` рассматривается как не прошедший — это консервативный подход, исключающий пропуск потенциальных угроз.

### 7.5. Хранение
Файл в состоянии `available` хранится в S3-совместимом хранилище. Срок хранения определяется политикой удаления соответствующего потребителя (раздел 5) и сценарием удаления (раздел 9).

Хеш содержимого фиксируется в момент загрузки и не изменяется. При повторной загрузке идентичного файла (сравнивается хеш) в первой версии дедупликация на уровне физического хранения не обязательна — это вопрос архитектурного слоя.

Резервное копирование пользовательских файлов в первой версии не реализуется как явная продуктовая функция (раздел 12.3). Вопрос инфраструктурного резервирования решается на стороне S3-провайдера.

### 7.6. Замена
Замена файла — это сценарий, в котором пользователь хочет «изменить» существующий файл (заменить логотип компании, заменить аватар, заменить документ CRM на исправленный). Платформа реализует это через:
- загрузку нового файла как самостоятельного `StoredFile`;
- создание новой связи `FileReference` к той же сущности с тем же атрибутом роли;
- удаление старой связи и старого файла.

С точки зрения интерфейса пользователь видит замену как одну атомарную операцию. С точки зрения хранения — это две операции (загрузка нового, удаление старого), выполняемые транзакционно: если загрузка нового не удалась, старый сохраняется.

Иммутабельность контента (раздел 3.3) обеспечивает, что прямой URL публичного файла указывает на конкретный байт-в-байт файл. После замены прямой URL старого файла перестаёт работать; URL нового файла — другой.

### 7.7. Удаление
Удаление файла происходит:
- по явному действию пользователя (удаление документа CRM, удаление аватара, удаление шаблона документа);
- автоматически при удалении сущности, к которой файл привязан (удаление поступления сырья влечёт удаление прикреплённых фотографий);
- автоматически при удалении компании (раздел 9.2);
- автоматически при удалении пользователя (для аватаров).

Физическое удаление файла из S3 выполняется отложенно — с задержкой, обеспечивающей возможность восстановления при ошибочном удалении. В первой версии задержка фиксируется как параметр архитектурного слоя; продуктовое требование — что после явного удаления пользователем файл больше не отображается ни в одном продуктовом контексте немедленно, но физическое удаление допускается выполнять отложенно.

Запись `StoredFile` после физического удаления сохраняется в состоянии `deleted` для целей журнала, но уже не указывает на физический файл.

## 8. Доступ к файлам

### 8.1. Публичные файлы
Публичные файлы (уровень `public`) доступны по прямому URL. Состав URL:
- эндпоинт S3-совместимого хранилища (фиксируется в архитектуре);
- путь к файлу;
- имя файла.

URL стабилен на весь срок существования файла. URL не содержит подписи и не имеет срока жизни. URL может быть скопирован и использован кем угодно.

Для публичных файлов клиент не запрашивает у бэкенда специальную ссылку — URL формируется по детерминированному правилу из идентификатора файла. Это обеспечивает кешируемость на уровне браузера и снижает нагрузку на бэкенд.

### 8.2. Приватные файлы и подписанные ссылки
Приватные файлы (уровни `authenticated`, `company-private`, `conversation-private`, `platform-private`) доступны только через подписанные временные ссылки. Бэкенд при запросе пользователя:
- проверяет факт авторизации (для всех уровней кроме `public`);
- проверяет права доступа к конкретному файлу в соответствии с правилами уровня (раздел 4);
- генерирует подписанную ссылку с ограниченным сроком жизни и возвращает её клиенту;
- клиент скачивает файл по этой ссылке непосредственно из S3.

Запросы подписанных ссылок журналируются в подсистеме файлового хранилища (раздел 11.1). Журнал генерации подписанных ссылок хранится двенадцать месяцев и доступен администратору для разбора инцидентов доступа.

### 8.3. Срок жизни подписанной ссылки
В первой версии срок жизни подписанной ссылки — пятнадцать минут с момента выдачи. По истечении срока ссылка перестаёт работать и пользователь должен запросить новую.

Срок является параметром платформенной конфигурации (`07-platform/admin-panel.md`, раздел 18.1) и может быть изменён администратором без изменения кода. Изменение срока влияет на новые ссылки; уже выданные ссылки сохраняют свой исходный срок.

### 8.4. Запрос подписанной ссылки и проверка прав
Бэкенд предоставляет защищённый эндпойнт для запроса подписанной ссылки на конкретный файл по его идентификатору. Эндпойнт:
- требует авторизации;
- проверяет права в соответствии с уровнем доступа файла и правилами раздела 4;
- при положительной проверке возвращает подписанную ссылку;
- при отрицательной проверке возвращает ответ «доступ запрещён» без указания, существует ли файл (исключает утечку идентификаторов).

Состояние компании пользователя влияет на доступ:
- в состояниях `demo`, `active`, `past_due` — пользователь имеет доступ к приватным файлам своей компании;
- в состоянии `suspended` — доступ к приватным файлам кабинета закрыт за исключением финансовых документов в разделе биллинга (`07-platform/subscriptions-and-billing.md`, раздел 5.5 и 8.8);
- в состоянии `archived` — доступ закрыт за исключением финансовых документов у собственника;
- в состоянии `blocked` — доступ закрыт полностью.

Платформенные сотрудники получают подписанные ссылки на файлы соответствующего уровня независимо от состояния компании-владельца файла.

## 9. Удаление файлов и удаление компании

### 9.1. Удаление файла из активной сущности
Удаление файла из активной сущности (например, удаление документа CRM пользователем компании, удаление шаблона документа, удаление аватара) выполняется в продуктовом сценарии модуля и подчиняется правилам модуля. Настоящий документ фиксирует только общее поведение хранилища: после явного удаления файл становится недоступен немедленно, физическое удаление выполняется отложенно (раздел 7.7).

### 9.2. Удаление компании — судьба файлов
При жёстком удалении компании по истечении срока архивации (`07-platform/subscriptions-and-billing.md`, раздел 16.3) файлы компании обрабатываются следующим образом:

| Категория файлов | Поведение при удалении компании |
|---|---|
| Логотип компании | удаляется |
| Медиа объявлений торговой площадки | удаляется (объявления удаляются вместе с компанией) |
| Медиа поступления сырья | удаляется (записи CRM удаляются) |
| Документы CRM | удаляется |
| Медиа отгрузки CRM | удаляется |
| Шаблоны документов CRM | удаляется |
| Финансовые документы платформы | сохраняются в изолированном архиве (раздел 9.3) |
| Вложения в обращениях поддержки | сохраняются (раздел 9.4) |
| Медиа контента (если автор-привязка к компании) | не применимо в первой версии — контент создаётся контент-менеджером, не компаниями |

Файлы пользователей, удалённых из компании ранее жёсткого удаления (аватары), регулируются правилами раздела 5.2.

### 9.3. Финансовые документы — изолированный архив
Финансовые документы платформы (счета, акты, чеки, счета-фактуры, УПД, корректирующие документы), относящиеся к удаляемой компании, переносятся в изолированный архив платформы. Архив:
- хранится бессрочно;
- доступен только администратору платформы;
- не отображается в основном интерфейсе и не индексируется в обычных журналах;
- предоставляется по требованию в случаях налоговых или аудитных запросов.

Конкретное техническое решение архива — изолированный bucket с расширенным доступом, либо отдельная таблица в базе данных с отдельной политикой хранения — фиксируется в архитектурном документе.

Этот режим обеспечивает соответствие требованиям 54-ФЗ и налогового законодательства, по которым налогоплательщик обязан хранить первичные документы независимо от продолжения отношений с контрагентом.

### 9.4. Обращения поддержки — сохраняются
Обращения в поддержку и их вложения сохраняются после удаления компании-автора. Это обеспечивает:
- целостность журнала работы поддержки;
- возможность ретроспективного аудита решений администраторов;
- соответствие подходу к `AdminActionLog` (`07-platform/admin-panel.md`, раздел 16.4) — служебные журналы хранятся бессрочно.

После удаления компании привязка обращения к компании сохраняется как идентификатор, без возможности раскрыть детальную информацию о компании. Привязка обращения к пользователю-автору сохраняется как идентификатор учётной записи; при сохранении служебной учётной записи пользователя после удаления компании имя автора сообщений в обращении остаётся отображаемым.

Удаление обращений и их вложений возможно только в рамках сценария категории `data_deletion` обращения поддержки (`07-platform/admin-panel.md`, раздел 14.2), оформленного отдельным решением администратора в рамках исполнения 152-ФЗ.

### 9.5. AdminActionLog и журналы доставок — сохраняются
Журналы платформенных подсистем не относятся к категориям файлов настоящего документа. Записи журналов хранятся в отдельных служебных хранилищах в соответствии с правилами `07-platform/admin-panel.md` (раздел 16.4 — `AdminActionLog` бессрочно), `07-platform/notifications.md` (раздел 14.3 — журнал доставок 12 месяцев) и иных доменных документов.

Если в рамках доменного журнала фигурирует ссылка на удалённый файл (например, в журнале кейса модерации остаётся ссылка на удалённое объявление и его медиафайлы), журнал сохраняет ссылку как идентификатор без возможности скачать файл — файл к этому моменту уже физически удалён.

## 10. Загрузка с клиента

### 10.1. Поддерживаемые типы файлов на клиенте
Клиент (веб-интерфейс или мобильный интерфейс) до отправки файла на бэкенд проверяет:
- расширение и MIME-тип файла на соответствие допустимым форматам категории;
- размер файла на соответствие лимиту категории;
- общее число файлов в коллекции на соответствие лимиту категории.

При несоответствии клиент отображает пользователю понятное сообщение и не отправляет файл. Дублирующие проверки выполняются на бэкенде — клиентская проверка является только удобством для пользователя, а контроль безопасности обеспечивает бэкенд.

### 10.2. Превью и обработка изображений
В первой версии платформа не выполняет автоматическую обработку изображений на сервере (resize, watermark, формирование превью разных размеров). Изображения отображаются в интерфейсе в исходном виде, с CSS-масштабированием.

Это упрощает архитектуру в первой версии. Серверная обработка изображений (генерация thumb-вариантов, сжатие, оптимизация для веба) — открытый вопрос для последующих версий (раздел 12.6).

Если пользователь загрузил очень большое изображение (близко к лимиту 10 МБ), клиент перед отправкой может выполнить браузерное сжатие; конкретные параметры сжатия — на усмотрение фронтенд-реализации.

### 10.3. Прогресс загрузки и обрыв соединения
При загрузке больших файлов (видео объявлений, документы CRM) клиент:
- отображает прогресс загрузки;
- обеспечивает возможность отмены загрузки пользователем до её завершения;
- при обрыве соединения возможность возобновления частичной загрузки в первой версии не предусматривается — пользователю предлагается повторить загрузку с начала.

Возобновляемые загрузки (multipart upload с продолжением) — за пределами первой версии.

### 10.4. Поведение при превышении лимита
При попытке загрузить файл, превышающий лимит размера или формата:
- клиент отображает понятное сообщение с указанием актуального лимита;
- файл не отправляется на бэкенд;
- запись `StoredFile` не создаётся.

При попытке добавить файл сверх лимита количества (например, 11-я фотография в объявлении при максимуме 10):
- клиент отображает сообщение с указанием максимального количества;
- файл не отправляется на бэкенд.

Серверная сторона дополнительно проверяет эти лимиты при получении запроса и возвращает ошибку при их нарушении (защита от обхода клиентской проверки).

## 11. Журналирование

### 11.1. Что фиксируется при загрузке
В подсистеме файлового хранилища фиксируется журнал операций. Записи журнала:
- идентификатор записи;
- идентификатор файла (`StoredFile`);
- тип операции (`upload_initiated`, `upload_completed`, `scan_started`, `scan_clean`, `scan_infected`, `scan_error`, `signed_url_generated`, `delete_requested`, `delete_completed`);
- идентификатор актора (пользователь или платформенный сотрудник);
- дата и время;
- структурированные параметры операции (для `signed_url_generated` — продолжительность жизни ссылки и категория проверки прав; для `scan_*` — результат и идентификатор движка проверки);
- идентификатор связанной сущности (`FileReference`), если применимо.

Для публичных файлов запись `signed_url_generated` не формируется — публичные файлы скачиваются по прямому URL без обращения к бэкенду.

Журнал доступа к публичным файлам через прямой URL в первой версии не ведётся — это уровень S3-провайдера, и использование таких журналов оставлено на архитектурный слой.

### 11.2. Что фиксируется при удалении
При удалении файла фиксируется запись операции `delete_requested` с указанием:
- актора (пользователь, платформенный сотрудник, либо автоматический процесс — например, при удалении компании);
- причины (явное действие пользователя, удаление родительской сущности, удаление компании, истечение срока обращения и так далее);
- состояния, в котором файл находился (`available`, `failed`).

Запись `delete_completed` фиксируется после физического удаления файла из S3 (с учётом отложенного удаления — раздел 7.7).

### 11.3. Доступ к журналу
Журнал файлового хранилища доступен:
- администратору платформы — полный доступ через раздел «Журналы» админ-панели (`07-platform/admin-panel.md`, раздел 15.1);
- модератору и контент-менеджеру — нет;
- пользователям компаний — нет.

Срок хранения журнала файлового хранилища — двенадцать месяцев.

При расследовании инцидентов администратор использует журнал в комбинации со срезами по сущности (раздел 15.2 admin-panel.md) — например, «все операции с файлами, связанными с конкретной компанией за период».

## 12. Открытые вопросы и ограничения первой версии

### 12.1. CDN и кеширование — за пределами первой версии
В первой версии публичные файлы отдаются напрямую с S3-эндпоинта без CDN. Подключение CDN-кеширования рассматривается в последующих версиях при росте нагрузки на публичные файлы — особенно на видео объявлений торговой площадки и медиа контентных модулей.

Конкретный порог («когда подключать CDN») является архитектурным решением и не фиксируется в продуктовом документе. Признаки готовности к подключению — устойчиво высокая суточная отдача публичного контента или устойчивые нарекания пользователей на скорость загрузки фотографий и видео.

### 12.2. Версионирование файлов — за пределами первой версии
В первой версии хранение нескольких редакций одного логического файла не поддерживается (раздел 3.3 — контент иммутабелен). При замене файла старый удаляется. Возможность хранить историю версий с возможностью отката — вопрос последующих версий, если будет продуктовый запрос.

### 12.3. Резервное копирование пользовательских файлов — за пределами первой версии
В первой версии явная функция резервного копирования пользовательских файлов на стороне платформы не предусматривается. Вопрос отказоустойчивости решается на уровне инфраструктуры S3-провайдера. Целевые показатели по восстановлению (RPO, RTO) — архитектурный документ.

### 12.4. Конкретный провайдер S3 — архитектурное решение
Выбор конкретного S3-совместимого провайдера (Selectel, Yandex Object Storage, российский провайдер, развёртывание собственного MinIO и иные варианты), конфигурация bucket-ов, регионов, репликации — архитектурный документ.

### 12.5. Антивирусный движок — архитектурное решение
Выбор конкретного антивирусного движка, режим интеграции (сервис в сети, sidecar, внешний API), периодичность обновления баз сигнатур, поведение при недоступности движка — архитектурный документ. Настоящий документ фиксирует только продуктовое требование о факте проверки и поведение по её результатам.

### 12.6. Пресеты обработки изображений — архитектурное решение
Серверная обработка изображений (генерация превью разных размеров, оптимизация для веба, watermark на публичных фото) в первой версии не выполняется (раздел 10.2). Подключение пресетов обработки — архитектурное решение для последующих версий.

### 12.7. Защита от хотлинкинга и обхода гейтинга — за пределами первой версии
В первой версии публичные файлы доступны по прямому URL и могут быть встроены в сторонние сайты или скачаны в обход гейтинга по подписке (для медиа контентных модулей). Меры защиты — Referer-проверки, токенизированные публичные URL для гейтинговых файлов, водяные знаки — рассматриваются в последующих версиях.

Этот компромисс допустим в первой версии, поскольку:
- основной гейтинг по подписке применяется на уровне страницы материала, а не на уровне файла;
- сценарий целенаправленного скачивания файла по прямой ссылке для обхода подписки требует знания идентификатора файла, который не индексируется и не выдаётся неавторизованным пользователям.

### 12.8. Возобновляемые загрузки (multipart upload) — за пределами первой версии
В первой версии большие файлы загружаются одним непрерывным потоком. При обрыве соединения возобновление частичной загрузки не поддерживается. Возобновляемые загрузки — открытый вопрос для последующих версий, особенно актуальный для видео объявлений (200 МБ) при нестабильном мобильном соединении.

### 12.9. Дедупликация на уровне физического хранения — архитектурное решение
Хеш содержимого фиксируется в каждой записи `StoredFile` (раздел 7.5), однако физическая дедупликация (хранение единственной копии байт-в-байт идентичных файлов) в первой версии не обязательна. Вопрос экономии хранилища — архитектурный документ.

## 13. Интеграция со смежными подсистемами

Настоящий раздел кратко фиксирует точки интеграции файлового хранилища с доменными подсистемами. Все правила поведения подсистем — в их доменных документах.

### 13.1. Аутентификация и онбординг
Аватары пользователей — категория файлов уровня `authenticated`. Загружаются пользователем в настройках профиля. Связь с auth-документом — `07-platform/auth-and-onboarding.md` в части видимости имени и аватара пользователя в местах его упоминания.

### 13.2. Биллинг и подписки
Финансовые документы платформы — категория файлов уровня `platform-private` со специальным сценарием выдачи подписанной ссылки сотрудникам компании-плательщика. Источник правил выпуска и хранения — `07-platform/subscriptions-and-billing.md`, раздел 8.

### 13.3. Уведомления
Уведомления могут содержать ссылки на файлы (например, ссылки на финансовые документы в уведомлениях категории «биллинг»). Подписанные ссылки в email-уведомлениях формируются с расширенным сроком жизни (предлагается 7 дней — параметр конфигурации) для удобства получателя; конкретное значение уточняется при разработке. Источник правил уведомлений — `07-platform/notifications.md`.

### 13.4. Модерация
Файлы — объект модерации в категориях:
- медиафайлы объявлений торговой площадки;
- медиафайлы профиля компании (логотип);
- медиафайлы материалов контентных модулей (через комментарии и реакции; сами материалы создаются контент-менеджером и не модерируются по `07-platform/moderation.md`, раздел 4.9).

При снятии модератором сущности (объявления, элемента профиля компании) связанные медиафайлы помечаются снятыми, но физически сохраняются для возможного пересмотра решения. Источник правил — `07-platform/moderation.md`.

### 13.5. Административная панель
Раздел «Журналы» админ-панели предоставляет единый интерфейс просмотра журналов файлового хранилища с фильтрами по подсистеме и по связанной сущности (`07-platform/admin-panel.md`, раздел 15.1). Отдельного раздела «Файлы» в админ-панели в первой версии не предусматривается — управление файлами выполняется в контексте соответствующих сущностей (компании, объявления, обращения).

### 13.6. Торговая площадка
Медиафайлы объявлений и связанные правила публикации — `04-marketplace/listings.md`, раздел 8.

### 13.7. CRM
Документы, медиа поступления сырья, медиа отгрузки, шаблоны документов — `03-crm/documents.md`, `03-crm/intake.md`, `03-crm/shipment.md`, `03-crm/ui.md`.

### 13.8. Контентные модули
Медиафайлы материалов контентных модулей — `05-content/news/`, `05-content/knowledge-base/`, `05-content/education/`, `05-content/price-indices/` (стабы; конкретные правила формируются при утверждении модулей).
