## 1. Назначение раздела

Настоящий документ фиксирует полную спецификацию административной панели ЭкоПлатформы. Документ является единым источником истины по составу разделов админ-панели, по их распределению между платформенными ролями, по модели доступа платформенных сотрудников к служебному контуру, по модели обращений в поддержку и по единому административному журналу.

Документ определяет:
- модель админ-панели как общего служебного кабинета для трёх платформенных ролей (администратор, модератор, контент-менеджер);
- состав разделов админ-панели и их распределение между ролями;
- сущности административного контура, не покрытые иными документами (платформенный сотрудник, административный журнал, обращение в поддержку, платформенный справочник, системная рассылка);
- модель доступа платформенных сотрудников: создание учётных записей, обязательное двухфакторное подтверждение, общий вход;
- модель единого административного журнала значимых действий (`AdminActionLog`);
- модель модуля поддержки — двусторонний канал между пользователем и администратором;
- системные рассылки — фильтры аудитории и каналы;
- служебные настройки платформы и аналитику первой версии;
- интерфейсные точки входа в подсистемы аутентификации, биллинга, уведомлений, модерации и контента.

Документ не дублирует доменные правила подсистем. Правила переходов состояний пользователя и компании — `07-platform/auth-and-onboarding.md`. Правила биллинга, тарифной модели, возвратов, изменения цен — `07-platform/subscriptions-and-billing.md`. Правила каталога событий уведомлений, шаблонов, очередей доставки — `07-platform/notifications.md`. Правила кейсов модерации, очереди, эскалаций и санкций — `07-platform/moderation.md`. Настоящий документ описывает только интерфейсный слой над этими подсистемами и собственные сущности административного контура.

## 2. Принципы

### 2.1. Единая админ-панель для трёх платформенных ролей
Административная панель — общий служебный кабинет для всех трёх платформенных ролей: администратор платформы, модератор, контент-менеджер. Все три роли заходят в один и тот же интерфейс по одному адресу. Состав видимых разделов и доступных операций определяется ролью.

Отдельных кабинетов для модератора и контент-менеджера в первой версии не предусмотрено.

### 2.2. Интерфейсный слой над доменными подсистемами
Админ-панель не вводит новых доменных правил. Правила биллинга, аутентификации, модерации, уведомлений и контента фиксируются в соответствующих доменных документах. Настоящий документ описывает только:
- состав экранов и операций админ-панели;
- модель доступа платформенных сотрудников к этим экранам;
- собственные сущности (раздел 3), не покрываемые иными документами;
- единый административный журнал поверх доменных журналов.

При расхождении между настоящим документом и доменным документом приоритет имеет доменный документ.

### 2.3. Гибридное журналирование
Журналирование действий в платформе — гибридное:

- каждая подсистема ведёт собственный доменный журнал (журнал кейсов модерации, журнал изменений шаблонов уведомлений, журнал доставок уведомлений, журналы аутентификации, журналы биллинга);
- админ-панель ведёт единый верхнеуровневый журнал значимых действий администратора и иных платформенных сотрудников — `AdminActionLog` (раздел 16);
- админ-панель предоставляет единый интерфейс просмотра разрозненных доменных журналов с фильтрами по подсистеме и по связанной сущности (раздел 15).

`AdminActionLog` фиксирует только закрытый перечень значимых действий (раздел 16.2). Технического журнала входов в админ-панель и просмотров экранов в первой версии не ведётся.

### 2.4. Общий вход и обязательное двухфакторное подтверждение
Платформенные сотрудники проходят процедуру входа на той же форме, что и пользователи компаний (`07-platform/auth-and-onboarding.md`, раздел 5). После успешного входа система определяет, что у учётной записи есть платформенная роль, и направляет пользователя в админ-панель.

Двухфакторное подтверждение по SMS на подтверждённый телефон обязательно для всех платформенных сотрудников и не может быть отключено самим сотрудником. Включение двухфакторного подтверждения происходит при создании учётной записи платформенного сотрудника (раздел 4.3).

### 2.5. Равенство администраторов
Все администраторы платформы имеют равный набор прав. Отдельной роли «суперадминистратор» не предусматривается. Любой администратор может:
- создавать новых платформенных сотрудников любых ролей (включая других администраторов);
- блокировать платформенных сотрудников;
- удалять платформенных сотрудников.

Первый администратор задаётся при развёртывании платформы (раздел 4.2). Все последующие платформенные сотрудники создаются администраторами через админ-панель.

### 2.6. Поддержка как самостоятельный сценарный модуль
Поддержка пользователей реализована как полноценный двусторонний канал между пользователем и администратором (раздел 14). Пользователь компании создаёт обращения и ведёт переписку из своего кабинета. Администратор ведёт обращения и переписывается с пользователем из админ-панели.

Поддержка не является внешним каналом (электронной почтой, мессенджером) и реализуется средствами самой платформы. В первой версии операторами поддержки выступают администраторы; отдельной роли «оператор поддержки» не вводится.

### 2.7. Принцип наименьшего доступа для модератора и контент-менеджера
Модератор и контент-менеджер имеют доступ только к тем разделам админ-панели, которые непосредственно соответствуют их функциям (раздел 6). Расширение доступа конкретного сотрудника возможно только через назначение ему дополнительной платформенной роли. Совмещение ролей у одного платформенного сотрудника допускается (раздел 4.1).

## 3. Сущности административного контура

В разделе фиксируется перечень сущностей, специфичных для админ-панели и не покрытых иными доменными документами. Состав полей каждой сущности подлежит окончательной фиксации в документе модели данных `08-architecture/data-model.md`.

### 3.1. Платформенный сотрудник (PlatformStaff)
Учётная запись пользователя со стороны платформы.

Ключевые атрибуты:
- идентификатор;
- ссылка на пользователя (`User`);
- набор платформенных ролей (`admin`, `moderator`, `content_manager`);
- состояние (`active`, `blocked`, `removed`);
- дата создания;
- автор создания (платформенный сотрудник);
- дата последней блокировки/разблокировки;
- автор последней блокировки/разблокировки;
- дата удаления;
- автор удаления.

Платформенный сотрудник — это пользователь платформы, не привязанный к компании. У такой учётной записи нет роли в компании, нет подписки и нет биллингового профиля.

### 3.2. Запись административного журнала (AdminActionLog)
Запись о значимом действии платформенного сотрудника в админ-панели.

Ключевые атрибуты:
- идентификатор;
- идентификатор актора (платформенный сотрудник);
- роли актора на момент действия;
- тип действия (закрытый перечень — раздел 16.2);
- объект действия (тип объекта и идентификатор объекта; например, компания, пользователь, тариф, шаблон уведомления, обращение в поддержку, кейс модерации);
- параметры действия (структурированные, в формате «ключ — значение»);
- свободный комментарий актора (опционально);
- дата и время действия.

Подробно — раздел 16.

### 3.3. Обращение в поддержку (SupportTicket)
Запись об обращении пользователя компании в поддержку платформы.

Ключевые атрибуты:
- идентификатор;
- автор обращения (пользователь);
- компания автора;
- категория (закрытый перечень — раздел 14.2);
- тема (короткая строка, заполняется пользователем);
- состояние (`new`, `in_progress`, `awaiting_user`, `resolved`, `closed`);
- ведущий администратор (платформенный сотрудник; назначается при первом действии администратора по обращению);
- связанная компания (опционально, для категорий с обязательной привязкой — раздел 14.8);
- связанный кейс модерации (опционально);
- связанный счёт, подписка, возврат (опционально);
- дата создания;
- дата последнего сообщения;
- дата перехода в `resolved`;
- дата закрытия.

### 3.4. Сообщение в обращении (SupportTicketMessage)
Запись о сообщении в переписке по обращению.

Ключевые атрибуты:
- идентификатор;
- идентификатор обращения;
- автор (пользователь компании или платформенный сотрудник);
- роль автора на момент сообщения;
- текст сообщения;
- дата создания;
- признак служебной заметки (видна только платформенным сотрудникам — см. раздел 14.7).

### 3.5. Вложение в обращении (SupportTicketAttachment)
Файл, прикреплённый к сообщению обращения.

Ключевые атрибуты:
- идентификатор;
- идентификатор сообщения;
- имя файла;
- тип содержимого;
- размер;
- ссылка на файл в хранилище;
- дата загрузки.

### 3.6. Платформенный справочник (PlatformDictionary)
Каталог настраиваемого справочника платформы (раздел 13).

Ключевые атрибуты:
- идентификатор;
- код справочника (`complaint_reasons`, `decision_reasons`, `restriction_modules`, `nomenclature`, `block_reasons` и иные — раздел 13.1);
- описание;
- признак активности.

### 3.7. Запись справочника (PlatformDictionaryEntry)
Элемент конкретного справочника.

Ключевые атрибуты:
- идентификатор;
- идентификатор справочника;
- код элемента;
- значение (название элемента);
- структурированные дополнительные атрибуты (зависят от справочника — например, у `complaint_reasons` может быть указан тип объекта, к которому код применим);
- признак активности;
- дата создания;
- автор создания;
- дата последнего изменения;
- автор последнего изменения.

### 3.8. Системная рассылка (SystemBroadcast)
Запись об инициированной администратором системной рассылке (раздел 17).

Ключевые атрибуты:
- идентификатор;
- автор (платформенный сотрудник);
- фильтр аудитории (структурированный, см. раздел 17.1);
- каналы (`email`, `in_app`);
- тема (только для email);
- тело;
- статус (`draft`, `scheduled`, `in_progress`, `completed`, `cancelled`);
- запланированная дата отправки (опционально);
- дата создания;
- дата начала отправки;
- дата завершения;
- число получателей в момент формирования;
- число успешно доставленных и число неуспешных по каналам.

### 3.9. Платформенная конфигурация (PlatformConfig)
Запись настраиваемого параметра платформы (раздел 18).

Ключевые атрибуты:
- идентификатор;
- код параметра;
- описание;
- тип значения (число, строка, флаг, перечисление);
- текущее значение;
- дата последнего изменения;
- автор последнего изменения.

История изменений параметров фиксируется в `AdminActionLog` (раздел 16.2).

## 4. Платформенные сотрудники и доступ к админ-панели

### 4.1. Роли и совмещение
В платформе действуют три платформенные роли:

- администратор платформы (`admin`) — `02-roles/admin.md`;
- модератор (`moderator`) — `02-roles/moderator.md`;
- контент-менеджер (`content_manager`) — `02-roles/content-manager.md`.

Один платформенный сотрудник может совмещать несколько ролей. Совокупный набор прав совмещающего сотрудника определяется объединением разрешений всех его ролей.

Платформенный сотрудник не может одновременно быть пользователем какой-либо компании. Регистрация компании на учётную запись платформенного сотрудника недоступна. Это ограничение принципиально и обеспечивает разделение служебного контура от пользовательского (`02-roles/overview.md`, раздел 2).

### 4.2. Первый администратор
Первый администратор платформы создаётся при развёртывании платформы средствами установочной процедуры (seed). В seed-процедуре фиксируются:
- электронная почта первого администратора;
- временный пароль (заменяется при первом входе);
- номер телефона для двухфакторного подтверждения.

Создание первого администратора через админ-панель невозможно — она ещё не доступна. Все последующие платформенные сотрудники создаются через админ-панель администратором (раздел 4.3).

Документ архитектуры развёртывания (`08-architecture/...`) фиксирует технические детали seed-процедуры. Настоящий документ фиксирует только продуктовое требование к её существованию.

### 4.3. Создание, блокировка, удаление платформенного сотрудника
**Создание.** Администратор в разделе «Сотрудники платформы» нажимает «Добавить сотрудника» и заполняет форму:
- электронная почта (обязательно);
- номер телефона (обязательно);
- фамилия, имя, отчество (обязательно; отчество опционально);
- набор платформенных ролей (один или несколько);
- временный пароль (генерируется системой; передаётся новому сотруднику автором создания вне платформы).

При сохранении формы создаётся пользователь и платформенный сотрудник. На электронную почту нового сотрудника отправляется приветственное письмо со ссылкой на вход. На номер телефона при первом входе высылается код двухфакторного подтверждения; включение двухфакторного режима выполняется автоматически и не может быть отключено сотрудником.

При первом входе сотрудник обязан сменить временный пароль (механизм аналогичен сценарию 4.6 `auth-and-onboarding.md` со служебным флагом «требуется смена пароля»).

**Блокировка.** Администратор может заблокировать платформенного сотрудника. При блокировке:
- учётная запись пользователя переводится в состояние `blocked` (`auth-and-onboarding.md`, раздел 9.4);
- все активные сессии сотрудника отзываются;
- сотрудник не может войти в систему.

Блокировка обратима. Действие фиксируется в `AdminActionLog`.

**Удаление.** Администратор может удалить платформенного сотрудника. При удалении:
- платформенный сотрудник переводится в состояние `removed`;
- учётная запись пользователя сохраняется в служебном виде, но не имеет ни ролей в компании, ни платформенных ролей; войти в систему пользователь не может;
- история действий сотрудника в журналах сохраняется и сохраняет привязку к идентификатору.

Удаление необратимо. Восстановление возможно только пересозданием новой учётной записи.

Администратор не может удалить или заблокировать сам себя. Если на платформе остался ровно один администратор, его удаление и блокировка невозможны до создания второго администратора.

### 4.4. Логин, двухфакторное подтверждение, сессии
**Логин.** Платформенный сотрудник заходит на той же форме входа, что и пользователи компаний (`auth-and-onboarding.md`, раздел 5.1). Признак платформенной роли определяется системой по факту наличия записи `PlatformStaff` для пользователя. После успешного входа сотрудник попадает в админ-панель; в кабинет компании платформенный сотрудник не попадает, поскольку у него нет привязки к компании.

**Двухфакторное подтверждение.** Двухфакторное подтверждение по SMS обязательно для всех платформенных сотрудников. Параметры (длина кода, срок жизни, лимиты) — те же, что и для пользователей компаний (`auth-and-onboarding.md`, раздел 8). Отключение двухфакторного режима для платформенного сотрудника недоступно ни сотруднику, ни администратору.

**Сессии.** Сессии платформенного сотрудника подчиняются общим правилам `auth-and-onboarding.md`, разделу 6. Признак «запомнить меня» доступен. Принудительный отзыв сессий платформенного сотрудника другим администратором — операция в разделе «Сотрудники платформы» (фиксируется в `AdminActionLog`).

**Восстановление пароля.** Платформенный сотрудник восстанавливает пароль по общей процедуре (`auth-and-onboarding.md`, раздел 7) — через email или SMS. Восстановление через администратора не предусмотрено.

## 5. Состав разделов админ-панели

В первой версии админ-панель содержит следующие разделы:

| Код | Название | Документ-ссылка |
|---|---|---|
| `users` | Пользователи | раздел 7 настоящего документа |
| `companies` | Компании | раздел 8 |
| `billing` | Биллинг | раздел 9 |
| `notifications` | Уведомления | раздел 10 |
| `moderation` | Модерация | раздел 11 |
| `content` | Контент | раздел 12 |
| `dictionaries` | Справочники | раздел 13 |
| `support` | Поддержка | раздел 14 |
| `journals` | Журналы | раздел 15 |
| `broadcasts` | Системные рассылки | раздел 17 |
| `settings` | Служебные настройки | раздел 18 |
| `analytics` | Аналитика | раздел 19 |
| `staff` | Сотрудники платформы | раздел 4.3 |

Раздел «Сотрудники платформы» технически — служебный экран управления платформенными учётными записями. В навигации он размещается отдельно от прочих разделов и доступен только администратору.

## 6. Доступ к разделам по ролям

### 6.1. Матрица «роль × раздел»

| Раздел | Администратор | Модератор | Контент-менеджер |
|---|---|---|---|
| Пользователи | полный | нет | нет |
| Компании | полный | только публичные данные карточки в контексте кейса | нет |
| Биллинг | полный | нет | нет |
| Уведомления | полный | нет | нет |
| Модерация | полный | работа с очередью и кейсами | нет |
| Контент | полный | нет | полный в пределах роли |
| Справочники | полный | только просмотр справочников модерации | только просмотр контентных классификаций (если применимо) |
| Поддержка | полный | нет | нет |
| Журналы | полный | свои действия в кейсах модерации | свои действия в контентном модуле |
| Системные рассылки | полный | нет | нет |
| Служебные настройки | полный | нет | нет |
| Аналитика | полный | только отчёт по SLA модерации (раздел 19.1) | нет |
| Сотрудники платформы | полный | нет | нет |

«Полный» означает доступ в пределах документа соответствующей роли (`02-roles/admin.md`, `02-roles/moderator.md`, `02-roles/content-manager.md`) и в пределах правил доменных подсистем. «Нет» означает, что пункт меню не отображается.

### 6.2. Что видит модератор
Модератор после входа видит навигацию из следующих пунктов:
- «Очередь модерации» (раздел 11.1);
- «Мои кейсы» — кейсы, удерживаемые им через lock (`07-platform/moderation.md`, раздел 6.3);
- «История моих решений» — журнал действий модератора в его кейсах (раздел 15.3);
- «Справочники модерации» — в режиме просмотра (раздел 13.3);
- «Отчёт по SLA модерации» — в режиме просмотра (раздел 19.1).

Иные разделы, в том числе карточки компаний вне контекста кейса, биллинг, поддержка, журналы по другим подсистемам, модератору недоступны.

### 6.3. Что видит контент-менеджер
Контент-менеджер после входа видит навигацию из следующих пунктов:
- разделы контентных модулей: новости, база знаний, обучение, индексы цен (раздел 12);
- категории, теги и иная контентная классификация (в пределах роли);
- «История моих действий» — журнал действий контент-менеджера в контентных модулях (раздел 15.3).

Иные разделы контент-менеджеру недоступны. Контент-менеджер не видит ни одной модерационной сущности, ни биллинговых данных, ни обращений в поддержку.

### 6.4. Что видит администратор
Администратор имеет доступ ко всем разделам админ-панели в полном объёме. Навигация состоит из всех пунктов, перечисленных в матрице раздела 6.1.

При совмещении ролей видимый набор пунктов — объединение пунктов, доступных каждой из ролей. Совмещение «администратор + модератор» даёт администратору тот же расширенный набор, что и одна роль администратора (поскольку администратор уже имеет полный доступ к разделу модерации).

## 7. Раздел «Пользователи»

### 7.1. Поиск и фильтры
Экран поиска пользователей предоставляет следующие фильтры:
- по идентификатору пользователя;
- по электронной почте (точное и частичное совпадение);
- по номеру телефона (точное совпадение);
- по фамилии, имени, отчеству (частичное совпадение);
- по идентификатору компании;
- по состоянию учётной записи (`active`, `blocked`, `removed_from_company`);
- по периоду регистрации.

Результат — список с краткой карточкой пользователя: идентификатор, ФИО, email, телефон, состояние, компания, роли в компании, дата регистрации.

### 7.2. Карточка пользователя
Карточка пользователя содержит:
- идентификационные данные (ФИО, email, телефон, состояние, дата регистрации);
- ссылку на компанию пользователя;
- набор ролей пользователя в компании;
- историю входов (свод по сценариям 5 и 6 `auth-and-onboarding.md`);
- историю смены контактов (события 9 и 9.3 `auth-and-onboarding.md`);
- активные сессии пользователя (с операцией принудительного отзыва — раздел 7.4);
- историю модерационных кейсов с участием пользователя как автора контента, как автора жалобы или как объекта санкции;
- историю обращений в поддержку, поданных пользователем;
- кнопку «Заблокировать пользователя» — точка входа в кейс модерации (раздел 7.6).

### 7.3. Журнал действий пользователя
Журнал действий пользователя — агрегированный срез из журналов аутентификации (`auth-and-onboarding.md`, раздел 13.5):
- успешный вход;
- неуспешная попытка входа;
- выход;
- смена пароля;
- смена email;
- смена телефона;
- восстановление пароля;
- включение и отключение двухфакторного режима;
- создание и принятие приглашения;
- удаление сотрудника из компании;
- изменение ролей сотрудника.

Доступ к журналу — только администратору. Срок хранения определяется доменным документом аутентификации.

### 7.4. Принудительный отзыв сессий
Принудительный отзыв сессий пользователя — вспомогательная операция администратора. Она применяется в случаях, когда требуется быстро вытолкнуть пользователя из всех устройств без применения санкции (например, при подозрении на компрометацию пароля, по обращению пользователя в поддержку, при временной потере устройства).

Принудительный отзыв сессий **не является санкцией модерации**. Он не переводит учётную запись в состояние `blocked` и не порождает кейса модерации.

При принудительном отзыве:
- все активные сессии пользователя отзываются (`auth-and-onboarding.md`, раздел 6);
- учётная запись остаётся в состоянии `active`;
- пользователь может войти в систему повторно с действующим паролем;
- пользователю направляется уведомление об отзыве сессий (`auth.sessions.revoked`, см. `notifications.md`, раздел 6.1.10);
- действие фиксируется в `AdminActionLog`.

Если требуется одновременно отозвать сессии и закрыть пользователю доступ — применяется санкция `user_block` через кейс модерации (раздел 7.6).

### 7.5. Управление приглашениями
Администратор может:
- просмотреть приглашения по конкретной компании (статус, email приглашаемого, роли, дата создания, срок жизни);
- отменить ещё не принятое приглашение (статус приглашения переводится в «отозвано»);
- увидеть, по какому email-адресу было создано приглашение и кем (собственник компании).

Создание приглашений со стороны администратора в первой версии не предусматривается. Создание приглашений — функция собственника компании (`auth-and-onboarding.md`, раздел 12.2).

### 7.6. Блокировка пользователя — точка входа в кейс модерации
Блокировка пользователя — санкция типа `user_block` (`07-platform/moderation.md`, разделы 8.6 и 9.1). Применить её можно только через кейс модерации.

В карточке пользователя кнопка «Заблокировать пользователя» открывает форму создания служебного кейса:
- администратор указывает причину блокировки (код из справочника + свободный комментарий);
- система создаёт кейс типа `suspicious_activity` (если кейса нет) или открывает существующий;
- администратор применяет санкцию `user_block` в рамках этого кейса (модель действий администратора по кейсу — `07-platform/moderation.md`, раздел 8.3).

Прямая блокировка пользователя без кейса в первой версии не предусматривается. Это обеспечивает единый контур журналирования санкций (журнал кейса модерации, раздел 11 `07-platform/moderation.md`).

## 8. Раздел «Компании»

### 8.1. Поиск и фильтры
Экран поиска компаний предоставляет следующие фильтры:
- по идентификатору компании;
- по ИНН (точное совпадение);
- по наименованию (частичное совпадение);
- по типу компании (заготовитель, трейдер, переработчик);
- по состоянию (`pending_payment`, `demo`, `active`, `past_due`, `suspended`, `blocked`, `archived`);
- по тарифу (базовый, расширенный);
- по региону;
- по периоду регистрации.

Результат — список с краткой карточкой: ИНН, наименование, тип, состояние, тариф, дата регистрации, число сотрудников.

### 8.2. Карточка компании, сводка состояний
Карточка компании содержит:
- реквизиты компании (`subscriptions-and-billing.md`, раздел 3.13);
- состояние компании и историю переходов состояний;
- сводку по подписке (текущий тариф, период, состояние подписки, признак автопродления);
- ссылку на раздел «Биллинг» с фильтром по этой компании;
- список сотрудников с их ролями и состояниями;
- ссылки на активные и архивные объявления компании (для заготовителя), активные предложения (для трейдера, переработчика);
- ссылки на действующие санкции по компании (раздел 11.5);
- историю обращений в поддержку, связанных с компанией;
- кнопки операций: «Назначить демо-режим», «Перевести в archived», «Восстановить из archived», «Снять блокировку», «Снять приостановку».

Все кнопки операций работают только в применимых состояниях; недопустимые в данном состоянии операции скрываются.

### 8.3. Ручное назначение демо-режима
Администратор может назначить демо-режим компании в исключительных случаях. Сценарий покрывает ситуации, в которых пользовательский сценарий активации демо недоступен:
- компания была удалена и пользователь повторно регистрируется по тому же ИНН (демо однократен на ИНН — `auth-and-onboarding.md`, раздел 11.7; ручное назначение позволяет администратору отдельно одобрить повторное демо в обоснованных случаях);
- компания уже находится в состоянии, отличном от `pending_payment`, и требуется перевести её в `demo` для решения отдельного сценария (например, при расследовании жалобы).

Форма назначения демо-режима содержит:
- идентификатор компании;
- срок демо-режима в часах (по умолчанию двадцать четыре);
- обязательная причина (код из справочника + свободный комментарий).

При сохранении компания переводится в состояние `demo`, фиксируется дата окончания, действие записывается в `AdminActionLog`.

### 8.4. Ручной перевод состояний
Администратор может выполнить следующие переводы состояний компании в исключительных случаях:

| Из состояния | В состояние | Сценарий |
|---|---|---|
| `pending_payment` | `archived` | Компания не оплатила первую подписку и явно не намерена оплачивать |
| `active` | `archived` | По обращению собственника в поддержку (`subscriptions-and-billing.md`, раздел 16.2) |
| `past_due` | `archived` | По обращению собственника в поддержку |
| `suspended` | `archived` | По обращению собственника в поддержку |
| `archived` | `pending_payment` | Восстановление компании по обращению собственника, если автоматический сценарий восстановления (`subscriptions-and-billing.md`, раздел 16.4) недоступен по техническим причинам |
| `blocked` | `active` | Снятие блокировки администратором (раздел 11.5) |
| `suspended` | `active` | Внеплановое снятие приостановки в исключительных биллинговых сценариях (раздел 9.7) |

Все переводы фиксируются в `AdminActionLog` с обязательным указанием причины. Биллинговые последствия каждого перевода — `subscriptions-and-billing.md`.

Переводы из `archived` напрямую в `active`, минуя `pending_payment`, в первой версии не поддерживаются: восстановление архивной компании всегда сопровождается оформлением новой подписки.

Переводы в `blocked` и из `blocked` через раздел «Компании» не выполняются. Блокировка компании — санкция типа `company_block` через кейс модерации (раздел 11.4). Снятие блокировки — отдельная операция администратора в разделе «Действующие санкции по компании» (раздел 11.5).

### 8.5. Сотрудники компании со стороны администратора
Администратор в карточке компании видит список сотрудников с их ролями и состояниями. Доступные операции:
- просмотр карточки конкретного сотрудника (раздел 7.2);
- принудительный отзыв сессий конкретного сотрудника (раздел 7.4);
- инициирование блокировки сотрудника через кейс модерации (раздел 7.6).

Удаление сотрудника из компании, изменение ролей сотрудника, создание приглашений со стороны администратора в первой версии не предусматривается. Эти действия — функция собственника компании.

### 8.6. Журнал действий по компании
Журнал действий по компании — агрегированный срез по событиям, связанным с компанией:
- переходы состояний компании (с автором перехода);
- биллинговые события (выпущенные счета, успешные и неуспешные платежи, возвраты, изменения автопродления);
- модерационные события (созданные кейсы, применённые санкции);
- административные действия по компании из `AdminActionLog`;
- события приглашений и удалений сотрудников.

## 9. Раздел «Биллинг»

### 9.1. Тарифы и длительности — каталог
Раздел «Тарифы» содержит:
- список тарифов с базовой месячной стоимостью (`subscriptions-and-billing.md`, раздел 3.1);
- список длительностей периода с процентами скидки (раздел 3.2);
- кнопку «Изменить цену тарифа» (раздел 9.2);
- кнопку «Изменить процент скидки за длительность».

Изменение цены тарифа и изменения процентов скидки фиксируются в `AdminActionLog`. Деактивация тарифа в первой версии не предусматривается (на платформе всегда активны два тарифа).

### 9.2. Изменение цены тарифа
Изменение цены тарифа выполняется по правилам `subscriptions-and-billing.md`, раздел 15. Форма содержит:
- идентификатор тарифа;
- новая базовая месячная стоимость;
- дата вступления в силу (не ранее четырнадцати дней с даты сохранения, по правилам `subscriptions-and-billing.md`, раздел 15);
- свободный комментарий.

При сохранении:
- запись о новой цене фиксируется в каталоге тарифов с датой вступления в силу;
- система формирует уведомления `billing.plan.price_change_announced` для собственников и активных бухгалтеров всех компаний с действующим тарифом (`notifications.md`, раздел 6.2.12);
- действие фиксируется в `AdminActionLog`.

### 9.3. Промокоды
Раздел «Промокоды» содержит:
- список действующих и завершённых промокодов;
- кнопку «Создать промокод».

Форма создания промокода:
- код (строка);
- тип действия (процентная скидка либо фиксированная сумма скидки);
- размер скидки;
- ограничения по тарифу (один или несколько);
- ограничения по длительности (одна или несколько);
- срок действия (с — по);
- максимальное число применений (общее);
- максимальное число применений на одну компанию;
- свободный комментарий.

Состав атрибутов промокода — `subscriptions-and-billing.md`, раздел 3.9. Применение промокода компанией — `subscriptions-and-billing.md`, раздел 13.

Деактивация промокода до окончания срока действия — отдельная операция в карточке промокода. Изменение действующего промокода в первой версии запрещено: вместо изменения создаётся новый промокод и деактивируется старый.

### 9.4. Подписки компаний — просмотр и операции
В карточке компании (раздел 8.2) доступна сводка по подписке. Дополнительные операции в разделе «Биллинг»:
- просмотр истории подписок компании;
- просмотр истории платежей (`subscriptions-and-billing.md`, раздел 17.3);
- просмотр истории документов (раздел 17.5);
- инициирование разовой покупки от имени компании (для случаев исключительного администрирования) — в первой версии не предусматривается; разовые покупки оформляются только пользователем компании.

### 9.5. Ручная сверка безналичных платежей
Ручная сверка безналичных платежей применяется в случаях, когда автоматическая сверка по банковской выписке (`subscriptions-and-billing.md`, раздел 7.4) не сработала: платёж пришёл с отличающимся назначением, по неверным реквизитам, с задержкой банка или иной нештатной ситуацией.

Экран ручной сверки содержит:
- список неопознанных входящих платежей из банковской выписки;
- список выпущенных и неоплаченных счетов;
- инструмент «Сопоставить» — администратор связывает конкретный входящий платёж с конкретным неоплаченным счётом.

После сопоставления:
- счёт переводится в статус «оплачен»;
- инициируется выпуск кассового чека и закрывающего акта (`subscriptions-and-billing.md`, разделы 8.5 и 8.6);
- активируется или продляется подписка по правилам биллинга;
- фиксируется в `AdminActionLog` и в журнале биллинга.

Если неопознанный платёж не соответствует ни одному счёту, администратор может инициировать возврат (раздел 9.6) с пометкой «ошибочный платёж».

### 9.6. Возвраты
Возвраты инициирует администратор по обращению пользователя в поддержку (`support` категории «Биллинг») или в иных обоснованных случаях.

Форма возврата:
- ссылка на платёж, подлежащий возврату;
- сумма возврата (полная или частичная);
- причина возврата (код из справочника + свободный комментарий);
- ссылка на связанное обращение в поддержку (опционально, заполняется автоматически при инициировании из тикета).

Жизненный цикл возврата (`requested → approved → processing → completed | failed`) и состав сущности — `subscriptions-and-billing.md`, раздел 3.12 и далее. Отклонение возврата (`rejected`) администратор оформляет с обязательным комментарием.

При успешном возврате:
- сумма возвращается через платёжный шлюз или банк;
- выпускается корректирующий документ (`subscriptions-and-billing.md`, раздел 14);
- инициируется уведомление `billing.refund.completed` (`notifications.md`, раздел 6.2.14);
- если возврат связан с применённым промокодом — судьба промокода фиксируется отдельным действием (`subscriptions-and-billing.md`, раздел 13.3);
- действие фиксируется в `AdminActionLog`.

### 9.7. Ручные корректировки биллинга — закрытый перечень
Под «ручной корректировкой биллинга» понимается закрытый перечень административных операций над биллинговыми сущностями компании, не покрываемых стандартными пользовательскими сценариями:

- ручное продление подписки на заданный срок без оплаты — для исключительных случаев (например, компенсация перебоя на платформе; технический grace);
- ручная смена тарифа компании без доплаты — в исключительных случаях (например, по обращению пользователя в поддержку);
- ручное снятие приостановки `suspended` без оплаты;
- ручная отмена запланированной смены тарифа или длительности на следующий период;
- ручное прекращение действия разовой покупки — в исключительных случаях.

Каждая такая операция требует:
- обязательного указания причины (код из справочника + свободный комментарий);
- ссылки на связанное обращение в поддержку (опционально);
- фиксации в `AdminActionLog`.

Перечень операций — закрытый. Иные ручные корректировки в первой версии не предусматриваются. Открытие новых операций — только через расширение настоящего раздела отдельным согласованием.

### 9.8. Финансовые документы — повторный выпуск, повторная отправка
Администратор в карточке компании в разделе документов (`subscriptions-and-billing.md`, раздел 17.5) может:
- повторно отправить выпущенный документ на email компании (например, при сообщении пользователя о неполученном письме);
- повторно сгенерировать PDF документа из исходных данных, если файл был утерян по техническим причинам;
- инициировать выпуск корректирующего документа в рамках сценария возврата (раздел 9.6).

Изменение содержимого уже выпущенных документов в первой версии запрещено. При обнаружении ошибки в документе он закрывается корректирующим документом, новый документ выпускается с правильными данными.

## 10. Раздел «Уведомления»

### 10.1. Шаблоны и редактирование
Подраздел «Шаблоны» содержит список шаблонов уведомлений в разрезе событий каталога (`notifications.md`, раздел 6). Состав каждого шаблона — `notifications.md`, раздел 8.1.

Экран редактирования шаблона:
- тема (для email-канала);
- тело;
- список доступных переменных (фиксирован кодовым контрактом события, см. `notifications.md`, раздел 8.2);
- предпросмотр с подставленными значениями переменных-образцов;
- свободный комментарий к изменению.

При сохранении выполняется валидация шаблона (`notifications.md`, раздел 8.2). Шаблон, не прошедший валидацию, не сохраняется.

Изменение шаблона вступает в силу немедленно, без перезапуска сервиса (`notifications.md`, раздел 8.4). Действие фиксируется в журнале изменений шаблонов и в `AdminActionLog`.

### 10.2. Журнал изменений шаблонов и откат
Подраздел «Журнал шаблонов» содержит хронологический список всех изменений шаблонов:
- идентификатор шаблона;
- автор изменения;
- дата;
- предыдущая редакция (тема и тело);
- новая редакция;
- комментарий.

Доступна операция «Откатить к этой редакции» — она создаёт новое изменение, в котором тема и тело шаблона возвращаются к выбранной редакции. Откат не переписывает историю, а добавляется как отдельная запись (`notifications.md`, раздел 8.5).

Срок хранения журнала изменений — без ограничения (`notifications.md`, раздел 8.5).

### 10.3. Журнал доставок
Подраздел «Журнал доставок» содержит хронологический список попыток доставки уведомлений (`notifications.md`, раздел 14). Состав записи — `notifications.md`, раздел 3.4.

Доступные фильтры:
- по событию (код события);
- по получателю (пользователь или email);
- по компании получателя;
- по каналу (email, SMS, in-app);
- по статусу (успешно, неуспешно, dead-letter, no_recipients);
- по периоду.

Тела сообщений в журнале не сохраняются (`notifications.md`, раздел 14.2). Восстановление содержимого сообщения по необходимости поддержки выполняется реконструкцией: журнал хранит идентификатор шаблона и значения переменных, использованных при отправке.

Срок хранения журнала доставок — двенадцать календарных месяцев (`notifications.md`, раздел 14.3).

### 10.4. Dead-letter и его обработка
Подраздел «Dead-letter» содержит уведомления, не доставленные после исчерпания всех попыток (`notifications.md`, раздел 10.2). Доступные операции:
- повторная попытка доставки одного уведомления — для случаев, когда временная ошибка провайдера ошибочно классифицирована как постоянная;
- массовая повторная попытка по фильтру (например, все email-сообщения с временной ошибкой за последние двадцать четыре часа);
- закрытие уведомления как «не подлежит повтору» — фиксирует, что администратор рассмотрел запись и не намерен повторять отправку;
- выгрузка списка в формате таблицы для разбирательства вне платформы.

Все операции с dead-letter фиксируются в `AdminActionLog`.

### 10.5. Bounce и spam complaint
Подраздел «Bounce и spam complaint» содержит:
- список email-адресов с зафиксированными hard bounce (`notifications.md`, раздел 12.3);
- список email-адресов с зафиксированными spam complaint;
- сводный показатель частоты bounce и complaint по периодам (для контроля репутации отправителя).

Доступные операции:
- пометить адрес как «исправленный» (после проверки, что пользователь обновил email) — снимает запрет на отправку;
- открыть карточку пользователя — для связи с пользователем через поддержку с просьбой обновить email.

Систематические hard bounce по конкретному адресу администратор разрешает по обращению пользователя в поддержку.

### 10.6. Системные рассылки
Точка входа в раздел системных рассылок (раздел 17) дублируется в навигации админ-панели. С точки зрения логики системные рассылки — это самостоятельная подсистема, опирающаяся на каталог уведомлений; однако с точки зрения управления и журналов она тесно связана с подразделом уведомлений.

## 11. Раздел «Модерация»

### 11.1. Общая очередь
Экран общей очереди модерации показывает кейсы, доступные модераторам и администратору, в хронологическом порядке поступления (`07-platform/moderation.md`, раздел 6.1).

Состав строки очереди:
- идентификатор кейса;
- тип кейса (`complaint`, `suspicious_activity`);
- модерируемая сущность (тип объекта и краткое описание);
- автор сущности (компания и пользователь);
- число агрегированных жалоб;
- дата создания кейса;
- состояние (`open`, `in_review`, `escalated`, `closed_by_admin`, `resolved`);
- модератор, удерживающий lock (если есть);
- срок действия lock.

Действия:
- «Взять в работу» — установить lock на кейс (`07-platform/moderation.md`, раздел 6.3);
- «Открыть» — перейти к экрану кейса (раздел 11.2);
- «Освободить» — снять собственный lock без вынесения решения.

Фильтры в первой версии — по типу кейса, по типу модерируемой сущности, по состоянию. Сортировка — только по дате создания (по возрастанию). Расширенная фильтрация и приоритизация — за пределами первой версии (`07-platform/moderation.md`, раздел 16.2).

### 11.2. Экран кейса
Экран рассмотрения кейса содержит:
- модерируемую сущность с предпросмотром публичных и служебных полей;
- историю жалоб, агрегированных в кейс;
- историю решений по кейсу;
- внутренние заметки модераторов и администраторов;
- форму нового решения (тип решения, код причины из справочника, свободный комментарий);
- кнопку «Эскалировать администратору» с формой эскалации (`07-platform/moderation.md`, раздел 8.2);
- кнопку «Освободить кейс».

Полный состав решений и санкций модератора — `07-platform/moderation.md`, раздел 7. Применение санкций администратором — раздел 11.4 настоящего документа.

### 11.3. Служебная очередь администратора
Экран служебной очереди администратора показывает:
- кейсы в состоянии `escalated` (эскалированные модератором — `07-platform/moderation.md`, раздел 8);
- кейсы с запросом пересмотра (отдельный сценарий — модератор или администратор инициировал пересмотр ранее закрытого кейса; `07-platform/moderation.md`, раздел 8.4).

Действия администратора по кейсу из служебной очереди:
- применить предложенную модератором санкцию (`apply_admin_sanction`);
- применить иную санкцию;
- вернуть кейс модератору с комментарием;
- закрыть кейс без санкции.

### 11.4. Применение санкций администратором
Администратор может применить три санкции:
- ограничение доступа компании к модулю (`module_restriction`) — `07-platform/moderation.md`, раздел 8.5;
- блокировка пользователя (`user_block`) — раздел 9.1 модерационного документа;
- блокировка компании (`company_block`) — раздел 9.2.

Каждая санкция применяется на экране кейса. Формы санкций:

- **Ограничение модулем.** Поля: код модуля из справочника `restriction_modules` (`marketplace`, `reviews`, `comments` и иные); срок действия в днях (1 — 365); код причины; свободный комментарий.
- **Блокировка пользователя.** Поля: срок действия (бессрочно либо в днях; по умолчанию бессрочно); код причины из справочника `block_reasons`; свободный комментарий.
- **Блокировка компании.** Поля: код причины из справочника `block_reasons`; свободный комментарий. Срок не задаётся — блокировка действует до явной разблокировки.

Применение санкции фиксируется в журнале кейса, в журнале аутентификации (для блокировки пользователя — переход состояния), в биллинговом журнале (для блокировки компании — приостановка биллинга), в `AdminActionLog`.

### 11.5. Действующие санкции по компании
Экран действующих санкций по компании показывает:
- предупреждения, направленные компании (`warning`);
- ограничения модулем с оставшимся сроком (`module_restriction`);
- блокировки сотрудников компании (`user_block`);
- блокировку компании (`company_block`).

Действия:
- «Снять ограничение модулем досрочно» — `07-platform/moderation.md`, раздел 8.5;
- «Разблокировать пользователя» — раздел 9.1 модерационного документа;
- «Разблокировать компанию» — раздел 9.2 модерационного документа.

Каждое снятие фиксируется в журнале кейса (как `revert_decision`, если соответствующее решение пересматривается, либо как самостоятельное снятие санкции — `07-platform/moderation.md`, раздел 11) и в `AdminActionLog`.

При разблокировке компании ранее снятые модератором конкретные сущности не возвращаются автоматически (`07-platform/moderation.md`, раздел 9.2).

### 11.6. Отчёт по SLA модерации
Отчёт по SLA модерации (`07-platform/moderation.md`, раздел 6.4) показывает:
- среднее и медианное время первого решения по кейсам типа `complaint`;
- среднее и медианное время первого решения по кейсам типа `suspicious_activity`;
- долю кейсов, превысивших целевое время первого решения;
- количество кейсов в очереди по состояниям.

Период отчёта — настраиваемый. Доступ — администратору и модератору в режиме просмотра.

### 12.1. Зона работы контент-менеджера
Раздел «Контент» — это зона работы контент-менеджера. В первой версии раздел содержит точки входа в четыре контентных модуля:

- «База знаний» — `05-content/knowledge-base/`;

Состав экранов каждого модуля, workflow создания, редактирования и публикации материалов фиксируется в документации соответствующего модуля. Настоящий документ описывает только точку входа в раздел «Контент» и общие принципы.

### 12.2. Замечание о стабах модулей
В первой версии все четыре контентных модуля находятся на стадии стабов (`docs/05-content/...`). По мере утверждения каждого модуля admin-panel.md дополняется ссылкой на соответствующую спецификацию. До утверждения модуль присутствует в навигации как заглушка с информационным сообщением «Модуль ещё не утверждён».

Контент-менеджер в первой версии видит навигацию из четырёх модулей. Объём фактически доступных операций равен объёму, зафиксированному в утверждённых документах.

### 13.2. Справочник номенклатуры
Справочник номенклатуры (`nomenclature`) — каталог видов сырья, используемый модулем торговой площадки (`04-marketplace/...`), CRM заготовителя (`03-crm/...`) и иными модулями платформы.

Справочник поддерживается администратором платформы. Контент-менеджер в первой версии не имеет доступа к редактированию номенклатуры. Формулировки в `04-marketplace/roles.md`, раздел 1.8 и `04-marketplace/other-integrations.md`, раздел 1.1 согласованы с настоящим документом.

Состав записи справочника номенклатуры:
- код вида сырья;
- название;
- категория (при наличии);
- единица измерения;
- описание;
- признак активности.

Деактивация вида сырья не удаляет существующие объявления, предложения и записи в CRM, использующие этот код. Создание новых сущностей с деактивированным кодом запрещено.

### 13.3. Справочники модерации
Три справочника модерации поддерживаются администратором (`07-platform/moderation.md`, раздел 14.2):

- **`complaint_reasons`** — коды причин жалоб с поддержкой различения базовых кодов и кодов по типам объектов (`07-platform/moderation.md`, раздел 5.3). Атрибуты записи: код, название, тип объекта (или «базовый»), признак активности;
- **`decision_reasons`** — коды причин решений модератора и администратора (`07-platform/moderation.md`, раздел 7.4). Атрибуты записи: код, название, применимость к типам решений, признак активности;
- **`restriction_modules`** — коды модулей для санкции `module_restriction` (`07-platform/moderation.md`, раздел 8.5). Атрибуты записи: код модуля, название, признак активности.

Дополнительно:
- **`block_reasons`** — коды причин блокировок пользователей и компаний (раздел 11.4 настоящего документа). Атрибуты записи: код, название, применимость (`user_block`, `company_block`, оба), признак активности.

Модератор имеет доступ к справочникам модерации в режиме просмотра. Контент-менеджер не имеет доступа.

### 13.4. Справочники биллинга и административные справочники причин
Справочники, описывающие коды причин для административных действий:

- **`refund_reasons`** — коды причин возвратов (раздел 9.6);
- **`manual_billing_reasons`** — коды причин ручных корректировок биллинга (раздел 9.7);
- **`demo_grant_reasons`** — коды причин ручного назначения демо-режима (раздел 8.3);
- **`state_transition_reasons`** — коды причин ручного перевода состояний компании (раздел 8.4).

Состав записи каждого справочника: код, название, признак активности.

### 13.5. Справочник регионов
Справочник регионов (`regions`) — каталог субъектов Российской Федерации, используемый платформой для:
- значений в фильтре «регион» на торговой площадке (`04-marketplace/public-ui.md`, раздел 1.4);
- сопоставления адреса с региональным признаком при сохранении адреса (`08-architecture/geo-logic.md`, раздел 5.2);
- агрегаций и аналитики платформы по региональному признаку.

Справочник поддерживается администратором платформы. Контент-менеджер не имеет доступа к редактированию.

Состав записи справочника регионов:
- код региона;
- название;
- признак активности.

Включение KLADR/ФИАС-идентификатора региона в состав записи — открытый вопрос (`08-architecture/geo-logic.md`, раздел 13.5) и в первой версии не является обязательным.

Деактивация записи региона не удаляет существующие адреса, ссылающиеся на этот код. Создание новых адресов с деактивированным кодом региона блокируется на этапе сопоставления (`08-architecture/geo-logic.md`, раздел 5.2).

### 13.6. Поддержка справочников
Экран поддержки конкретного справочника:
- список действующих и неактивных записей;
- кнопка «Добавить запись»;
- операции по существующей записи: «Редактировать», «Деактивировать», «Активировать».

Удаление записей справочников запрещено: записи могут быть только деактивированы. Это обеспечивает целостность исторических данных, использующих эти коды.

Все операции со справочниками фиксируются в `AdminActionLog`.

## 14. Раздел «Поддержка»

### 14.1. Назначение модуля
Модуль поддержки — двусторонний канал переписки между пользователем компании и администратором платформы. Модуль реализован средствами платформы и не использует внешних каналов (электронная почта, мессенджер). Уведомления о новых сообщениях поступают по тем же каналам, что и иные уведомления (`07-platform/notifications.md`).

В первой версии операторами поддержки выступают администраторы платформы. Отдельной роли «оператор поддержки» не вводится.

### 14.2. Категории обращений
В первой версии поддерживается закрытый список категорий обращений:

| Код | Название | Назначение |
|---|---|---|
| `billing` | Биллинг | Оплата, документы, возвраты, реквизиты плательщика, неуспешная сверка безнала |
| `moderation_review` | Пересмотр решения модерации | Несогласие пользователя с решением модератора по жалобе или с применённой санкцией (`07-platform/moderation.md`, раздел 2.8) |
| `company_management` | Управление компанией | Передача прав собственника, архивирование компании по инициативе собственника, смена типа компании, исправление ИНН после регистрации |
| `technical` | Технический вопрос | Ошибка в работе платформы, проблема со входом, не открывается экран, ошибка интеграции |
| `data_deletion` | Удаление персональных данных | Запросы по 152-ФЗ |
| `other` | Иное | Иные обращения, в том числе обратная связь по платформенному контенту |

Список категорий — закрытый. Изменение состава категорий — отдельным согласованием.

### 14.3. Сущности модуля поддержки
Состав сущностей — раздел 3:
- `SupportTicket` — обращение;
- `SupportTicketMessage` — сообщение в обращении;
- `SupportTicketAttachment` — вложение в сообщении.

### 14.4. Кто может создавать обращения
Обращение может создать любой авторизованный пользователь компании, в любой категории. Ограничений по роли пользователя в компании на создание обращений нет.

При обработке обращения администратор проверяет, что роль автора обращения соответствует запрашиваемому действию. Например, при обращении категории `company_management` с запросом «передать права собственника» автор обращения должен быть собственником; в противном случае администратор отвечает «недостаточно прав» и закрывает обращение.

Платформенные сотрудники не создают обращения в поддержку.

### 14.5. Жизненный цикл обращения
Обращение проходит через состояния:

- `new` — создано пользователем, ни один администратор ещё не работал;
- `in_progress` — взято в работу администратором (фиксируется ведущий администратор), идёт переписка;
- `awaiting_user` — администратор задал уточняющий вопрос, ожидается ответ пользователя;
- `resolved` — администратор пометил обращение как решённое;
- `closed` — обращение окончательно закрыто.

Переходы:

```
new ─── взято в работу ──▶ in_progress
in_progress ─── задан уточняющий вопрос ──▶ awaiting_user
awaiting_user ─── ответ пользователя ──▶ in_progress
in_progress ─── решено ──▶ resolved
resolved ─── автозакрытие через 14 дней ──▶ closed
resolved ─── ответ пользователя ──▶ in_progress
awaiting_user ─── 14 дней без ответа ──▶ closed
```

Срок четырнадцать дней является параметром платформенной конфигурации (раздел 18.1).

### 14.6. Точка входа со стороны пользователя
В кабинете пользователя предусмотрены:
- раздел «Поддержка» в навигации кабинета;
- кнопка «Создать обращение» с формой: категория, тема, текст, вложения;
- список собственных обращений пользователя с фильтрами по категории и состоянию;
- экран конкретного обращения с историей переписки и формой нового сообщения;
- быстрые точки входа из других разделов кабинета: на экране биллинга — «Связаться с поддержкой по биллингу», на экране кейса (для пользователя — на экране сущности под санкцией) — «Связаться с поддержкой по решению модерации» (с автозаполнением категории и связи с кейсом).

Видимость раздела «Поддержка» в кабинете пользователя — для всех пользователей всех ролей внутри активной компании. В состоянии `archived` доступ к разделу сохраняется только у собственника. В состоянии `blocked` доступ к разделу закрыт.

Конкретные экраны кабинета пользователя в части поддержки фиксируются настоящим документом и не дублируются в иных местах.

### 14.7. Точка входа со стороны администратора
В админ-панели в разделе «Поддержка» администратор видит:
- очередь новых и в работе обращений с фильтрами по категории, состоянию, ведущему администратору;
- экран конкретного обращения с историей переписки;
- форму нового сообщения с возможностью добавить вложение;
- форму служебной заметки — сообщение, видимое только платформенным сотрудникам, не пользователю (`SupportTicketMessage` с признаком служебной заметки);
- кнопки переходов состояний: «Взять в работу», «Запросить уточнение», «Пометить решённым», «Закрыть»;
- ссылки на связанные сущности: компания автора, кейс модерации, счёт, возврат.

При первом действии администратора по обращению (взятие в работу) обращение получает ведущего администратора. Передача обращения другому администратору выполняется явным действием — операция фиксируется в `AdminActionLog`.

Модератор и контент-менеджер не имеют доступа к разделу «Поддержка». При необходимости вовлечь модератора в обращение категории `moderation_review` администратор инициирует пересмотр кейса через служебную очередь администратора в разделе «Модерация» (раздел 11.3); пользователь и модератор не контактируют напрямую через тикет.

### 14.8. Связь обращения с подсистемами
Обязательные привязки:
- категория `billing` — связь с компанией обязательна; связь с конкретным счётом, подпиской, возвратом — опциональна;
- категория `moderation_review` — связь с кейсом модерации обязательна;
- категория `company_management` — связь с компанией обязательна;
- категория `data_deletion` — связь с компанией и пользователем-заявителем обязательна;
- категории `technical` и `other` — без обязательной привязки.

При создании обращения пользователем привязка к компании заполняется автоматически (компания автора). Привязки к счёту, кейсу, подписке, возврату — автоматически при инициировании обращения из соответствующего экрана; вручную пользователь связи между обращениями и сущностями не устанавливает.

Администратор может расширить или скорректировать привязки обращения вручную в любой момент.

### 14.9. Вложения
Сообщения в обращении могут содержать вложения. Лимиты:
- до десяти мегабайт на один файл;
- до пяти файлов на одно сообщение;
- допустимые типы — изображения (JPG, PNG, WEBP), PDF, офисные форматы (DOCX, XLSX), текстовые файлы (TXT, CSV).

Конкретные требования к хранилищу, антивирусной проверке, ограничениям имени файла, поведению при детекте угрозы и при ошибке проверки — `07-platform/storage.md`, раздел 5.10 (вложения в обращениях поддержки) и разделы 7, 10. Настоящий документ фиксирует только продуктовые лимиты в контексте модуля поддержки.

Срок хранения вложений равен сроку хранения обращения (бессрочно — раздел 14.11).

### 14.10. Уведомления по обращениям
Подсистема уведомлений (`07-platform/notifications.md`) после применения сопутствующих правок поддерживает категорию «поддержка» с пятью событиями:

| Код события | Триггер | Получатель |
|---|---|---|
| `support.ticket.created` | Пользователь создал обращение | все администраторы платформы |
| `support.ticket.replied_by_admin` | Администратор отправил сообщение пользователю | автор обращения |
| `support.ticket.replied_by_user` | Пользователь отправил сообщение | ведущий администратор |
| `support.ticket.resolved` | Администратор пометил обращение решённым | автор обращения |
| `support.ticket.reopened` | Обращение возвращено в работу (ответ пользователя в статусе `resolved`) | ведущий администратор |

Категория «поддержка» — отключаемая, по умолчанию включена. Каналы — email и in-app. Адресация и шаблоны — `07-platform/notifications.md` после применения сопутствующих правок.

Уведомление `support.ticket.created` направляется всем активным администраторам, поскольку до взятия в работу у обращения нет ведущего администратора. Последующие уведомления `support.ticket.replied_by_user` и `support.ticket.reopened` направляются только ведущему администратору.

### 14.11. Журналирование действий
Действия по обращению фиксируются в журнале обращения:
- создание обращения;
- взятие в работу (с указанием администратора);
- передача обращения другому администратору;
- каждое сообщение пользователя и администратора (со ссылкой на сообщение в `SupportTicketMessage`);
- служебная заметка администратора;
- запрос уточнения (`in_progress → awaiting_user`);
- ответ пользователя (`awaiting_user → in_progress`);
- решение (`in_progress → resolved`);
- закрытие (`resolved → closed`);
- автозакрытие по таймауту (`awaiting_user → closed`).

Срок хранения обращений и их журнала — без ограничения. Обращения и их сообщения не подлежат жёсткому удалению, кроме случаев, оформленных через категорию `data_deletion` отдельным решением администратора в рамках исполнения требований 152-ФЗ.

Действия администратора, влияющие на состояние обращения (взятие в работу, передача, закрытие), дополнительно фиксируются в `AdminActionLog`.

### 14.12. SLA в первой версии
В первой версии жёстких целевых показателей (SLA) для обращений нет. Целевые ориентиры по времени первого ответа администратора и по времени решения обращения по категориям могут быть введены в последующих версиях. Жёсткого механизма штрафов и автоматических эскалаций по таймауту нет.

Превышение нерегламентированного «разумного» времени ответа фиксируется в журнале обращений (по фактическим датам сообщений) и доступно администратору в виде свободной выгрузки.

## 15. Раздел «Журналы» — единый интерфейс

### 15.1. Срез по подсистемам
Раздел «Журналы» предоставляет единый интерфейс просмотра разрозненных доменных журналов с фильтром по подсистеме:

| Подсистема | Источник записей |
|---|---|
| Аутентификация | журналы аутентификации (`auth-and-onboarding.md`, раздел 13.5) |
| Биллинг | журналы биллинга и платежей (`subscriptions-and-billing.md`) |
| Уведомления — доставки | `notifications.md`, раздел 14 |
| Уведомления — изменения шаблонов | `notifications.md`, раздел 8.5 |
| Модерация | журнал кейсов модерации (`07-platform/moderation.md`, раздел 11) |
| Поддержка | журналы обращений (раздел 14.11) |
| Административные действия | `AdminActionLog` (раздел 16) |

Каждая запись имеет указание подсистемы, типа события, актора, даты, связанной сущности.

### 15.2. Срез по сущности
Кроме среза по подсистеме, доступен срез по конкретной сущности — все записи журналов всех подсистем, относящиеся к этой сущности:
- по компании;
- по пользователю;
- по платформенному сотруднику;
- по кейсу модерации;
- по обращению в поддержку;
- по подписке, счёту, возврату.

Срез по сущности — это сборка из всех доменных журналов с автоматической фильтрацией по идентификатору сущности.

### 15.3. Срез по администратору
Срез по конкретному платформенному сотруднику показывает все записи всех журналов, в которых сотрудник выступает актором. Источники — `AdminActionLog`, журнал кейсов модерации, журнал изменений шаблонов уведомлений, журналы обращений.

Модератор и контент-менеджер имеют доступ к собственному срезу — для самопроверки и подготовки отчётов о собственной работе. Доступ к срезу другого сотрудника модератору и контент-менеджеру закрыт.

### 15.4. Доступ
Доступ к разделу «Журналы»:
- администратор — полный доступ ко всем журналам, всем срезам;
- модератор — собственный срез по своей деятельности;
- контент-менеджер — собственный срез по своей деятельности.

Сроки хранения каждого журнала определяются доменным документом подсистемы. Настоящий документ не вводит новых сроков хранения для доменных журналов.

## 16. Административный журнал (AdminActionLog)

### 16.1. Принцип: журналируется значимое
`AdminActionLog` — единый верхнеуровневый журнал значимых действий платформенных сотрудников в админ-панели. В журнал попадают только действия из закрытого перечня (раздел 16.2). Просмотры экранов, открытие карточек, технические переходы между разделами админ-панели в журнал не попадают.

`AdminActionLog` не заменяет доменные журналы (журнал кейсов модерации, журнал изменений шаблонов уведомлений, биллинговые журналы). Доменное действие, попадающее в `AdminActionLog`, обычно также фиксируется в доменном журнале своей подсистемы. `AdminActionLog` обеспечивает единый сводный взгляд на действия администраторов поверх разнородных доменных журналов.

### 16.2. Перечень значимых действий
Действия, подлежащие обязательной фиксации в `AdminActionLog`:

**Платформенные сотрудники:**
- создание платформенного сотрудника;
- блокировка и разблокировка платформенного сотрудника;
- удаление платформенного сотрудника;
- передача обращения в поддержку другому администратору;
- принудительный отзыв сессий платформенного сотрудника.

**Пользователи:**
- принудительный отзыв сессий пользователя (раздел 7.4);
- инициирование блокировки пользователя через создание служебного кейса (раздел 7.6);
- отмена приглашения сотрудника собственником компании выполняется собственником и администратором не журналируется в `AdminActionLog`;
- отмена приглашения администратором (раздел 7.5).

**Компании:**
- ручное назначение демо-режима (раздел 8.3);
- ручной перевод состояний компании (раздел 8.4);
- разблокировка компании (раздел 11.5).

**Биллинг:**
- изменение цены тарифа (раздел 9.2);
- изменение процента скидки за длительность;
- создание, деактивация промокода (раздел 9.3);
- ручная сверка безналичных платежей (раздел 9.5);
- инициирование, одобрение, отклонение возврата (раздел 9.6);
- ручная корректировка биллинга (раздел 9.7);
- повторный выпуск, повторная отправка финансового документа (раздел 9.8);
- изменение конфигурационного параметра биллинга в служебных настройках.

**Уведомления:**
- изменение шаблона уведомления (раздел 10.1);
- откат шаблона к предыдущей редакции (раздел 10.2);
- повторная попытка доставки из dead-letter, закрытие записи dead-letter (раздел 10.4);
- пометка email-адреса как «исправленный» (раздел 10.5);
- инициирование системной рассылки (раздел 17).

**Модерация:**
- применение санкции администратором (раздел 11.4);
- снятие санкции (раздел 11.5);
- пересмотр ранее закрытого кейса (`07-platform/moderation.md`, раздел 8.4) — фиксируется в журнале кейса; в `AdminActionLog` дублируется как сводное действие.

**Справочники:**
- добавление, редактирование, деактивация, активация записи справочника (раздел 13.5).

**Поддержка:**
- изменение состояния обращения администратором (`new → in_progress`, переход в `resolved`, переход в `closed`, передача обращения другому администратору) — раздел 14.11.

**Служебные настройки:**
- изменение конфигурационного параметра платформы (раздел 18).

Перечень — закрытый. Расширение перечня — отдельным согласованием.

### 16.3. Состав записи
Состав записи `AdminActionLog`:
- идентификатор записи;
- идентификатор актора (платформенный сотрудник);
- роли актора на момент действия;
- тип действия (из перечня раздела 16.2);
- объект действия (тип и идентификатор; например, «компания, ID 12345», «тариф, ID 1», «обращение в поддержку, ID 9876»);
- параметры действия — структурированные «ключ — значение»; например, для «изменение цены тарифа» — «прежняя цена», «новая цена», «дата вступления в силу»;
- свободный комментарий актора (при наличии — обязательно для большинства действий, см. формы операций);
- дата и время действия.

Параметры действия дают возможность реконструировать суть выполненной операции по записи журнала без обращения к доменным журналам.

### 16.4. Срок хранения — бессрочно
`AdminActionLog` хранится бессрочно. Записи журнала не подлежат удалению ни при каких обстоятельствах, в том числе:
- при удалении компании, фигурирующей в записи как объект;
- при удалении платформенного сотрудника, фигурирующего в записи как актор;
- при удалении пользователя.

При удалении сотрудника или пользователя запись сохраняет идентификатор актора и идентификатор объекта; восстановление детальной информации об актёре или объекте может быть невозможным после удаления, но факт действия и его параметры остаются в журнале.

Конкретные технические решения для бессрочного хранения (партиционирование, архивные таблицы, S3) — `08-architecture/...`. Настоящий документ фиксирует только продуктовое требование.

### 16.5. Доступ
Доступ к `AdminActionLog`:
- администратор — полный доступ;
- модератор — нет;
- контент-менеджер — нет.

Срез по конкретному платформенному сотруднику (раздел 15.3) предоставляет ограниченный доступ модератору и контент-менеджеру к собственным действиям в `AdminActionLog`. Доступ модератора и контент-менеджера к действиям других платформенных сотрудников через срезы и через прямой доступ к журналу закрыт.

## 17. Системные рассылки

### 17.1. Аудитории и фильтры
Системная рассылка — однократная массовая отправка сообщения по аудитории, выбранной администратором. В первой версии поддерживаются следующие фильтры аудитории:

**По охвату:**
- все пользователи всех активных компаний;
- все собственники активных компаний.

**По типу компании:**
- собственники компаний типа «заготовитель»;
- собственники компаний типа «трейдер»;
- собственники компаний типа «переработчик».

**По роли получателя:**
- бухгалтеры всех компаний;
- бухгалтеры компаний типа «заготовитель»;
- бухгалтеры компаний типа «трейдер»;
- бухгалтеры компаний типа «переработчик».

**По тарифу:**
- собственники компаний с базовым тарифом;
- собственники компаний с расширенным тарифом.

**По состоянию компании:**
- собственники компаний в состоянии `active`;
- собственники компаний в состоянии `past_due`;
- собственники компаний в состоянии `suspended`;
- собственники компаний в состоянии `archived`.

**Точечная адресация:**
- одна конкретная компания (по ИНН) и её сотрудники.

Несколько фильтров могут комбинироваться по принципу пересечения (например, «собственники компаний типа „заготовитель“ с расширенным тарифом в состоянии `active`»).

Пользователи в состоянии `blocked` или `removed_from_company` исключаются из адресации автоматически на момент отправки.

### 17.2. Каналы
Каналы системной рассылки — email и in-app. SMS-канал для системных рассылок не используется (`07-platform/notifications.md`, раздел 2.3).

В одной рассылке могут быть выбраны оба канала; в этом случае сообщение отправляется по обоим каналам каждому получателю.

### 17.3. Категория и адресация
Все системные рассылки относятся к категории «системные» подсистемы уведомлений (`07-platform/notifications.md`, раздел 5).

Категория «системные» — отключаемая, по умолчанию включена. Пользователи, отключившие категорию «системные» в выбранном канале, исключаются из адресации рассылки в этом канале на момент отправки. Получатели, отключившие все каналы для категории «системные», в рассылку не попадают.

### 17.4. Лимиты и ограничения первой версии
**Лимиты:**
- максимальный размер тела сообщения — десять тысяч символов;
- максимальный размер темы (для email) — двести символов;
- одновременно может выполняться не более одной активной рассылки; новая рассылка может быть инициирована только после завершения предыдущей.

**Ограничения первой версии:**
- персонализация переменных в шаблоне рассылки не поддерживается; тело и тема — статический текст;
- отложенная по расписанию рассылка не поддерживается; рассылка инициируется немедленно;
- отмена в процессе отправки не поддерживается;
- A/B-тестирование, сегментация, динамическое формирование аудитории — за пределами первой версии.

### 17.5. Журналирование
Каждая системная рассылка фиксируется в `SystemBroadcast` (раздел 3.8). Состав фиксируемых данных:
- автор рассылки;
- фильтр аудитории;
- каналы;
- тема и тело;
- дата создания;
- дата начала отправки;
- дата завершения;
- число получателей в момент формирования;
- число успешно доставленных и число неуспешных по каналам.

Записи в журнале доставок (`notifications.md`, раздел 14) формируются для каждого фактического получателя по каждому каналу, как и для иных типов уведомлений.

Инициирование рассылки фиксируется в `AdminActionLog`.

## 18. Служебные настройки платформы

### 18.1. Конфигурационные параметры
Раздел «Служебные настройки» содержит редактируемые параметры платформенной конфигурации. В первой версии в разделе зафиксированы:

**Биллинг:**
- режим НДС платформы (`subscriptions-and-billing.md`, раздел 21.1) — «без НДС» либо «с НДС» с указанием ставки;
- форма закрывающего документа в режиме «с НДС» (`subscriptions-and-billing.md`, раздел 21.2) — «акт + счёт-фактура» либо «УПД»;
- срок пребывания в состоянии `archived` до жёсткого удаления (`subscriptions-and-billing.md`, раздел 16.3) — по умолчанию двенадцать месяцев;
- процент скидки за длительность периода — отдельный параметр для каждой длительности (1, 3, 6, 12 месяцев).

**Поддержка:**
- срок автозакрытия обращения из `awaiting_user` без ответа пользователя — по умолчанию четырнадцать дней;
- срок автозакрытия обращения из `resolved` без ответа пользователя — по умолчанию четырнадцать дней.

**Уведомления:**
- максимальное число попыток повтора доставки (`notifications.md`, раздел 10.2) — по умолчанию три;
- задержки между попытками — по умолчанию одна минута, пять минут, тридцать минут.

**Аутентификация:**
- лимиты SMS-кодов на один номер за период (`auth-and-onboarding.md`, раздел 13.2);
- срок жизни приглашения сотрудника — по умолчанию семь дней;
- срок жизни одноразового кода SMS;
- срок жизни одноразовой ссылки подтверждения email.

**Модерация:**
- максимальное число одновременно удерживаемых модератором lock-ов (`07-platform/moderation.md`, раздел 6.3) — по умолчанию три;
- срок действия lock-а на кейс — по умолчанию пятнадцать минут;
- срок защитного окна отзыва (`07-platform/moderation.md`, раздел 10) — по умолчанию три минуты;
- целевое время первого решения по кейсу типа `complaint` (раздел 6.4 модерационного документа) — по умолчанию сорок восемь часов;
- целевое время первого решения по кейсу типа `suspicious_activity` — по умолчанию двадцать четыре часа.

Каждый параметр имеет код, описание, тип значения, текущее значение, дату последнего изменения, автора последнего изменения. Изменение параметра фиксируется в `AdminActionLog`.

Часть параметров (например, режим НДС) при изменении влияет на доменное поведение начиная с даты изменения — например, документы, выпускаемые после изменения, формируются по новым правилам, а документы, выпущенные ранее, остаются в исторической форме.

### 18.2. Журнал изменений настроек
История изменений конфигурационных параметров доступна на экране каждого параметра — список всех изменений с указанием прежнего значения, нового значения, автора и даты. Журнал изменений настроек хранится бессрочно (через `AdminActionLog`).

## 19. Аналитика первой версии

### 19.1. SLA модерации
Отчёт по SLA модерации (`07-platform/moderation.md`, раздел 6.4) — раздел 11.6 настоящего документа. Доступен администратору и модератору.

### 19.2. Dead-letter уведомлений
Сводка по dead-letter (`notifications.md`, раздел 10.2) — число записей по периодам, по событиям, по каналам. Доступна администратору в подразделе «Dead-letter» (раздел 10.4).

### 19.3. Bounce и spam complaint
Сводка по bounce и spam complaint (`notifications.md`, раздел 12.3) — число и доля по периодам, по событиям. Доступна администратору в подразделе «Bounce и spam complaint» (раздел 10.5).

### 19.4. Сводка по биллингу
Минимальная сводка по биллингу:
- число активных подписок по тарифам;
- число активных компаний по типам и состояниям;
- число платежей за период (успешные, неуспешные, возвраты);
- сумма поступлений за период по способам оплаты.

Доступна администратору. Период сводки — настраиваемый.

### 19.5. Что в аналитику не входит
В первой версии аналитический контур админ-панели ограничен пунктами 19.1 — 19.4. Не входят и подлежат рассмотрению в последующих версиях:
- продуктовая аналитика поведения пользователей внутри кабинета (воронки, удержание, конверсия);
- маркетинговые отчёты;
- финансовая отчётность платформы (P&L, юнит-экономика);
- аналитика по торговой площадке (число активных объявлений, среднее время от подачи до сделки, среднее число предложений на объявление);
- аналитика по CRM-контуру компаний (объёмы поступлений сырья, оборачиваемость склада);
- аналитика по контентному модулю (просмотры, лайки, комментарии);
- экспорт аналитических данных в форматы BI-инструментов.

Внутренние метрики компании (CRM, торговля, склад) находятся в кабинете компании, а не в админ-панели.

## 21. Открытые вопросы и ограничения первой версии

### 21.1. Отдельная роль «оператор поддержки» — за пределами первой версии
В первой версии операторами поддержки выступают администраторы. Отдельная роль «оператор поддержки» с урезанным набором прав и закрепляемой специализацией по категориям обращений — вопрос последующих версий.

### 21.2. SLA по обращениям в поддержку — за пределами первой версии
В первой версии целевых показателей по времени первого ответа и времени решения обращений нет. Введение SLA, автоматических эскалаций по таймауту, дашбордов SLA — вопрос последующих версий.

### 21.3. Иерархия администраторов — за пределами первой версии
Все администраторы платформы равноправны. Введение «суперадминистратора» с эксклюзивным правом создавать других администраторов, либо разделение администраторов по доменам ответственности (биллинг, модерация, инфраструктура) — вопрос последующих версий.

### 21.4. Технический журнал доступа к экранам — за пределами первой версии
В первой версии не ведётся журнал входов в админ-панель и журнал просмотров отдельных экранов. Введение такого журнала и требования к нему — вопрос последующих версий.

### 21.5. Расширенные фильтры аудитории системных рассылок — за пределами первой версии
В первой версии фильтры аудитории — те, что зафиксированы в разделе 17.1. Расширения (по региону, по дате регистрации, по сегментам поведения, по комбинированным предикатам) — вопрос последующих версий.

### 21.6. Персонализация тела системной рассылки — за пределами первой версии
В первой версии тело системной рассылки — статический текст без подстановок переменных. Персонализация (например, обращение по имени, подстановка реквизитов компании) — вопрос последующих версий.

### 21.7. Отложенная и плановая отправка системной рассылки — за пределами первой версии
В первой версии рассылка отправляется немедленно. Возможность задать дату и время отправки, отменить рассылку до начала отправки — вопрос последующих версий.

### 21.8. Расширенная аналитика админ-панели — за пределами первой версии
Перечень аналитических разделов первой версии зафиксирован разделом 19. Расширения (продуктовая аналитика, маркетинговые отчёты, финансовая отчётность, аналитика по торговой площадке и CRM, экспорт в BI-инструменты) — вопрос последующих версий.

### 21.9. Прямая блокировка пользователя без кейса модерации — за пределами первой версии
В первой версии блокировка пользователя оформляется только через кейс модерации (раздел 7.6). Альтернативный путь «быстрой блокировки без кейса» (например, для случаев очевидной автоматической атаки) — вопрос последующих версий.

### 21.10. Создание приглашений администратором — за пределами первой версии
В первой версии создание приглашений сотрудников — функция собственника компании. Возможность администратора пригласить сотрудника от имени собственника (например, по обращению в поддержку, когда собственник недоступен) — вопрос последующих версий.

### 21.11. Изменение и удаление действующих промокодов — за пределами первой версии
В первой версии изменение действующего промокода запрещено; вместо изменения создаётся новый промокод и деактивируется старый. Возможность редактирования действующих промокодов — вопрос последующих версий.

### 21.12. Регламент возврата средств и судьба промокодов
Окончательный регламент возвратов в исключительных случаях (например, ошибочный перевод, частичное возмещение услуги) и политика учёта применённых промокодов при возврате (`subscriptions-and-billing.md`, раздел 21.9) — фиксируются в рабочих процедурах поддержки. Настоящий документ описывает только интерфейс инициирования возврата.

### 21.13. Перевод обращений между категориями — за пределами первой версии
В первой версии категория обращения, заданная пользователем при создании, не меняется. Возможность администратора переклассифицировать обращение (например, обращение, поданное как «Иное», но фактически относящееся к биллингу) — вопрос последующих версий.
