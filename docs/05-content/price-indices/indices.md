## 1. Назначение раздела
Настоящий документ фиксирует состав сущностей модуля «Индексы цен», их атрибуты, связи и поведение. Документ продолжает [overview.md](./overview.md) и развивает разделы 5, 6 и 8 шапки.

Документ определяет:
- состав полей сущностей `NomenclatureCategory`, `PriceIndex`, `PriceIndexValue`;
- связь с платформенным справочником номенклатуры (`07-platform/admin-panel.md`, раздел 13.2);
- алгоритмы расчёта текущей цены, недельного изменения и тренда;
- правила работы с историческими значениями (добавление, редактирование, удаление);
- правила валидации перед публикацией;
- каскадные эффекты при удалении.

Состав поля для модели данных подлежит окончательной фиксации в `08-architecture/data-model.md`. Настоящий документ задаёт продуктовый набор полей.

## 2. NomenclatureCategory — категория номенклатуры

### 2.1. Назначение
Справочный элемент: группа номенклатур, выступающая вкладкой в публичной части индексов и используемая для группировки видов сырья на торговой площадке. Появляется как сопутствующая правка к `07-platform/admin-panel.md`, раздел 13.2 — продвижение поля «категория» из текстового атрибута номенклатуры в первоклассную сущность.

Подробности сопутствующей правки — [other-integrations.md](./other-integrations.md), раздел 4.

### 2.2. Состав полей
- `id` — идентификатор;
- `name` — отображаемое название категории (строка, обязательное; уникальное в пределах платформы по нормализованной форме);
- `slug` — фрагмент URL для прямой ссылки на вкладку (`/indices/category/{slug}`); генерируется автоматически из `name` транслитерацией; уникальный;
- `position` — порядковый номер вкладки в публичном интерфейсе (целое число; уникальный в пределах справочника);
- `isActive` — признак активности (булево; по умолчанию `true`);
- `createdAt` — момент создания категории;
- `createdBy` — идентификатор актора, создавшего категорию (администратор).

### 2.3. Связь с номенклатурой
Связь между `Nomenclature` и `NomenclatureCategory` — many-to-one: одна номенклатура принадлежит одной категории; одна категория содержит произвольное число номенклатур.

В сопутствующей правке к `admin-panel.md, раздел 13.2`: поле `nomenclature.category` (текст) заменяется на `nomenclature.categoryId` (ссылка на `NomenclatureCategory`). Миграция существующих данных — [other-integrations.md](./other-integrations.md), раздел 4.4.

### 2.4. Категория по умолчанию
В первой версии номенклатура обязательно принадлежит какой-либо категории. Категория «Без группы» или нулевая категория не предусматривается. При создании новой номенклатуры администратор выбирает категорию из выпадающего списка существующих или создаёт новую категорию через тот же диалог.

### 2.5. Управление порядком
Порядок вкладок в публичной части определяется полем `position`. Администратор управляет порядком через перетаскивание (drag-and-drop) на экране управления категориями ([cabinet-content-manager.md](./cabinet-content-manager.md), раздел 4).

При перетаскивании пересчитываются поля `position` затронутых категорий для сохранения уникальности и непрерывности номеров.

### 2.6. Деактивация категории
Категория с `isActive = false`:
- не отображается как вкладка в публичной части индексов;
- скрывается из выпадающих списков выбора категории при редактировании номенклатуры;
- сохраняет связи с номенклатурами (номенклатуры не меняют свой `categoryId`).

При активации категория снова становится вкладкой; её номенклатуры с опубликованными индексами появляются в публичной части.

## 3. PriceIndex — индекс цены

### 3.1. Назначение
Сущность, связывающая конкретную номенклатуру с историей значений её цены. Один индекс — для одной номенклатуры. У одной номенклатуры — не более одного индекса.

### 3.2. Состав полей
- `id` — идентификатор;
- `nomenclatureId` — ссылка на номенклатуру (one-to-one; уникальная);
- `description` — необязательное служебное описание индекса для контент-менеджера и администратора (строка; информационное поле, в публичной части не отображается);
- `status` — состояние публикации: `draft`, `published`;
- `firstPublishedAt` — момент первой публикации (заполняется автоматически);
- `lastUpdatedAt` — момент последней редакции (заполняется автоматически при изменении значений или метаданных);
- `createdBy` — идентификатор контент-менеджера, создавшего индекс;
- `values` — упорядоченный по дате список значений (раздел 4).

### 3.3. Состояния публикации
- `draft` — индекс виден только в админ-панели; в публичной части на вкладке группы не отображается;
- `published` — индекс виден всем пользователям с активной подпиской и состоянием компании, разрешающим работу с функциональными разделами.

Переходы:
- `draft → published` — выполняется контент-менеджером или администратором;
- `published → draft` — выполняется только администратором.

Полные правила переходов и валидация перед публикацией — раздел 6 настоящего документа и [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 7.

### 3.4. Связь с номенклатурой
Каждый индекс относится к ровно одной номенклатуре. Номенклатура может иметь индекс или не иметь. Если индекс есть и опубликован, номенклатура отображается на вкладке своей категории в публичной части индексов.

При деактивации номенклатуры (`nomenclature.isActive = false`) индекс автоматически становится невидимым в публичной части (как если бы был в `draft`), но не меняет своё `status`. Сами значения сохраняются. При повторной активации номенклатуры индекс снова отображается.

### 3.5. Единица измерения
Цена индекса хранится и отображается в единице, заданной полем `unit` номенклатуры (`07-platform/admin-panel.md`, раздел 13.2). На рынке вторсырья типичная единица — рубль за тонну (₽/т). Платформа поддерживает любые единицы, заданные на номенклатуре, и отображает их рядом с числовым значением цены.

## 4. PriceIndexValue — значение индекса

### 4.1. Назначение
Отдельная запись в истории значений индекса: пара «дата + цена».

### 4.2. Состав полей
- `id` — идентификатор;
- `priceIndexId` — ссылка на индекс цены;
- `date` — календарная дата, к которой относится значение (без времени; локальная дата платформы);
- `price` — цена в единице измерения номенклатуры (число с двумя знаками после запятой);
- `createdAt` — момент внесения записи в систему;
- `lastUpdatedAt` — момент последнего изменения значения;
- `createdBy` — идентификатор контент-менеджера, внёсшего запись;
- `lastUpdatedBy` — идентификатор актора, последним изменившего запись.

### 4.3. Уникальность
Сочетание (`priceIndexId`, `date`) уникально. По одной номенклатуре на одну дату — не более одного значения.

При попытке добавить значение на дату, для которой уже существует значение, контент-менеджеру предлагается:
- заменить существующее значение (запись обновляется, `lastUpdatedAt` устанавливается в текущее время);
- отменить ввод.

### 4.4. Гранулярность
Гранулярность значений — один день. Несколько значений на одну дату (утренняя, вечерняя цена) в первой версии не поддерживаются. Метки времени в часах/минутах не используются.

### 4.5. Будущие даты
Контент-менеджер может ввести значение с датой в будущем (например, «прогнозная цена»). Платформа не отклоняет такие значения, но при расчёте текущей цены и недельного изменения (раздел 5) используются только значения с датой ≤ сегодняшнего дня.

Опубликованные значения с будущей датой невидимы в публичной части до наступления указанной даты.

### 4.6. Удаление значений
Контент-менеджер может удалить любое значение в индексе в состоянии `draft`. В индексе в состоянии `published` удалять значения может также контент-менеджер — это редактирование уже опубликованного индекса, не структурное изменение модуля; действие фиксируется в служебном журнале (раздел 8).

При удалении значения, на которое опирается текущая цена или недельное изменение (раздел 5), пересчёт выполняется автоматически на следующих обращениях к индексу.

## 5. Расчёт текущей цены, недельного изменения и плашки тренда

### 5.1. Текущая цена
Текущая цена индекса — `price` из последнего по дате значения `PriceIndexValue` с условием `date ≤ сегодняшний день`.

Алгоритм:
1. Из всех значений индекса выбираются записи с `date ≤ today`;
2. Из выбранных берётся запись с максимальной `date`;
3. `currentPrice` — `price` этой записи; `currentDate` — `date` этой записи.

Если не нашлось ни одной записи с `date ≤ today` — индекс в публичной части не показывается ([overview.md](./overview.md), раздел 6).

### 5.2. Недельное изменение
Алгоритм расчёта недельного изменения:
1. Берётся `currentPrice` и `currentDate` из раздела 5.1;
2. Ищется запись с `date = currentDate - 7 дней`;
3. Если такой записи нет — берётся ближайшая более ранняя запись в окне `currentDate - 14 дней` ≤ `date` < `currentDate`;
4. Если в окне 14 дней назад не найдено ни одной записи — недельное изменение не рассчитывается, плашка тренда не отображается на карточке;
5. В остальных случаях вычисляется: `weeklyChange = (currentPrice - previousPrice) / previousPrice × 100`, округление до одного знака после запятой;
6. Отображается с указанием знака: «+1.8%», «−2.1%», «0.0%».

### 5.3. Плашка тренда
Плашка тренда отображается на карточке индекса при наличии рассчитанного недельного изменения. Состояние плашки определяется по порогам (см. [overview.md](./overview.md), раздел 8.3):

- `weeklyChange > +stagnationThreshold` → **«Рост»** (зелёная плашка);
- `−stagnationThreshold ≤ weeklyChange ≤ +stagnationThreshold` → **«Стагнация»** (серая плашка);
- `weeklyChange < −stagnationThreshold` → **«Падение»** (красная плашка).

`stagnationThreshold` — параметр конфигурации платформы. Значение по умолчанию — `1.0` (то есть ±1%). Изменение порога выполняется в служебных настройках администратором (`07-platform/admin-panel.md`, раздел 18).

### 5.4. Поведение при редких данных
- Если значений у индекса нет вообще — индекс невидим в публичной части (раздел 5.1);
- Если значение только одно — отображается текущая цена, недельное изменение и плашка тренда не отображаются;
- Если значений несколько, но в окне 14 дней назад от текущего нет — отображается только текущая цена.

### 5.5. Обновление при добавлении новых значений
При внесении нового значения:
- если новое значение становится последним по дате (≤ today) — пересчитываются `currentPrice`, `currentDate`, `weeklyChange`, плашка тренда на следующих обращениях к индексу;
- если новое значение — историческое (не последнее) — пересчёт затрагивает только график (раздел 7), но не плашки.

## 6. Правила валидации перед публикацией

### 6.1. Валидация индекса перед `draft → published`
Индекс может быть переведён в `published`, если выполнены условия:
- `nomenclatureId` указан и ссылается на существующую номенклатуру;
- номенклатура с `isActive = true`;
- индекс содержит хотя бы одно значение с `date ≤ today`;
- все значения индекса имеют корректные `date` (валидная календарная дата) и `price` (положительное число).

Если хотя бы одно условие не выполнено, перевод в `published` блокируется с пояснительным сообщением. Контент-менеджер исправляет указанные ошибки и повторяет попытку.

### 6.2. Повторная публикация
При снятии индекса с публикации (`published → draft`, только администратор) и повторной публикации проверки раздела 6.1 выполняются заново.

## 7. Поведение графика по периодам

### 7.1. Период отображения
Карточка индекса в публичной части содержит селектор периода с четырьмя кнопками:
- `1М` — последние 30 календарных дней;
- `3М` — последние 90 календарных дней;
- `6М` — последние 180 календарных дней;
- `1Г` — последние 365 календарных дней.

При клике на кнопку график пересчитывается под выбранный период; никакие другие элементы карточки не меняются.

### 7.2. Поведение при коротких данных
Если у индекса данных меньше выбранного периода, график показывает все имеющиеся точки в пределах этого периода без блокировки кнопки.

Пример (по согласованному ранее сценарию пользователя):
- Если у индекса данные только за последние 4 месяца, и пользователь выбрал `1Г` — график показывает 4 месяца данных;
- Если выбран `6М` — также 4 месяца;
- Если выбран `3М` — показывает 3 месяца (точки в окне последних 90 дней);
- Если выбран `1М` — показывает 1 месяц.

Все кнопки селектора всегда доступны, независимо от объёма данных.

### 7.3. Граница окна
Период считается от текущей даты назад. Дата начала окна = `today - {период в днях}`. Точки с `date < {дата начала}` не отображаются.

### 7.4. Соединение точек
График — ломаная линия, соединяющая существующие точки в порядке возрастания даты. Между датами без значений интерполяция не выполняется (см. [overview.md](./overview.md), раздел 4) — линия идёт прямо от точки к точке.

Заполнение области под графиком (градиент) — элемент дизайн-системы, не влияющий на семантику данных.

## 8. Удаление и каскады

### 8.1. Удаление индекса
- **Контент-менеджер** удаляет индекс только в состоянии `draft`, не имеющий ранее опубликованных редакций (`firstPublishedAt` не заполнен). Удаление подтверждается простым диалогом.
- **Администратор** удаляет индекс в любом состоянии. При удалении опубликованного индекса открывается диалог с обязательным полем «Причина удаления». Действие фиксируется в `AdminActionLog`.

При удалении индекса каскадно удаляются:
- все значения `PriceIndexValue` этого индекса.

Связь с номенклатурой при этом не удаляется — сама номенклатура остаётся в справочнике без индекса.

### 8.2. Удаление номенклатуры
При удалении номенклатуры администратором (`07-platform/admin-panel.md`, раздел 13.6) каскадно удаляется её индекс (если был) и все его значения.

### 8.3. Удаление категории
Удаление категории номенклатуры требует, чтобы все её номенклатуры были предварительно перемещены в другую категорию или деактивированы. Если в категории остаются активные номенклатуры с опубликованными индексами, удаление категории блокируется с пояснением.

### 8.4. Журналирование
Все удаления индексов, изменения порогов тренда и переходы состояний публикации фиксируются в `AdminActionLog` (`07-platform/admin-panel.md`, раздел 16) с указанием актора, времени, идентификатора сущности и причины.

Изменения отдельных значений `PriceIndexValue` не фиксируются в `AdminActionLog` (это массовая контентная операция); фиксируются на уровне общего журнала контентных правок, общего с другими модулями (открытый вопрос для последующих версий — см. [overview.md](./overview.md), раздел 9).
