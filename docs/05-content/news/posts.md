## 1. Назначение раздела
Настоящий документ фиксирует состав сущностей новости, их атрибуты, связи и поведение. Документ продолжает [overview.md](./overview.md) и развивает разделы 5 и 6 шапки.

Документ определяет:
- состав полей сущностей `NewsPost`, `NewsContentBlock`, `NewsTag`, `NewsLike`;
- типы блоков контента новости и состав их полей;
- правила работы с тегами;
- правила лайков на новостях;
- правила валидации перед публикацией;
- поведение при ошибках встраивания видео и недоступности аудиофайлов;
- каскадные эффекты при удалении.

Состав поля для модели данных подлежит окончательной фиксации в `08-architecture/data-model.md`. Настоящий документ задаёт продуктовый набор полей.

## 2. NewsPost — новость

### 2.1. Назначение
Корневая единица модуля «Новости». Представляет собой публикацию в ленте: заголовок, лид, обложка, последовательность блоков контента, теги.

### 2.2. Состав полей
- `id` — идентификатор;
- `title` — заголовок новости (markdown с инлайн-форматированием: жирный, курсив, выделение; без заголовков, ссылок, изображений; обязательное; лимиты — [limits.md](./limits.md), раздел 2);
- `lead` — лид-текст для карточки в ленте (строка, обязательное);
- `coverImage` — обложка для карточки в ленте (ссылка на файл изображения в файловом хранилище; опциональное; если не задано — карточка отображается без изображения);
- `slug` — фрагмент URL для прямой ссылки (строка, уникальная в пределах платформы); генерируется автоматически из `title` при первой публикации, может быть переопределена контент-менеджером;
- `status` — состояние публикации: `draft`, `published`;
- `firstPublishedAt` — момент первой публикации (заполняется автоматически);
- `lastUpdatedAt` — момент последней редакции (заполняется автоматически);
- `createdBy` — идентификатор контент-менеджера, создавшего новость;
- `contentBlocks` — упорядоченный список блоков контента (раздел 3);
- `tags` — список связей с тегами (раздел 4); может быть пустым;
- `likeCount` — счётчик лайков, агрегируемый по `NewsLike` (раздел 5).

### 2.3. Состояния публикации
- `draft` — новость видна только в админ-панели; в ленте пользователей не отображается;
- `published` — новость видна в ленте и доступна по прямой ссылке всем пользователям с активной подпиской.

Переходы:
- `draft → published` — выполняется контент-менеджером или администратором;
- `published → draft` — выполняется только администратором.

Полные правила переходов и валидации перед публикацией — раздел 8 настоящего документа и [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 7.

### 2.4. Слаг и URL
Слаг — короткий идентификатор новости для использования в URL прямой ссылки `/news/{slug}`.

Правила формирования слага:
- автогенерация при первой публикации новости: транслитерация `title` (без markdown-разметки) с заменой пробелов и пунктуации на дефис, приведением к нижнему регистру, ограничением длины до 80 символов;
- контент-менеджер может переопределить слаг вручную в редакторе перед первой публикацией;
- после первой публикации слаг фиксируется и не меняется автоматически. Изменение слага после публикации возможно только администратором; при изменении старый слаг сохраняется как редирект на новый (на уровне маршрутизации платформы);
- слаг должен быть уникален в пределах платформы; конфликт разрешается добавлением числового суффикса.

Поведение редиректов и срок их хранения — открытый вопрос для последующих версий.

## 3. NewsContentBlock — блок контента новости

### 3.1. Назначение
Упорядоченный элемент тела новости. Новость состоит из последовательности блоков; ни один тип блока не является обязательным; порядок задаётся контент-менеджером.

### 3.2. Состав полей
- `id` — идентификатор;
- `newsPostId` — ссылка на новость;
- `position` — порядковый номер блока внутри новости (целое число; уникально в пределах новости);
- `type` — тип блока (раздел 3.3);
- `payload` — структура полей, специфичная для типа блока (раздел 3.3).

### 3.3. Типы блоков
В первой версии поддерживаются семь типов блоков:

#### 3.3.1. `heading` — заголовок
Заголовок секции внутри новости. Для крупных разделительных заголовков.

Поля `payload`:
- `text` — текст заголовка (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 4).

#### 3.3.2. `subheading` — подзаголовок
Подзаголовок внутри секции.

Поля `payload`:
- `text` — текст подзаголовка (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 4).

#### 3.3.3. `paragraph` — абзац
Текстовый абзац в формате markdown. Поддерживает инлайн-форматирование (жирный, курсив, выделение), инлайн-ссылки, маркированные и нумерованные списки.

Поля `payload`:
- `markdown` — текст абзаца (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 4).

#### 3.3.4. `image` — изображение
Одиночное изображение с опциональной подписью.

Поля `payload`:
- `fileId` — ссылка на файл изображения в файловом хранилище (обязательное; уровень доступа `public`; см. `07-platform/storage.md`, раздел 5.11);
- `caption` — подпись под изображением (строка, опциональная);
- `altText` — альтернативный текст для доступности (строка, опциональная).

#### 3.3.5. `gallery` — галерея изображений
Упорядоченный набор изображений. Отображается в публичном интерфейсе как сетка миниатюр с возможностью просмотра в полном размере (lightbox) и переключения между изображениями.

Поля `payload`:
- `images` — упорядоченный список изображений (как в `image`: `fileId`, `caption`, `altText`); список не может быть пустым.

#### 3.3.6. `video` — видео
Встроенный видеоролик с внешнего хостинга Rutube.

Поля `payload`:
- `rutubeUrl` — URL ролика Rutube (строка, обязательное; должна быть валидной ссылкой на ролик Rutube — раздел 6.1);
- `caption` — подпись под видео (строка, опциональная).

#### 3.3.7. `audio` — аудио
Встроенный аудиофайл (подкаст), хранящийся на стороне платформы.

Поля `payload`:
- `fileId` — ссылка на аудиофайл в файловом хранилище (обязательное; уровень доступа `authenticated` либо `public` — определяется правилами модуля и фиксируется в сопутствующих правках `07-platform/storage.md`);
- `episodeTitle` — название выпуска подкаста, отображаемое над плеером (строка, опциональная; например, «Post Reports / Подкаст»);
- `caption` — подпись под плеером (строка, опциональная);
- `durationSeconds` — длительность аудио в секундах (целое число; заполняется автоматически из метаданных файла при загрузке; если извлечь не удалось — заполняется контент-менеджером вручную).

### 3.4. Поведение при удалении файла блока
Если файл изображения, галереи или аудио (поле `fileId`) удалён из файлового хранилища (например, вручную администратором через сценарий поддержки), при отображении новости на месте блока выводится плашка «Файл недоступен» с указанием идентификатора блока. Контент-менеджер видит ту же плашку в редакторе и заменяет файл новым.

Для блока `audio` дополнительно отображается сообщение «Аудио временно недоступно» в плеере.

## 4. NewsTag — тег новости

### 4.1. Назначение
Справочный элемент для группировки новостей по тематике. Один тег применяется к произвольному числу новостей; одна новость может иметь от нуля до нескольких тегов.

### 4.2. Состав полей
- `id` — идентификатор;
- `name` — отображаемое название тега (строка, обязательное; уникальное в пределах платформы по нормализованной форме — без учёта регистра и крайних пробелов);
- `slug` — фрагмент URL для фильтра ленты (строка, уникальная); генерируется автоматически из `name` транслитерацией;
- `createdAt` — момент создания тега;
- `createdBy` — идентификатор актора, создавшего тег (контент-менеджер или администратор);
- `usageCount` — производный показатель: число опубликованных новостей с этим тегом. Используется в служебной статистике и в выпадающем списке автодополнения для приоритизации часто используемых тегов.

### 4.3. Связь с новостями
Связь между `NewsPost` и `NewsTag` — many-to-many. Реализуется через промежуточную сущность `NewsPostTag` с полями `newsPostId`, `newsTagId`, `addedAt`. На уровне продукта эта сущность не оперируется — пользователь и контент-менеджер видят теги как простой список.

### 4.4. Создание и удаление
- Тег создаётся контент-менеджером или администратором через редактор новости (см. [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 5);
- Тег без активных связей с опубликованными новостями автоматически не удаляется; администратор удаляет его вручную через экран управления тегами;
- При удалении тега все связи `NewsPostTag` для этого тега удаляются; новости остаются нетронутыми (без удалённого тега в списке тегов);
- При удалении новости связи `NewsPostTag` для этой новости удаляются; теги остаются.

### 4.5. Переименование и склейка
Переименование тега выполняется только администратором (см. [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 8.2):
- меняется поле `name`;
- поле `slug` пересчитывается; старый слаг сохраняется как редирект на новый.

Склейка двух тегов (объединение `tagA → tagB`) выполняется только администратором:
- все связи `NewsPostTag` с `tagA` переносятся на `tagB` (с защитой от дубликатов);
- `tagA` удаляется;
- старый слаг `tagA` сохраняется как редирект на слаг `tagB`.

Действия фиксируются в `AdminActionLog`.

## 5. NewsLike — лайк на новости

### 5.1. Назначение
Запись о факте того, что конкретный пользовательский аккаунт компании поставил лайк конкретной новости.

### 5.2. Состав полей
- `id` — идентификатор;
- `userId` — идентификатор пользовательского аккаунта компании;
- `newsPostId` — идентификатор новости;
- `createdAt` — момент проставления лайка.

### 5.3. Уникальность и поведение
Сочетание (`userId`, `newsPostId`) уникально. Повторное проставление лайка тем же пользователем по той же новости не создаёт новой записи; обратное действие (нажатие лайка повторно при уже стоящем) удаляет запись (toggle).

Счётчик `NewsPost.likeCount` — производный показатель, агрегируется как `count(NewsLike where newsPostId = ...)`. На стороне реализации может кешироваться для производительности; при наличии кеша обновление выполняется при каждой операции лайка.

### 5.4. Видимость
Имена пользователей, поставивших лайк, в первой версии не отображаются. На странице новости и на карточке в ленте показывается только число лайков. Список лайкнувших — открытый вопрос для последующих версий.

### 5.5. Снятие лайка
Пользователь может снять лайк в любой момент нажатием на ту же кнопку. При снятии запись `NewsLike` удаляется, счётчик уменьшается.

### 5.6. Поведение в состояниях компании
- `active`, `demo`, `past_due` — пользователь может ставить и снимать лайки;
- `suspended`, `archived`, `blocked` — лента и страницы новостей недоступны, действия с лайками невозможны. Существующие лайки сохраняются.

## 6. Видео и аудио

### 6.1. Видео Rutube
Поведение видео в новостях полностью идентично поведению видео в обучении (`05-content/education/lessons.md`, раздел 8):
- Поле `rutubeUrl` принимает только ссылки на отдельные ролики Rutube;
- Платформа выполняет минимальную валидацию формата; не выполняет проверку доступности при сохранении;
- При недоступности ролика на месте блока выводится плашка «Видео временно недоступно».

### 6.2. Аудио

#### 6.2.1. Хранение
Аудиофайлы загружаются контент-менеджером во встроенное файловое хранилище платформы. Категория файлов фиксируется как сопутствующая правка к `07-platform/storage.md` (раздел 5.x — «аудио контентных модулей»; см. [other-integrations.md](./other-integrations.md), раздел 3).

#### 6.2.2. Допустимые форматы
- MP3 (`.mp3`);
- M4A (`.m4a`).

Иные аудиоформаты (WAV, FLAC, OGG, AAC) в первой версии не поддерживаются.

#### 6.2.3. Длительность
При загрузке файла платформа извлекает длительность из метаданных файла и сохраняет в `durationSeconds` блока. Если метаданные не содержат длительности или извлечь её не удалось, контент-менеджер вводит длительность вручную в формате «MM:SS» или «HH:MM:SS».

Длительность отображается на плеере как общее время трека и используется в карточке новости в ленте, если аудиоблок единственный или ключевой в новости.

#### 6.2.4. Плеер
На странице новости аудиоблок отображается как стандартный плеер платформы:
- кнопка play/pause;
- прогресс-бар с возможностью перемотки;
- индикаторы текущего времени и общей длительности;
- регулятор громкости.

Конкретные параметры плеера и его UX — на стороне дизайн-системы.

#### 6.2.5. Поведение при недоступности файла
Если аудиофайл удалён из хранилища или временно недоступен, на месте плеера отображается сообщение «Аудио временно недоступно». Остальные блоки новости остаются доступны.

## 7. Обложка новости

### 7.1. Назначение
Обложка `NewsPost.coverImage` — изображение, отображаемое на карточке новости в ленте. Не путать с блоком `image` в теле новости — обложка не входит в `contentBlocks` и не отображается на странице самой новости (страница новости показывает только блоки).

### 7.2. Источник
Обложка может быть:
- загружена контент-менеджером отдельно (через стандартный компонент загрузки в файловое хранилище);
- не задана — в этом случае карточка в ленте отображается без изображения, в одну колонку текста.

### 7.3. Лимиты
Лимиты на размер и формат — `07-platform/storage.md`, раздел 5.11. Дополнительный лимит на отображение в карточке (соотношение сторон, рекомендуемое разрешение) — на стороне дизайн-системы.

## 8. Правила валидации перед публикацией

### 8.1. Валидация новости перед `draft → published`
Новость может быть переведена в `published`, если выполнены условия:
- `title` заполнен;
- `lead` заполнен;
- новость содержит хотя бы один блок контента (любого типа);
- все блоки прошли валидацию своих полей (раздел 3.3);
- все ссылки на файлы в блоках указывают на существующие файлы в файловом хранилище;
- слаг (`slug`) уникален в пределах платформы.

### 8.2. Повторная публикация
При снятии новости с публикации (`published → draft`, только администратор) и повторной публикации проверки раздела 8.1 выполняются заново. Слаг сохраняется (если не был изменён администратором).

## 9. Удаление и каскады

### 9.1. Удаление новости
- **Контент-менеджер** удаляет новость только в состоянии `draft`, не имеющую ранее опубликованных редакций (`firstPublishedAt` не заполнен). Удаление подтверждается простым диалогом «Удалить безвозвратно?».
- **Администратор** удаляет новость в любом состоянии. При удалении опубликованной новости открывается диалог с обязательным полем «Причина удаления».

При удалении новости каскадно удаляются:
- все блоки контента этой новости;
- все связи `NewsPostTag` (теги остаются);
- все лайки `NewsLike` по этой новости;
- все комментарии и ответы на эту новость, их лайки и вложения (см. [comments.md](./comments.md), раздел 8).

Файлы изображений, галерей и аудио, привязанные через `fileId` блоков, удаляются по правилам файлового хранилища (`07-platform/storage.md`, раздел 7.7).

### 9.2. Журналирование
Удаления новостей фиксируются в `AdminActionLog` (`07-platform/admin-panel.md`, раздел 16) с указанием актора, времени, идентификатора новости и причины. Причина обязательна при удалении опубликованных новостей администратором; для удаления черновиков контент-менеджером причина опциональна.
