## 1. Назначение раздела
Настоящий документ фиксирует устройство раздела «Новости» административной панели — рабочего пространства контент-менеджера, в котором создаются, редактируются, публикуются и снимаются с публикации новости, а также управляется справочник тегов.

Документ продолжает [overview.md](./overview.md), раздел 10, и развивает workflow контент-менеджера до уровня экранов и операций.

Документ определяет:
- размещение раздела в админ-панели;
- список новостей и его фильтры;
- редактор новости и его подразделы;
- редакторы каждого типа блока контента;
- управление тегами;
- управление порядком блоков;
- workflow перехода между состояниями;
- удаление черновиков и удаление опубликованного;
- журналирование действий;
- открытые вопросы.

## 2. Размещение в админ-панели
Раздел «Новости» — пункт бокового меню административной панели (`07-platform/admin-panel.md`, раздел 6). Доступен платформенным сотрудникам с ролью `content_manager` или `admin`.

При нажатии на пункт меню открывается экран списка новостей (раздел 3). Внутри раздела доступны переходы к редактору новости (раздел 4) и к экрану управления тегами (раздел 8).

## 3. Список новостей

### 3.1. Состав экрана
Экран отображает все новости платформы независимо от их состояния (`draft` и `published`). Каждая строка списка содержит:
- заголовок новости (отрисовка с инлайн-форматированием);
- состояние публикации;
- список тегов;
- момент последней редакции;
- автор последней редакции (контент-менеджер или администратор);
- счётчики просмотров — открытый вопрос для последующих версий, в первой версии не отображаются;
- ссылка «Редактировать» — переход к редактору новости (раздел 4).

### 3.2. Фильтры
- по состоянию публикации (`draft`, `published`, оба);
- по тегу (один или несколько; пересечение);
- свободный поиск по заголовку и лиду.

### 3.3. Сортировка
По умолчанию — по моменту последней редакции, новые сверху. Альтернативные сортировки — по моменту первой публикации (для опубликованных), по заголовку.

### 3.4. Действие «Создать новость»
Кнопка «Создать новость» открывает редактор новой новости (раздел 4). При первом сохранении новость создаётся в состоянии `draft`.

### 3.5. Действие «Управление тегами»
Кнопка «Управление тегами» в верхней части экрана открывает экран управления тегами (раздел 8). Кнопка отображается только администратору; контент-менеджер этой кнопки не видит.

## 4. Редактор новости

### 4.1. Состав полей редактора
Редактор позволяет задать следующие атрибуты новости (см. [posts.md](./posts.md), раздел 2.2):
- заголовок (`title`) — поле с поддержкой инлайн-форматирования (раздел 4.2);
- лид (`lead`) — простое текстовое поле;
- обложка (`coverImage`) — кнопка загрузки изображения через файловое хранилище;
- слаг (`slug`) — поле редактируется автоматически на основе заголовка; контент-менеджер может переопределить вручную перед первой публикацией;
- теги (`tags`) — поле с автодополнением и возможностью создать новый тег (раздел 5);
- блоки контента (`contentBlocks`) — компоновщик блоков (раздел 6).

Поле `status` редактируется отдельной кнопкой публикации (раздел 7), не как обычное поле формы.

### 4.2. Редактор заголовка с инлайн-форматированием
Заголовок поддерживает три типа инлайн-форматирования:
- **жирный** (markdown: `**текст**`);
- *курсив* (markdown: `*текст*`);
- ==выделение== (markdown: `==текст==`, отрисовка с цветным фоном).

Для удобства контент-менеджера над полем заголовка отображается панель с тремя кнопками («Жирный», «Курсив», «Выделение»). Нажатие кнопки оборачивает выделенный текст соответствующей разметкой. Доступен и прямой ввод markdown в поле — оба пути эквивалентны.

Заголовки внутри заголовка, инлайн-ссылки, изображения, списки в заголовке не поддерживаются.

### 4.3. Подразделы внутри редактора новости
В редакторе новости доступны вкладки:
- **Параметры** — поля раздела 4.1 (заголовок, лид, обложка, слаг, теги);
- **Содержание** — компоновщик блоков (раздел 6);
- **Публикация** — кнопки управления состоянием и список валидационных ошибок (раздел 7).

### 4.4. Сохранение
Все изменения в редакторе сохраняются по нажатию кнопки «Сохранить» в нижней части экрана. Автосохранение в первой версии не реализуется.

### 4.5. Предпросмотр
В редакторе новости доступна кнопка «Просмотреть как пользователь» — открывает страницу новости в виде, идентичном публичному интерфейсу (см. [public-ui.md](./public-ui.md), раздел 5). Используется для проверки оформления до публикации.

Предпросмотр доступен и для черновиков. Открывается в новом окне с пометкой «Режим предпросмотра» в служебной плашке.

## 5. Управление тегами при редактировании новости

### 5.1. Поле ввода тегов
В подразделе «Параметры» редактора новости расположено поле ввода тегов с автодополнением:
- контент-менеджер начинает вводить название тега;
- платформа отображает выпадающий список совпадений из существующих тегов, отсортированный по `usageCount` (часто используемые сверху);
- контент-менеджер выбирает один из вариантов либо нажимает «Создать новый тег: ...» в нижней части списка, чтобы создать новый.

Добавленные теги отображаются как «чипы» под полем ввода с крестиком для удаления.

### 5.2. Создание нового тега
При создании нового тега в момент написания новости платформа:
1. Нормализует название (обрезка пробелов, единый регистр для проверки уникальности);
2. Генерирует `slug` транслитерацией;
3. Создаёт запись `NewsTag` с `createdBy` равным актору;
4. Добавляет тег в список тегов текущей новости.

Тег существует в справочнике сразу после создания, даже если новость ещё не опубликована.

### 5.3. Удаление тега из новости
Удаление чипа тега в редакторе разрывает связь между текущей новостью и тегом. Сам тег остаётся в справочнике.

## 6. Компоновщик блоков

### 6.1. Принцип
Компоновщик блоков отображает блоки новости сверху вниз в порядке, заданном `position`. Между блоками и над первым блоком отображается кнопка-разделитель «+ Добавить блок» — открывает меню типов блоков (раздел 6.2).

Каждый отображаемый блок имеет:
- иконку типа блока;
- редактор полей блока (зависит от типа, раздел 6.2);
- кнопку «Удалить блок»;
- ручку перетаскивания (drag-and-drop) для изменения порядка.

### 6.2. Редакторы блоков по типам

#### 6.2.1. `heading` — заголовок
Поле ввода строки `text`. Лимиты — [limits.md](./limits.md), раздел 4.

#### 6.2.2. `subheading` — подзаголовок
Поле ввода строки `text`. Лимиты — [limits.md](./limits.md), раздел 4.

#### 6.2.3. `paragraph` — абзац
Многострочное поле ввода markdown. Поддерживается панель быстрого форматирования: жирный, курсив, маркированный список, нумерованный список, инлайн-ссылка. Заголовки внутри абзаца не поддерживаются.

#### 6.2.4. `image` — изображение
Кнопка загрузки файла открывает компонент загрузки в файловое хранилище (`07-platform/storage.md`). После загрузки отображается миниатюра. Поля:
- `caption` — текстовая подпись (опционально);
- `altText` — альтернативный текст (опционально, рекомендуется заполнять).

Кнопка «Заменить изображение» загружает новый файл и удаляет старую связь.

#### 6.2.5. `gallery` — галерея
Список изображений с кнопкой «Добавить изображение». Каждый элемент списка устроен как блок `image` с возможностью изменения порядка перетаскиванием. Галерея не может быть пустой — при попытке сохранить блок без единого изображения отображается ошибка валидации.

#### 6.2.6. `video` — видео
Поле ввода `rutubeUrl`. Платформа выполняет минимальную валидацию формата ссылки (см. [posts.md](./posts.md), раздел 6.1) при сохранении. При некорректной ссылке отображается ошибка.

Поле `caption` — текстовая подпись (опционально).

#### 6.2.7. `audio` — аудио
Кнопка «Загрузить аудиофайл» открывает компонент загрузки в файловое хранилище. После загрузки:
- платформа извлекает длительность из метаданных файла и заполняет `durationSeconds` автоматически;
- если длительность извлечь не удалось, контент-менеджер вводит её вручную в формате «MM:SS» или «HH:MM:SS».

Поля:
- `episodeTitle` — название выпуска подкаста (опционально);
- `caption` — подпись под плеером (опционально);
- `durationSeconds` — длительность (заполнена автоматически или вручную).

Кнопка «Заменить аудиофайл» загружает новый файл и удаляет старую связь.

### 6.3. Управление порядком
Порядок блоков, изображений в галереях, тегов задаётся перетаскиванием. Каждый элемент имеет ручку перетаскивания. При перетаскивании пересчитываются поля `position` затронутых элементов. Изменения сохраняются автоматически при отпускании элемента — отдельная кнопка «Сохранить порядок» не требуется.

## 7. Workflow публикации

### 7.1. Состояния и переходы
Состояния новости: `draft`, `published` (см. [posts.md](./posts.md), раздел 2.3).

Переходы:
- `draft → published` — выполняется контент-менеджером или администратором;
- `published → draft` — выполняется только администратором.

### 7.2. Кнопка «Опубликовать»
В подразделе «Публикация» редактора новости доступна кнопка «Опубликовать». При нажатии:
1. Платформа выполняет валидацию (см. [posts.md](./posts.md), раздел 8.1);
2. При успешной валидации новость переводится в `published`, в поле `firstPublishedAt` подставляется текущее время (если поле пустое);
3. При ошибках валидации отображается список ошибок с переходами к местам, где их нужно исправить. Состояние не меняется.

### 7.3. Кнопка «Снять с публикации»
Кнопка отображается только администратору. При нажатии новость переводится в `draft`. Действие фиксируется в `AdminActionLog` с обязательным указанием причины.

При снятии с публикации новость исчезает из ленты пользователей и становится недоступной по прямой ссылке (страница возвращает HTTP 404 или редирект на ленту с уведомлением).

### 7.4. Двухстадийный workflow
Создание и редактирование выполняются непосредственно контент-менеджером без отдельной стадии согласования. Промежуточное состояние «на ревью» в первой версии не предусматривается.

## 8. Управление тегами (отдельный экран)

### 8.1. Кому доступен экран
Экран «Управление тегами» доступен только администратору платформы. Контент-менеджер не имеет к нему доступа и не может переименовывать, склеивать или удалять чужие теги.

### 8.2. Состав экрана
Экран отображает справочник тегов в виде таблицы:
- название тега (`name`);
- слаг (`slug`);
- число опубликованных новостей с этим тегом (`usageCount`);
- момент создания;
- автор создания (контент-менеджер или администратор);
- кнопки операций: «Переименовать», «Склеить», «Удалить».

Доступен поиск по названию тега и фильтр «только теги без активных связей».

### 8.3. Переименование тега
По кнопке «Переименовать» открывается диалог с полем нового названия. После подтверждения:
1. Платформа проверяет, что новое название не дублирует другой существующий тег (по нормализованной форме);
2. Меняет поле `name`;
3. Пересчитывает `slug`;
4. Сохраняет старый слаг как редирект на новый (на уровне маршрутизации платформы).

Действие фиксируется в `AdminActionLog`.

### 8.4. Склейка тегов
По кнопке «Склеить» открывается диалог с выбором целевого тега из выпадающего списка существующих тегов. После подтверждения:
1. Платформа переносит все связи `NewsPostTag` исходного тега на целевой (с защитой от дубликатов);
2. Удаляет исходный тег;
3. Сохраняет старый слаг исходного тега как редирект на слаг целевого.

Действие фиксируется в `AdminActionLog`.

### 8.5. Удаление тега
По кнопке «Удалить» открывается диалог подтверждения. Удалить можно тег с любым `usageCount` — связи разрываются, новости остаются нетронутыми (в их списке тегов удалённый тег больше не отображается).

Действие фиксируется в `AdminActionLog`.

## 9. Удаление новостей

### 9.1. Удаление черновиков
Контент-менеджер удаляет:
- новость в состоянии `draft`, не имеющую ранее опубликованных редакций (`firstPublishedAt` пуст).

Удаление подтверждается простым диалогом «Удалить безвозвратно?». Действие в `AdminActionLog` не записывается, либо записывается в служебном виде без обязательной причины.

### 9.2. Удаление опубликованного
Удаление опубликованных новостей выполняется только администратором. При удалении:
- открывается диалог с обязательным полем «Причина удаления»;
- удаление выполняется каскадно по правилам [posts.md](./posts.md), раздел 9.1;
- действие фиксируется в `AdminActionLog`.

## 10. Журналирование

### 10.1. Что фиксируется в `AdminActionLog`
Следующие действия в разделе «Новости» админ-панели фиксируются в `AdminActionLog` (`07-platform/admin-panel.md`, раздел 16):
- публикация новости (`draft → published`);
- снятие новости с публикации (`published → draft`);
- удаление опубликованной новости администратором;
- переименование, склейка, удаление тега администратором;
- изменение слага опубликованной новости администратором.

### 10.2. Что не фиксируется
- редактирование полей новости в любом состоянии;
- создание новых новостей как черновиков;
- изменение порядка элементов;
- удаление контент-менеджером собственных черновиков;
- загрузка и замена медиафайлов в блоках;
- создание новых тегов через редактор новости;
- проставление и снятие лайков пользователями (это пользовательская активность, не административное действие);
- создание комментариев пользователями.

Эти операции являются массовыми и журналируются на уровне общего журнала редактирования контентных сущностей, отдельного от `AdminActionLog`. Конкретный журнал контентных правок и его правила хранения — открытый вопрос для последующих версий.

## 11. Открытые вопросы первой версии

### 11.1. Конкурентное редактирование
Возможность одного контент-менеджера временно занять новость в редактировании (lock) и предотвратить одновременное редактирование двумя сотрудниками в первой версии не реализуется. При одновременном редактировании двух пользователей побеждает тот, чьё сохранение прошло позже.

### 11.2. Версионирование
История редакций новости с возможностью отката к предыдущей версии в первой версии не реализуется. Изменения, внесённые контент-менеджером, перезаписывают предыдущую редакцию.

### 11.3. Журнал контентных правок
Отдельный журнал редактирования новостей и блоков (кто, когда, что изменил) в первой версии не реализуется. Журналируются только переходы состояний публикации и удаление опубликованного контента (раздел 10).

### 11.4. Управление тегами контент-менеджером
В первой версии переименование и удаление тегов — функция администратора. Возможность контент-менеджера управлять тегами в ограниченном объёме (например, переименование только тех, что он сам создал) — открытый вопрос.

### 11.5. Импорт новостей и черновиков
Импорт новостей из внешних источников (RSS, Google Docs, файлы Word) в первой версии не реализуется. Контент-менеджер компонует новости вручную в редакторе.
