## 1. Назначение раздела
Настоящий документ фиксирует устройство комментариев под новостями: модель сущностей, правила публикации, тредовая структура, лайки на комментариях, вложения изображений, поведение в состояниях компании, модерация и каскадные эффекты при удалении.

Документ продолжает [overview.md](./overview.md) (раздел 8) и [posts.md](./posts.md) (раздел 9.1, удаление комментариев).

Документ определяет:
- состав сущностей `Comment`, `CommentLike`, `CommentAttachment`;
- сценарий публикации комментария и ответа;
- структуру треда (один уровень вложенности);
- лайки на комментариях;
- вложения изображений в комментариях;
- модерацию по общей модели платформы;
- каскадное удаление при удалении новости и при удалении комментария первого уровня.

## 2. Comment — комментарий

### 2.1. Назначение
Сообщение пользовательского аккаунта компании под новостью. Комментарий имеет один или ноль родительских комментариев (раздел 4) и состоит из текста и опционального набора вложенных изображений.

### 2.2. Состав полей
- `id` — идентификатор;
- `newsPostId` — ссылка на новость;
- `parentCommentId` — ссылка на родительский комментарий, если данный комментарий является ответом; `null` для комментария первого уровня (раздел 4);
- `userId` — идентификатор пользовательского аккаунта компании, оставившего комментарий;
- `text` — текст комментария (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 7);
- `createdAt` — момент создания комментария;
- `lastUpdatedAt` — момент последней служебной правки (например, скрытие модератором; раздел 7);
- `status` — состояние комментария (раздел 6);
- `attachments` — список вложений (раздел 5);
- `likeCount` — счётчик лайков, агрегируемый по `CommentLike` (раздел 4).

Поле `text` — простая строка без markdown-форматирования. Инлайн-форматирование (жирный, курсив, ссылки) в комментариях первой версии не поддерживается.

### 2.3. Состояния комментария
Комментарий может находиться в одном из состояний:

- `published` — обычное состояние сразу после публикации; виден всем, кто видит новость;
- `hidden_by_moderator` — модератор скрыл комментарий по жалобе или собственной инициативе. Комментарий не виден пользователям, кроме автора и платформенных сотрудников; вместо текста под новостью на месте комментария отображается плашка «Комментарий скрыт модератором» (с возможностью раскрытия для самого автора);
- `removed_by_admin` — администратор удалил комментарий полностью. Комментарий невидим никому, включая автора; запись в БД сохраняется для целей журнала;
- `removed_with_news` — комментарий удалён вместе с новостью (см. [posts.md](./posts.md), раздел 9.1). Состояние используется в служебных целях; для пользователя такого комментария не существует.

В первой версии пользователь не может редактировать или удалять собственный комментарий (см. `overview.md`, раздел 8.5). Изменение состояний — только модератором/администратором.

## 3. Сценарий публикации комментария

### 3.1. Кто может оставлять комментарии
Любой пользовательский аккаунт компании в состоянии `active`, `demo` или `past_due` с доступом к новостям может оставлять комментарии и ответы.

В состояниях `suspended`, `archived`, `blocked` комментирование недоступно.

В состояниях, в которых на компанию или конкретного пользователя действует модерационная санкция типа `module_restriction` для модуля `comments` (`07-platform/moderation.md`, раздел 8.5), новые комментарии не публикуются: при попытке отправки отображается сообщение об ограничении и сроке его окончания.

### 3.2. Форма публикации
На странице новости (см. [public-ui.md](./public-ui.md), раздел 5) пользователь видит форму ввода комментария:
- многострочное текстовое поле;
- кнопка «Прикрепить» — открывает выбор файлов изображений (до трёх; см. раздел 5);
- кнопка «Отправить».

При нажатии «Отправить» платформа:
1. Проверяет валидность данных: непустой текст, число вложений ≤ 3, общий размер вложений в пределах лимита (см. [limits.md](./limits.md), раздел 8);
2. Создаёт запись `Comment` со статусом `published`;
3. Создаёт записи `CommentAttachment` по каждому вложению;
4. Возвращает обновлённый список комментариев под новостью.

Антивирусная проверка вложений выполняется по правилам `07-platform/storage.md`, раздел 7.2; при детекте угрозы вложение не сохраняется и комментарий не публикуется.

### 3.3. Публикация ответа
Пользователь нажимает «Ответить» на конкретном комментарии или ответе. Открывается такая же форма (раздел 3.2). Поле `parentCommentId` нового комментария устанавливается:
- на `id` целевого комментария первого уровня — если пользователь ответил на комментарий первого уровня;
- на `id` родительского комментария первого уровня для исходного комментария — если пользователь ответил на ответ.

Это обеспечивает плоскую тредовую структуру с одним уровнем вложенности (раздел 4).

## 4. Тредовая структура — один уровень вложенности

### 4.1. Принцип
В первой версии поддерживается один уровень вложенности комментариев:
- комментарии первого уровня (`parentCommentId = null`);
- ответы — комментарии второго уровня (`parentCommentId = id` комментария первого уровня).

Ответ на ответ интерпретируется как ответ на тот же комментарий первого уровня и не создаёт третьего уровня вложенности. Все ответы на одно «дерево» отображаются в одной плоской ветке под комментарием первого уровня в хронологическом порядке.

### 4.2. Отображение в UI
- Комментарий первого уровня отображается с отступом от левого края страницы по умолчанию;
- Ответы под комментарием первого уровня отображаются с дополнительным отступом, единым для всех ответов в ветке;
- Между комментариями первого уровня — увеличенный вертикальный отступ; между ответами в одной ветке — стандартный.

### 4.3. Сортировка
Комментарии первого уровня сортируются по умолчанию по дате публикации, новые сверху. Ответы внутри ветки сортируются по дате публикации, новые снизу (хронологический порядок дискуссии).

Альтернативные сортировки (по числу лайков, по релевантности) в первой версии не поддерживаются.

### 4.4. Адресация ответа
В тексте ответа допускается обращение по имени к получателю в свободной форме («@Иван, согласен» или «Иван, согласен»). Автокомплит имён участников и парсинг конструкций `@username` в первой версии не реализуются (раздел 9.2).

При публикации ответа в системе только устанавливается связь через `parentCommentId`; платформа не разбирает текст для определения адресата.

## 5. CommentAttachment — вложения изображений

### 5.1. Назначение
Связь комментария с одним из вложенных изображений. Один комментарий — от 0 до 3 вложений. Один файл — одно вложение (один комментарий не может ссылаться на один и тот же файл дважды).

### 5.2. Состав полей
- `id` — идентификатор;
- `commentId` — ссылка на комментарий;
- `fileId` — ссылка на файл изображения в файловом хранилище (обязательное; уровень доступа `public`; см. `07-platform/storage.md`, раздел 5.11);
- `position` — порядковый номер вложения в комментарии (целое число; уникально в пределах комментария).

### 5.3. Допустимые форматы и лимиты
- Допустимые форматы: JPG, PNG, WEBP;
- Размер файла — по правилам `07-platform/storage.md`, раздел 6.1 (изображения, до 10 МБ на файл);
- Число вложений на комментарий — до 3 ([limits.md](./limits.md), раздел 8).

Видео, аудио, документы, архивы и иные форматы файлов в комментариях в первой версии не поддерживаются.

### 5.4. Отображение в UI
- На карточке комментария вложения отображаются в виде сетки миниатюр под текстом;
- Клик по миниатюре открывает изображение в полноэкранном просмотре (lightbox) с возможностью переключения между вложениями этого комментария.

### 5.5. Удаление вложений
Пользователь не может удалять отдельные вложения у уже опубликованного комментария — собственный комментарий целиком неизменяем после публикации (см. `overview.md`, раздел 8.5).

Каскадное удаление вложений выполняется при удалении комментария (раздел 8) или при модерационном удалении (раздел 7).

## 6. CommentLike — лайк на комментарии

### 6.1. Назначение
Запись о факте того, что конкретный пользовательский аккаунт компании поставил лайк конкретному комментарию.

### 6.2. Состав полей
- `id` — идентификатор;
- `userId` — идентификатор пользовательского аккаунта компании;
- `commentId` — идентификатор комментария;
- `createdAt` — момент проставления лайка.

### 6.3. Уникальность и поведение
Сочетание (`userId`, `commentId`) уникально. Поведение лайка идентично лайку новости (см. [posts.md](./posts.md), раздел 5):
- toggle: повторное нажатие снимает лайк;
- автор комментария может лайкнуть собственный комментарий (стандартное поведение интерактивных лент);
- счётчик `Comment.likeCount` — производный показатель.

### 6.4. Видимость
Имена пользователей, поставивших лайк, в первой версии не отображаются. На карточке комментария показывается только число лайков. Список лайкнувших — открытый вопрос для последующих версий.

### 6.5. Лайки и состояния комментария
- Лайк нельзя поставить на скрытый или удалённый комментарий;
- При смене состояния комментария на `hidden_by_moderator` или `removed_by_admin` ранее проставленные лайки сохраняются в БД, но не отображаются. Счётчик не обнуляется (фиксированная история);
- При восстановлении комментария (раздел 7.4) лайки снова отображаются.

## 7. Модерация комментариев

### 7.1. Общая модель
Модерация комментариев — пост-модерация по общей модели платформы (`07-platform/moderation.md`). Комментарии публикуются сразу; жалобы и решения модератора рассматриваются после публикации.

### 7.2. Кто и на что подаёт жалобу
- Любой пользовательский аккаунт компании может подать жалобу на любой опубликованный комментарий или ответ через кнопку «Пожаловаться» на карточке комментария;
- Состав жалобы (код причины, свободный комментарий) — по правилам `07-platform/moderation.md`, раздел 5;
- Окно подачи — пока комментарий находится в состоянии `published`.

### 7.3. Действия модератора
Модератор по жалобе или собственной инициативе может:
- оставить комментарий без изменений;
- скрыть комментарий — комментарий переходит в `hidden_by_moderator`;
- эскалировать кейс администратору с предложением применить санкцию против пользователя.

Скрытие модератором — обратимое действие. Восстановление комментария (`hidden_by_moderator → published`) выполняется тем же модератором или администратором по решению `decision.reverted` (`07-platform/moderation.md`, раздел 8.4).

### 7.4. Действия администратора
Администратор имеет полный набор действий модератора. Дополнительно администратор:
- удаляет комментарий полностью — комментарий переходит в `removed_by_admin`. Это необратимое действие в рамках публичного интерфейса; восстановление возможно только через прямое вмешательство в БД и фиксируется как исключительный сценарий;
- применяет санкции против пользователя или компании по общей модели модерации.

### 7.5. Санкция «ограничение модуля комментариев»
Санкция типа `module_restriction` с модулем `comments` (`07-platform/moderation.md`, раздел 8.5) применяется к компании в целом. Пользовательский аккаунт компании, к которой применена санкция, не может оставлять новые комментарии и ответы; существующие комментарии остаются опубликованными до их отдельного скрытия модератором.

При попытке отправить комментарий в условиях действующей санкции система отображает сообщение «Возможность оставлять комментарии ограничена. Срок окончания: ...». Подробное поведение — `07-platform/moderation.md`, раздел 8.5.

### 7.6. Уведомления
Уведомления о решениях модератора и администратора по комментариям направляются по общей модели (`07-platform/notifications.md`, раздел 6.4) — категория «модерация»:
- `moderation.complaint.resolved` — автору жалобы;
- `moderation.content.removed` — автору скрытого/удалённого комментария;
- `moderation.warning.issued` — пользовательскому аккаунту компании при предупреждении;
- `moderation.module_restriction.applied` / `.lifted` — пользовательскому аккаунту компании при применении/снятии санкции.

## 8. Удаление и каскады

### 8.1. Удаление комментария первого уровня
При удалении комментария первого уровня администратором (раздел 7.4) каскадно удаляются:
- все ответы на этот комментарий первого уровня;
- лайки `CommentLike` по комментарию первого уровня и по всем его ответам;
- вложения `CommentAttachment` комментария и его ответов (вместе с файлами в хранилище — по правилам `07-platform/storage.md`, раздел 7.7).

Пользовательский интерфейс получает обновление: вся ветка исчезает с экрана.

### 8.2. Удаление ответа
При удалении ответа администратором (раздел 7.4) каскадно удаляются:
- лайки `CommentLike` по этому ответу;
- вложения `CommentAttachment` этого ответа.

Комментарий первого уровня и остальные ответы остаются.

### 8.3. Удаление вместе с новостью
При удалении новости (см. [posts.md](./posts.md), раздел 9.1) каскадно удаляются все комментарии и ответы под этой новостью со всеми связанными лайками и вложениями. Состояние комментариев в этом случае — `removed_with_news` (раздел 2.3); для пользователей таких комментариев больше не существует.

### 8.4. Скрытие модератором — не удаление
Скрытие комментария модератором (`hidden_by_moderator`) не удаляет связанные сущности: лайки и вложения сохраняются и снова отображаются при восстановлении комментария.

### 8.5. Журналирование
Все модерационные действия и удаления администратором фиксируются по общей модели (`07-platform/moderation.md`, раздел 11; `07-platform/admin-panel.md`, раздел 16). Создание комментариев пользователями и проставление лайков в `AdminActionLog` не фиксируется.

## 9. Открытые вопросы первой версии

### 9.1. Редактирование и удаление пользователем
Возможность пользователю редактировать или удалять собственный комментарий в первой версии не реализуется (см. `overview.md`, раздел 8.5). Решение об этой функции в последующих версиях принимается отдельно с учётом баланса между гибкостью пользователя и стабильностью треда обсуждения.

### 9.2. Упоминания и автокомплит
Конструкции `@username` и автокомплит участников треда в первой версии не поддерживаются. Адресация ответа выполняется через нажатие кнопки «Ответить» на конкретном комментарии (раздел 3.3).

### 9.3. Список лайкнувших
Видимость списка пользовательских аккаунтов компаний, поставивших лайк новости или комментарию, в первой версии не реализуется (раздел 6.4 и [posts.md](./posts.md), раздел 5.4).

### 9.4. Сортировки
Альтернативные сортировки комментариев (по числу лайков, по релевантности) в первой версии не поддерживаются (раздел 4.3).

### 9.5. Уведомления об ответах
Событие «вам ответили на комментарий» в каталоге уведомлений первой версии отсутствует (см. `overview.md`, раздел 12.3). Расширение каталога — открытый вопрос для последующих версий.

### 9.6. Использование модели в форуме
Модуль «Форум» (`09-forum/`) предполагает сообщения с тредовой структурой, лайками и вложениями, схожие с комментариями новостей. На уровне модели данных эти сущности могут быть унифицированы при детальной фиксации в `08-architecture/data-model.md`. Настоящий документ описывает только комментарии новостей и не претендует на универсальность.
