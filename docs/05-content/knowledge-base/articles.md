## 1. Назначение раздела
Настоящий документ фиксирует состав сущностей модуля «База знаний», их атрибуты, связи и поведение. Документ продолжает [overview.md](./overview.md) и развивает разделы 5, 6 и 7 шапки.

Документ определяет:
- состав полей сущности `KnowledgeBaseArticle`;
- иерархию дерева до трёх уровней;
- типы блоков контента и состав их полей;
- специализированные блоки `checklist` и `image_checklist`;
- блок `file` для прикреплённых документов;
- правила валидации перед публикацией;
- правила перемещения узлов в дереве;
- каскадные эффекты при удалении.

Состав поля для модели данных подлежит окончательной фиксации в `08-architecture/data-model.md`. Настоящий документ задаёт продуктовый набор полей.

## 2. KnowledgeBaseArticle — статья базы знаний

### 2.1. Назначение
Корневая единица модуля «База знаний». Каждая статья — узел дерева трёх уровней (категория, вид или подвид) со своим заголовком, подзаголовком, обложкой и последовательностью блоков контента. Не существует «пустых» узлов — у каждого узла есть собственная статья со своим контентом.

### 2.2. Состав полей
- `id` — идентификатор;
- `parentId` — ссылка на родительскую статью (`null` для корневых статей-категорий);
- `title` — заголовок статьи (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 2.1);
- `subtitle` — подзаголовок статьи (строка, опциональное; например, «Гофрированный картон, коробки, ящики»);
- `coverImage` — обложка статьи (ссылка на файл изображения в файловом хранилище; опциональное);
- `slug` — фрагмент URL для прямой ссылки (строка, уникальная в пределах базы знаний); генерируется автоматически из `title` при первой публикации, может быть переопределена контент-менеджером;
- `position` — порядковый номер статьи среди статей того же родителя (целое число; уникально в пределах одного `parentId`);
- `iconType` — необязательный иконографический тип для отображения в дереве (например, `paper`, `plastic`, `metal`, `glass`, `waste`, `hazard`); набор иконок — на стороне дизайн-системы. Поле опциональное; по умолчанию используется типовая иконка статьи;
- `status` — состояние публикации: `draft`, `published`;
- `firstPublishedAt` — момент первой публикации (заполняется автоматически);
- `lastUpdatedAt` — момент последней редакции (заполняется автоматически);
- `createdBy` — идентификатор контент-менеджера, создавшего статью;
- `contentBlocks` — упорядоченный список блоков контента (раздел 4).

### 2.3. Состояния публикации
- `draft` — статья видна только в админ-панели; в публичной части дерева не отображается;
- `published` — статья видна в публичной части пользователям с активной подпиской.

Переходы:
- `draft → published` — выполняется контент-менеджером или администратором;
- `published → draft` — выполняется только администратором.

Полные правила переходов и валидация перед публикацией — раздел 6 настоящего документа.

### 2.4. Слаг и URL
URL статьи: `/knowledge-base/{slug}`, независимо от иерархии дерева (см. [overview.md](./overview.md), раздел 9).

Правила формирования слага:
- автогенерация при первой публикации статьи: транслитерация `title` (без markdown-разметки) с заменой пробелов и пунктуации на дефис, приведением к нижнему регистру, ограничением длины до 80 символов;
- контент-менеджер может переопределить слаг вручную в редакторе перед первой публикацией;
- после первой публикации слаг фиксируется и не меняется автоматически. Изменение возможно только администратором; при изменении старый слаг сохраняется как редирект на новый;
- слаг должен быть уникален в пределах базы знаний (категории, виды и подвиды используют единое пространство слагов); конфликт разрешается добавлением числового суффикса.

## 3. Иерархия дерева

### 3.1. Уровни дерева
Дерево статей имеет до трёх уровней:

```
Корневая статья (категория, parentId = null)
└── Дочерняя статья (вид, parentId = категория)
    └── Внучатая статья (подвид, parentId = вид)
```

- **Категория** — статья без родителя (`parentId = null`);
- **Вид** — статья с родителем-категорией;
- **Подвид** — статья с родителем-видом.

Глубже трёх уровней дерево не растёт. Платформа отклоняет создание статьи с родителем, у которого уже есть свой родитель и родитель родителя (то есть три уровня предков по цепочке).

### 3.2. Опциональность средних и нижних уровней
- Категория может содержать ноль и более видов;
- Вид может содержать ноль и более подвидов;
- Категория может вообще не иметь видов — тогда у пользователя в навигации она отображается как одиночный узел без раскрытия;
- Вид может не иметь подвидов — отображается без раскрытия.

### 3.3. Перемещение узлов
Контент-менеджер может перемещать статьи между ветвями дерева (drag-and-drop в админ-панели; см. [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 4).

Правила перемещения:
- При перемещении статьи её потомки перемещаются вместе с ней (вся ветвь);
- Перемещение не должно приводить к превышению глубины дерева 3 уровнями. Если статья переносится со всеми потомками, и итоговая глубина превышает 3, перемещение блокируется с пояснительным сообщением;
- Перемещение не меняет идентификатор статьи (`id`) и её URL (`slug`);
- Перемещение перенумеровывает поле `position` для соседей в новом и в старом родителе.

Действие фиксируется в `AdminActionLog`, если затронута опубликованная статья (см. [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 9.1).

### 3.4. Порядок узлов в пределах родителя
Поле `position` определяет порядок отображения статей в дереве среди статей того же родителя. Порядок задаётся контент-менеджером через перетаскивание (drag-and-drop) в дереве.

При перетаскивании пересчитываются поля `position` затронутых статей для сохранения уникальности и непрерывности номеров.

## 4. KnowledgeBaseContentBlock — блок контента статьи

### 4.1. Назначение
Упорядоченный элемент тела статьи. Статья состоит из последовательности блоков; ни один тип блока не является обязательным; порядок задаётся контент-менеджером.

### 4.2. Состав полей
- `id` — идентификатор;
- `articleId` — ссылка на статью;
- `position` — порядковый номер блока внутри статьи (целое число; уникально в пределах статьи);
- `type` — тип блока (раздел 4.3);
- `payload` — структура полей, специфичная для типа блока (раздел 4.3).

### 4.3. Типы блоков
В первой версии поддерживаются десять типов блоков. Стандартные блоки (1–7) идентичны блокам новостей (`05-content/news/posts.md`, раздел 3.3); блок `file` (8) — новый, специфичный для базы знаний; блоки 9–10 — специализированные.

#### 4.3.1. `heading` — заголовок
Поля `payload`:
- `text` — текст заголовка (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 4.1).

#### 4.3.2. `subheading` — подзаголовок
Поля `payload`:
- `text` — текст подзаголовка (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 4.1).

#### 4.3.3. `paragraph` — абзац
Текстовый абзац в формате markdown. Поддерживает инлайн-форматирование (жирный, курсив), инлайн-ссылки, маркированные и нумерованные списки.

Поля `payload`:
- `markdown` — текст абзаца (строка, обязательное; лимиты — [limits.md](./limits.md), раздел 4.1).

#### 4.3.4. `image` — изображение
Поля `payload`:
- `fileId` — ссылка на файл изображения в файловом хранилище (обязательное; уровень доступа `public`);
- `caption` — подпись под изображением (строка, опциональная);
- `altText` — альтернативный текст для доступности (строка, опциональная).

#### 4.3.5. `gallery` — галерея изображений
Поля `payload`:
- `images` — упорядоченный список изображений (как в `image`); список не может быть пустым.

#### 4.3.6. `video` — видео
Поля `payload`:
- `rutubeUrl` — URL ролика Rutube (строка, обязательное; правила валидации — `05-content/news/posts.md`, раздел 6.1);
- `caption` — подпись под видео (строка, опциональная).

#### 4.3.7. `audio` — аудио
Поля `payload`:
- `fileId` — ссылка на аудиофайл в файловом хранилище (обязательное);
- `episodeTitle` — название аудиозаписи, отображаемое над плеером (строка, опциональная);
- `caption` — подпись под плеером (строка, опциональная);
- `durationSeconds` — длительность в секундах (целое число; заполняется автоматически или вручную при невозможности извлечения).

#### 4.3.8. `file` — прикреплённый файл документа
Прикреплённый файл документа, отображаемый в статье как карточка с иконкой типа файла, отображаемым именем, размером и кнопкой скачивания.

Поля `payload`:
- `fileId` — ссылка на файл в файловом хранилище (обязательное; уровень доступа `public` либо `authenticated` — определяется правилами модуля и фиксируется в сопутствующих правках `07-platform/storage.md`);
- `displayName` — отображаемое имя файла на странице статьи (строка, обязательное; по умолчанию равно имени файла, контент-менеджер может переопределить);
- `description` — короткое описание файла под именем (строка, опциональная; например, «ГОСТ 10700-97 на макулатуру», «Шаблон договора поставки»).

Допустимые форматы файлов:
- PDF;
- DOCX, XLSX, ODT, ODS;
- TXT, CSV;
- ZIP-архивы (по необходимости);
- иные форматы — расширение каталога фиксируется в сопутствующих правках `07-platform/storage.md`, раздел 5.11 и [other-integrations.md](./other-integrations.md), раздел 3.

#### 4.3.9. `checklist` — цветной чек-лист
Цветной чек-лист с заголовком и набором пунктов. Используется для специализированных секций «Принимается», «Не принимается», «Риски», «Нюансы закупки» и иных похожих по структуре.

Поля `payload`:
- `title` — заголовок чек-листа (строка, обязательное; например, «Принимается»);
- `style` — стиль оформления:
  - `positive` — зелёный, для блоков типа «Принимается»;
  - `negative` — красный, для блоков типа «Не принимается»;
  - `warning` — жёлтый, для блоков типа «Риски»;
  - `info` — синий, для блоков типа «Нюансы закупки» и иных нейтральных подсказок;
- `items` — упорядоченный список строк-пунктов; список не может быть пустым; каждый пункт — простая строка без markdown-разметки.

Конкретный набор иконок для каждого стиля (галочка для `positive`, крестик для `negative`, восклицательный знак для `warning`, информационный знак для `info`) — на стороне дизайн-системы.

#### 4.3.10. `image_checklist` — изображение с цветным чек-листом
Карточка с изображением и цветным чек-листом под ним. Используется для визуальных карточек вида «Собран верно» (стиль `positive`) и «Собран неправильно» (стиль `negative`).

Поля `payload`:
- `title` — заголовок карточки (строка, обязательное; например, «Собран верно», «Собран неправильно»);
- `style` — стиль оформления (как в `checklist`: `positive`, `negative`, `warning`, `info`);
- `image` — структура с полями `fileId`, `caption`, `altText` (как в блоке `image`); обязательное;
- `items` — упорядоченный список строк-пунктов; список не может быть пустым.

### 4.4. Поведение при удалении файла блока
Если файл изображения, галереи, аудио или документа (поле `fileId`) удалён из файлового хранилища, при отображении статьи на месте блока выводится плашка «Файл недоступен» с указанием идентификатора блока. Контент-менеджер видит ту же плашку в редакторе и заменяет файл новым.

Для блока `audio` дополнительно отображается сообщение «Аудио временно недоступно». Для блока `file` отображается сообщение «Файл документа недоступен; обратитесь к контент-менеджеру».

## 5. Связь с торговой площадкой и индексами цен
В первой версии модуль базы знаний **не реализует** связь с торговой площадкой (`04-marketplace/`) и индексами цен (`05-content/price-indices/`). Статья базы знаний — самостоятельный справочный материал; не имеет ссылки на номенклатуру платформенного справочника, не отображает текущую цену, не содержит кнопок открытия торговой площадки.

См. [overview.md](./overview.md), раздел 13.7. Поведение в последующих версиях, при включении торговой площадки в продукт, — открытый вопрос.

## 6. Правила валидации перед публикацией

### 6.1. Валидация статьи перед `draft → published`
Статья может быть переведена в `published`, если выполнены условия:
- `title` заполнен;
- `slug` уникален в пределах базы знаний;
- статья содержит хотя бы один блок контента (любого типа);
- все блоки прошли валидацию своих полей (раздел 4.3);
- все ссылки на файлы в блоках указывают на существующие файлы в файловом хранилище;
- глубина статьи в дереве (включая её и предков) не превышает 3 уровней;
- родительская статья (если есть) сама находится в состоянии `published` либо в состоянии `draft`. В первой версии не блокируем публикацию ребёнка под неопубликованным родителем — это даёт гибкость контент-менеджеру; однако при отображении в публичной части ребёнок не показывается, если предок в `draft` (раздел 7).

### 6.2. Повторная публикация
При снятии статьи с публикации (`published → draft`, только администратор) и повторной публикации проверки раздела 6.1 выполняются заново.

## 7. Видимость статьи в публичной части

### 7.1. Условия отображения
Статья отображается в публичной части дерева, если:
- статья в состоянии `published`;
- все её предки в дереве (родитель, родитель родителя) тоже в состоянии `published`;
- состояние компании пользователя позволяет работу с функциональными разделами платформы.

Если хотя бы один предок в состоянии `draft`, статья и её ветвь не отображается в публичной части (даже если сама статья опубликована). После публикации предка ветвь становится видимой.

### 7.2. Опубликованная статья без предка
Корневая статья (категория) не имеет предков, поэтому условие «все предки в `published`» выполнено автоматически.

## 8. Удаление и каскады

### 8.1. Удаление статьи без потомков
- **Контент-менеджер** удаляет статью только в состоянии `draft`, не имеющую ранее опубликованных редакций (`firstPublishedAt` пуст).
- **Администратор** удаляет статью в любом состоянии. При удалении опубликованной статьи открывается диалог с обязательным полем «Причина удаления».

При удалении статьи каскадно удаляются:
- все блоки контента этой статьи;
- ссылки `articleId` подменяются для целевых сущностей (в первой версии таких связей нет, поскольку статьи не ссылаются друг на друга формально).

Файлы изображений, аудио, документов, привязанные через `fileId` блоков, удаляются по правилам файлового хранилища (`07-platform/storage.md`, раздел 7.7).

### 8.2. Удаление статьи с потомками
Если у статьи есть потомки (дочерние виды или подвиды), удаление статьи требует одного из двух действий:
- **Каскадное удаление с потомками** — администратор подтверждает удаление статьи и всех её потомков. Все статьи ветви удаляются;
- **Перемещение потомков** — администратор сначала перемещает потомков в другие ветви или повышает их уровень, затем удаляет родительскую статью.

Платформа отображает диалог: «У статьи {N} потомков. Удалить вместе с потомками или сначала переместить?». Кнопка «Удалить всё» доступна только администратору.

Контент-менеджер не может удалять статью, у которой есть потомки. Чтобы удалить такую статью, необходимо сначала переместить или удалить потомков.

### 8.3. Журналирование
Все удаления опубликованных статей фиксируются в `AdminActionLog` (`07-platform/admin-panel.md`, раздел 16) с указанием актора, времени, идентификатора статьи и причины. Причина обязательна при удалении опубликованных статей администратором; для удаления черновиков контент-менеджером причина опциональна.
