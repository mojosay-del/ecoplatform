## 1. Назначение раздела
Настоящий документ фиксирует точки интеграции модуля «База знаний» со смежными подсистемами платформы: подписками и биллингом, файловым хранилищем, уведомлениями, модерацией, административной панелью и будущим модулем «База документации».

Документ продолжает [overview.md](./overview.md), раздел 13, и развивает каждую интеграцию до уровня конкретных правил поведения.

Документ определяет:
- влияние подписки на доступ к модулю;
- категории файлов в файловом хранилище;
- сопутствующую правку к `07-platform/storage.md` для блока `file`;
- отсутствие специфичных уведомлений модуля в первой версии;
- неприменение модерации к статьям базы знаний;
- размещение раздела в админ-панели;
- связь с будущим модулем «База документации».

## 2. Подписки и биллинг

### 2.1. Доступ к статьям
Все опубликованные статьи доступны при наличии любой действующей подписки платформы (см. [overview.md](./overview.md), раздел 8.1). Гейтинга на уровне отдельных статей или категорий нет.

Это согласовано с моделью подписок (`07-platform/subscriptions-and-billing.md`, раздел 4) и мастер-документом продукта (`00-product/master.md`, раздел 9.2).

### 2.2. Состояния компании
Поведение доступа в состояниях компании, отличных от `active`:

| Состояние компании | Раздел «База знаний» |
|---|---|
| `active` | доступен |
| `demo` | доступен |
| `past_due` | доступен |
| `suspended` | недоступен |
| `archived` | недоступен |
| `blocked` | недоступен |

### 2.3. Биллинговые события
Модуль «База знаний» не порождает событий биллинга. Покупки внутри модуля (например, разовая покупка доступа к одной статье) в первой версии не предусматриваются.

## 3. Файловое хранилище

### 3.1. Категории файлов
Все медиафайлы модуля «База знаний» относятся к категории «медиафайлы контентных модулей» (`07-platform/storage.md`, раздел 5.11). Это касается:
- обложек статей (`coverImage`);
- изображений в блоках `image`;
- изображений в галереях `gallery`;
- изображений в блоках `image_checklist`;
- аудиофайлов в блоках `audio` (категория «аудио контентных модулей» — сопутствующая правка `07-platform/storage.md` из модуля «Новости»; см. `05-content/news/other-integrations.md`, раздел 3.3);
- прикреплённых файлов документов в блоках `file` (раздел 3.3 настоящего документа — сопутствующая правка для расширения категории документов).

Уровни доступа файлов и правила выдачи подписанных ссылок — `07-platform/storage.md`, разделы 4 и 8.

### 3.2. Видеоролики
Видеоролики на стороне платформы не хранятся (см. [overview.md](./overview.md), раздел 14). Платформа сохраняет только URL ролика на стороне Rutube в составе блока `video`.

### 3.3. Документы — сопутствующая правка к `07-platform/storage.md`
Прикреплённые файлы документов (блок `file` — см. [articles.md](./articles.md), раздел 4.3.8) — категория файлов, требующая дополнительного описания в `07-platform/storage.md`. Категория уже частично определена как часть «медиафайлов контентных модулей» (раздел 5.11), но для блока `file` требуется явная фиксация допустимых форматов и максимальных размеров.

**Сопутствующая правка к `07-platform/storage.md`:**

В раздел 5 «Каталог потребителей» добавляется новая подкатегория «Документы контентных модулей» (или расширяется существующая 5.11 включением документов):

- **Применимо к:** платформенный контент модулей «База знаний» и «База документации» (когда последняя будет специфицирована).
- **Уровень доступа:** `public` либо `authenticated` — определяется правилами модуля. По умолчанию `authenticated` для документов базы знаний (доступ требует активной подписки), `public` — для иных применений.
- **Кто загружает:** контент-менеджер платформы.
- **Кто видит:** пользовательский аккаунт компании с активной подпиской — через подписанную ссылку при нажатии «Скачать» в блоке `file` на странице статьи.
- **Лимиты:**
  - размер файла — до 25 МБ (стандартный лимит документов из существующего раздела 6.3 `07-platform/storage.md`);
  - допустимые форматы — PDF, DOCX, XLSX, ODT, ODS, TXT, CSV, ZIP;
  - число блоков `file` на одну статью — до 30 ([limits.md](./limits.md), раздел 4.5);
- **Привязка к сущности:** блок `file` статьи (`KnowledgeBaseContentBlock` с `type = file`).
- **Политика удаления:**
  - при удалении блока `file`, статьи или ветви — файл удаляется по правилам `07-platform/storage.md`, раздел 7.7;
  - при замене файла контент-менеджером — старый файл удаляется, новый загружается как самостоятельный `StoredFile`.

Сопутствующая правка фиксируется отдельно в `07-platform/storage.md` и должна быть применена вместе с настоящим документом.

### 3.4. Удаление файлов при удалении контента
Каскадное удаление файлов при удалении статьи или каскадное удаление ветви выполняется по правилам [articles.md](./articles.md), раздел 8. Поведение хранилища при отложенном удалении — `07-platform/storage.md`, раздел 7.7.

### 3.5. Загрузка файлов через интерфейс контент-менеджера
Загрузка изображений, аудио и документов выполняется контент-менеджером через стандартный компонент загрузки в файловое хранилище (`07-platform/storage.md`, раздел 7.1). Антивирусная проверка обязательна для всех загружаемых файлов (`07-platform/storage.md`, раздел 7.2).

При неуспехе антивирусной проверки файл не сохраняется и связанный блок не публикуется до выбора другого файла.

## 4. Уведомления

### 4.1. Отсутствие специфичных событий в первой версии
Подсистема уведомлений (`07-platform/notifications.md`) в первой версии не порождает событий, специфичных для модуля «База знаний». Каталог событий не расширяется.

В частности, в первой версии не реализованы:
- уведомление о публикации новой статьи;
- уведомление о публикации статьи в избранной категории;
- уведомление пользователя о значимом обновлении статьи, которую он ранее открывал;
- уведомление контент-менеджера об устаревших статьях.

### 4.2. Системные рассылки
Администратор платформы может информировать пользователей о публикации новых статей или значимом обновлении базы знаний через системные рассылки (`07-platform/admin-panel.md`, раздел 17). Это не специфичный для базы знаний механизм, а общая функция админ-панели.

### 4.3. Расширение каталога — открытый вопрос
Добавление специфичных уведомлений модуля «База знаний» в каталог событий — открытый вопрос для последующих версий.

## 5. Модерация

### 5.1. Не применяется
Контент модуля «База знаний» создаётся платформенным сотрудником (контент-менеджером), не пользователями компаний. Поэтому модерация (`07-platform/moderation.md`) к статьям базы знаний не применяется.

Снятие статьи с публикации администратором — административное действие в рамках workflow раздела (см. [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 7.3), не модерационная санкция.

### 5.2. Жалобы пользователей на статьи
В первой версии не предусмотрена форма подачи жалобы на статью базы знаний (например, «фактическая ошибка», «устаревшая информация»). Обратная связь от пользователей собирается через обращения в поддержку (`07-platform/admin-panel.md`, раздел 14) с категорией `other`.

Подключение специализированного механизма обратной связи по статьям базы знаний — открытый вопрос для последующих версий.

## 6. Административная панель

### 6.1. Размещение раздела
Раздел «База знаний» в админ-панели — часть структуры `07-platform/admin-panel.md`. Доступен платформенным сотрудникам с ролями `content_manager` и `admin`. Подробное устройство — [cabinet-content-manager.md](./cabinet-content-manager.md).

### 6.2. Журналирование
Действия контент-менеджера и администратора в части базы знаний частично фиксируются в едином `AdminActionLog` (`07-platform/admin-panel.md`, раздел 16). Перечень фиксируемых событий — [cabinet-content-manager.md](./cabinet-content-manager.md), раздел 9.

### 6.3. Аналитика
Аналитика использования базы знаний (число просмотров статей, популярные категории, средняя глубина чтения) в первой версии не реализуется (`07-platform/admin-panel.md`, раздел 19.5).

## 7. Связь с будущим модулем «База документации»

### 7.1. Архитектурное переиспользование
Модуль «База знаний» — первый модуль платформы с архитектурой «дерево статей с блочным редактором». Та же архитектура предполагается для отдельного модуля «База документации» (`05-content/documentation/`), который будет специфицирован отдельно.

Сходства между двумя модулями:
- иерархия дерева до 3 уровней;
- блочный редактор с теми же стандартными блоками (`heading`, `subheading`, `paragraph`, `image`, `gallery`, `video`, `audio`, `file`);
- двухстадийный workflow `draft` → `published`;
- управление контент-менеджером, удаление опубликованного администратором с фиксацией в `AdminActionLog`;
- независимый URL статьи от иерархии дерева (с редиректами при переименовании);
- связь с подписками — все материалы доступны при любой подписке;
- отсутствие комментариев, лайков, реакций — справочный материал.

### 7.2. Различия
Модуль «База документации» предполагается специфицировать с учётом следующих отличий:
- содержание ориентировано на документы (шаблоны договоров, инструкции, регламенты, ГОСТы) — преобладают блоки `file`;
- специализированные блоки `checklist` и `image_checklist` базы знаний для документации, вероятно, не нужны или будут заменены другими — например, `instruction_step` (нумерованный шаг инструкции с заголовком и текстом);
- категоризация дерева отличается от базы знаний — например, «Договоры», «Инструкции», «Нормативы», «Шаблоны».

Конкретные решения по UI, набору специализированных блоков и иерархии дерева для документации фиксируются при детальной проработке модуля.

### 7.3. На уровне модели данных
На уровне модели данных сущности `KnowledgeBaseArticle` (модуль «База знаний») и аналогичная сущность модуля «База документации» могут быть унифицированы при детальной фиксации в `08-architecture/data-model.md`. Возможные подходы:
- единая сущность `ContentArticle` с полем-дискриминатором (knowledge_base / documentation);
- две независимые сущности с общим набором полей;
- абстрактная база `Article` и две конкретные реализации (`KnowledgeBaseArticle`, `DocumentationArticle`).

Решение принимается при подготовке `08-architecture/data-model.md` после спецификации модуля «База документации».

### 7.4. Модуль «База документации» в первой версии
В первой версии (MVP) модуль «База документации» **не реализуется** и существует как стаб (`05-content/documentation/README.md`). После готовности базы знаний и её эксплуатационной обкатки решения по UI и UX могут быть скорректированы при детальной проработке модуля документации.

В корневом README платформы (`docs/README.md`) модуль документации указан в разделе «05-content/» как стаб с пометкой о будущей реализации.

## 8. Связь с торговой площадкой и индексами цен

### 8.1. В MVP — отсутствует
В первой версии (MVP) модули «Торговая площадка» (`04-marketplace/`) и «Индексы цен» (`05-content/price-indices/`) **не связаны** с модулем «База знаний».

Торговая площадка в первой версии не входит в MVP (см. договорённость с заказчиком при детализации текущего модуля). База знаний — самостоятельный справочный модуль без cross-link на номенклатуры платформенного справочника, индексы цен или активные объявления.

### 8.2. Возможные cross-link в последующих версиях
В последующих версиях, при включении торговой площадки в продукт, возможна интеграция базы знаний с торговой площадкой:
- кнопка «Найти покупателя по этой номенклатуре» на странице статьи о виде сырья;
- автогенерируемая инфо-карта со средней ценой из индекса и прямыми ссылками на торговую площадку;
- встроенный фильтр объявлений на странице статьи.

Эти возможности — открытый вопрос для последующих версий и не реализуются в MVP.
