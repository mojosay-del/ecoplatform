## 1. Назначение документа

Документ фиксирует продуктовое поведение платформы в части работы с адресами, координатами, зонами и расстояниями.

Документ определяет:
- единую модель адреса в платформе;
- источники адресов (компания, объявление, CRM);
- сценарий ввода адреса;
- справочник регионов;
- алгоритм отображения объявления на карте маркетплейса (круг радиусом четыре километра со случайным смещением);
- расчёт расстояний для сортировки маркетплейса;
- фильтр по региону на маркетплейсе;
- использование адресов в CRM.

Картографический провайдер фиксируется отдельно в `08-architecture/maps-provider.md`.

## 2. Модель адреса

### 2.1. Состав полей
Структурированный адрес в платформе содержит следующие поля:

| Поле | Тип | Назначение |
|---|---|---|
| `addressLine` | строка | человекочитаемое полное представление адреса |
| `regionCode` | строка | код региона по справочнику `regions` (раздел 5) |
| `regionName` | строка | название региона для отображения |
| `city` | строка | город или иной населённый пункт |
| `latitude` | число | широта |
| `longitude` | число | долгота |
| `provider` | строка | идентификатор картографического провайдера, выполнившего геокодирование |
| `providerPlaceId` | строка | внутренний идентификатор адресного объекта у провайдера |
| `verified` | булево | признак того, что адрес выбран из подсказок Suggest API |

Все поля заполняются в момент сохранения адреса автоматически по результату обращения к провайдеру. Пользователь не вводит и не редактирует эти поля напрямую — он только выбирает подсказку из списка.

### 2.2. Откуда берутся значения
- `addressLine`, `latitude`, `longitude`, `city`, `providerPlaceId` — из ответа провайдера на выбранную подсказку;
- `regionCode`, `regionName` — определяются сопоставлением региона из ответа провайдера со справочником `regions` (раздел 5.2);
- `provider` — фиксируется адаптером провайдера в момент геокодирования;
- `verified` — устанавливается в `true`, если адрес выбран из подсказок Suggest API.

### 2.3. Признак verified
Признак нужен для будущих версий, в которых может быть допущен приём непроверенных адресов (массовый импорт контрагентов, миграция данных) с последующей пользовательской разметкой. В первой версии все вводы через UI требуют выбора из подсказок (`07-platform/auth-and-onboarding.md`, раздел 4.6), поэтому все UI-адреса имеют `verified: true`. Поле остаётся в модели для совместимости.

### 2.4. Provider и providerPlaceId
Поля используются для:
- идентификации принадлежности `providerPlaceId` к конкретному провайдеру при подключении другого провайдера в будущем;
- повторного обращения к провайдеру для уточнения данных по адресному объекту.

В первой версии при единственном активном провайдере поле `provider` фактически имеет одно значение. Поля остаются в модели для совместимости с заменяемостью провайдера (см. `08-architecture/maps-provider.md`, раздел 7).

## 3. Источники адресов в платформе

### 3.1. Адрес компании
Каждая компания имеет один фактический адрес площадки, заполняемый при регистрации (`07-platform/auth-and-onboarding.md`, разделы 4.2 и 4.6) и редактируемый собственником в профиле компании.

Адрес компании используется:
- как адрес партии по умолчанию при создании объявления (раздел 3.2);
- как точка отсчёта для сортировки маркетплейса по расстоянию (раздел 9).

### 3.2. Адрес партии в объявлении
Каждое объявление содержит собственный адрес партии (`04-marketplace/listings.md`, раздел 8.1). По умолчанию при создании объявления адрес партии заполняется значением адреса компании; автор может изменить его на другой через тот же компонент ввода с автодополнением.

Адрес партии хранится как **snapshot** — независимая копия структурированного адреса, не связанная FK с адресом компании. Подробнее — раздел 12.

### 3.3. Адреса в CRM
В CRM заготовителя адреса используются в:
- карточке поставщика — адреса точек забора (`03-crm/contractors.md`, раздел 1.4);
- карточке покупателя — адрес приёмки (`03-crm/contractors.md`, раздел 1.5);
- карточке сделки закупки в сценарии выездного забора — точка забора (`03-crm/funnel-purchase.md`, раздел 1.3).

Адреса CRM хранятся в той же структурированной модели (раздел 2.1), но в первой версии не используются в гео-функциях. Подробнее — раздел 11.

## 4. Сценарий ввода адреса

### 4.1. Единый компонент
Все формы ввода адресов в платформе (регистрация, профиль компании, объявления, CRM) используют единый компонент. Компонент обращается к Suggest API через абстракцию `MapsProvider.suggestAddress` (`08-architecture/maps-provider.md`, раздел 6.2).

### 4.2. Сценарий
1. Пользователь начинает вводить адрес.
2. Компонент с задержкой отправляет запрос к Suggest API.
3. В выпадающем списке отображаются подсказки.
4. Пользователь выбирает одну подсказку.
5. По выбранной подсказке компонент обращается к `MapsProvider.geocodeBySuggestion` и получает полный структурированный адрес.
6. Платформа сопоставляет регион ответа со справочником `regions` (раздел 5.2) и заполняет `regionCode` и `regionName`.
7. Адрес сохраняется в форме как структурированная сущность.

### 4.3. Что не принимается
- адрес, введённый вручную и не выбранный из подсказок (`07-platform/auth-and-onboarding.md`, раздел 4.6) — форма не сохраняется;
- подсказка, для которой не удалось определить координаты или регион — отклоняется как недопустимый адрес с просьбой выбрать другую подсказку.

### 4.4. Поведение при недоступности геосервиса
При недоступности Suggest API форма ввода адреса блокируется. Пользователь видит сообщение о временных технических проблемах. Регистрация компании, редактирование адреса компании, создание объявления и редактирование контрагентов с изменением адреса в этот момент невозможны.

Поведение при недоступности отдельных API провайдера — `08-architecture/maps-provider.md`, раздел 8.

## 5. Справочник регионов

### 5.1. Назначение справочника
Справочник `regions` — единый платформенный справочник субъектов Российской Федерации. Источник истины для:
- значений в фильтре «регион» на торговой площадке (`04-marketplace/public-ui.md`, раздел 1.4);
- сопоставления адреса с регионом (раздел 5.2);
- агрегаций и аналитики платформы по региональному признаку.

### 5.2. Сопоставление адреса с регионом
В момент сохранения адреса:
1. Из ответа геокодера извлекается название и идентификатор региона.
2. Платформа находит соответствующую запись в справочнике `regions` по нормализованному названию региона или по KLADR/ФИАС-идентификатору, если он включён в справочник.
3. В адрес записываются `regionCode` и `regionName` из найденной записи справочника.

Если регион не удалось сопоставить, адрес отклоняется как непригодный (раздел 4.3). Это редкий случай для адресов в РФ при использовании Яндекс.Геокодера.

### 5.3. Поддержка справочника
Справочник `regions` поддерживается администратором платформы в админ-панели (`07-platform/admin-panel.md`, раздел 13.x — добавляется в рамках текущего этапа документации).

Удаление записей справочника запрещено (только деактивация), что соответствует общему правилу `07-platform/admin-panel.md`, раздел 13.5.

## 6. Координаты и точка партии

### 6.1. Реальная точка партии
Координаты точки партии (`latitude`, `longitude` адреса партии) хранятся в системе и являются скрытой информацией. До момента принятия предложения покупатель не видит ни точки, ни точного адреса (`04-marketplace/listings.md`, раздел 9.1).

### 6.2. Отображаемый центр
В момент публикации объявления платформа вычисляет **отображаемый центр круга** — отдельную пару координат, смещённую от реальной точки на случайную величину (раздел 7). Отображаемый центр сохраняется в объявлении и не пересчитывается на протяжении его жизни.

Структурно объявление содержит две пары координат:
- координаты адреса партии (реальная точка) — скрытые;
- координаты отображаемого центра — видимые.

Все публичные представления объявления на карте используют отображаемый центр.

## 7. Алгоритм круга и случайного смещения

### 7.1. Радиус круга
Радиус круга, отображающего объявление на карте маркетплейса, — **четыре километра** (`04-marketplace/listings.md`, раздел 9.2). Значение фиксировано и не настраивается.

### 7.2. Алгоритм генерации смещения
Смещение отображаемого центра от реальной точки генерируется в момент публикации объявления по следующему алгоритму:

1. Генерируется случайный угол `φ` из равномерного распределения на отрезке `[0, 2π)`.
2. Генерируется случайное значение `u` из равномерного распределения на отрезке `[0, 1]`.
3. Расстояние смещения `d` вычисляется как `d = R · √u`, где `R = 4 км` — радиус круга. Использование `√u` обеспечивает равномерное распределение точки по площади круга, а не по радиусу. Без квадратного корня смещение скашивалось бы к центру, ослабляя скрытие.
4. Смещение пересчитывается в дельты координат с учётом широты:
   - `Δlat = (d · cos φ) / 111.32`
   - `Δlon = (d · sin φ) / (111.32 · cos(latitude))`
   где `d` — в километрах, `latitude` — широта реальной точки в радианах, `111.32` — длина одного градуса широты в километрах (приближение).
5. Координаты отображаемого центра:
   - `displayLatitude = latitude + Δlat`
   - `displayLongitude = longitude + Δlon`

Алгоритм даёт равномерное распределение отображаемого центра по площади круга радиусом четыре километра вокруг реальной точки.

### 7.3. Стабильность смещения
Сгенерированный отображаемый центр сохраняется в объявлении и не меняется в течение его жизни (`04-marketplace/listings.md`, раздел 9.2). Покупатель, периодически открывающий маркетплейс, видит стабильное положение круга.

### 7.4. Поведение при переопубликации
При переопубликации объявления из архива (`04-marketplace/listings.md`, раздел 7.7) создаётся **новое** объявление с новым идентификатором. Алгоритм генерации смещения выполняется заново. Новое объявление получает новый отображаемый центр, не связанный со смещением исходного архивного объявления.

Это поведение защищает от попытки сопоставить смещения нескольких объявлений по одной партии для триангуляции реальной точки.

### 7.5. Поведение при редактировании адреса до первого предложения
До появления первого ценового предложения автор объявления может свободно редактировать его поля (`04-marketplace/listings.md`, раздел 10.2), включая адрес партии. При изменении адреса партии:
- сохраняются новые координаты реальной точки;
- алгоритм генерации смещения выполняется заново;
- объявление получает новый отображаемый центр.

После появления первого предложения прямое редактирование адреса невозможно (`04-marketplace/listings.md`, раздел 10.3); изменение возможно только через переопубликацию (раздел 7.4).

## 8. Раскрытие точного адреса

Точный адрес партии раскрывается покупателю одновременно с раскрытием контактных данных продавца — после принятия продавцом ценового предложения покупателя. До этого момента покупатель видит только круг радиусом четыре километра.

Полная логика раскрытия — `04-marketplace/listings.md`, раздел 9.3.

## 9. Расчёт расстояний

### 9.1. Алгоритм
Расстояние между двумя точками рассчитывается по формуле haversine. Формула вычисляет расстояние по дуге большого круга на сферической модели Земли и не зависит от картографического провайдера.

Формула:
```
a = sin²((lat2 − lat1) / 2) + cos(lat1) · cos(lat2) · sin²((lon2 − lon1) / 2)
c = 2 · atan2(√a, √(1 − a))
distance = R · c
```
где `R = 6371 км` — средний радиус Земли, координаты — в радианах.

### 9.2. От какой точки до какой
Расстояние, используемое в сортировке маркетплейса, рассчитывается от:
- **источника** — координат адреса компании пользователя из его профиля;
- **цели** — координат **отображаемого центра** круга объявления (раздел 6.2), а не реальной точки партии.

Использование отображаемого центра в качестве цели обеспечивает:
- консистентность сортировки с тем, что покупатель видит на карте;
- исключение побочного канала утечки реальной точки партии через сравнение позиций объявлений в сортировке.

### 9.3. Где используется
Расчёт расстояний используется в одной точке первой версии:
- сортировка списка объявлений на торговой площадке по расстоянию от пользователя (`04-marketplace/public-ui.md`, раздел 1.5, значение по умолчанию).

Расстояние не отображается в карточке объявления и не используется в фильтрах.

### 9.4. Что не используется в первой версии
- расчёт расстояния по дорогам через маршрутный API провайдера — не подключён (`08-architecture/maps-provider.md`, раздел 3.4);
- расчёт времени в пути;
- логистические расчёты в CRM (стоимость доставки по расстоянию, оптимальный маршрут вывоза) — раздел 11.

## 10. Фильтр по региону на маркетплейсе

### 10.1. Источник значений
В фильтре «регион» (`04-marketplace/public-ui.md`, раздел 1.4) отображаются значения справочника `regions` (раздел 5). Список фиксирован справочником, не строится по distinct-значениям активных объявлений.

Это даёт детерминированный список регионов независимо от того, есть ли в данный момент активные объявления в каждом из них.

### 10.2. Как объявление попадает в выбранный регион
Объявление попадает в выдачу фильтра тогда и только тогда, когда `regionCode` адреса партии входит в множество выбранных пользователем регионов фильтра.

Фильтр поддерживает множественный выбор регионов (`04-marketplace/public-ui.md`, раздел 1.4). Множество из одного элемента — частный случай той же логики, поэтому она применяется независимо от количества выбранных регионов.

## 11. Адреса в CRM

### 11.1. Хранение
Адреса в CRM (точки забора у поставщика, адрес приёмки у покупателя, точка забора в сделке закупки) хранятся в той же структурированной модели (раздел 2.1) и вводятся через тот же компонент с автодополнением (раздел 4).

Координаты сохраняются как побочный продукт ввода адреса.

### 11.2. Использование в первой версии
В первой версии адреса CRM используются исключительно как информационные поля:
- отображение в карточке контрагента и сделки;
- свободное чтение менеджером для планирования вывоза или приёмки.

Никакие гео-функции в CRM в первой версии не реализуются.

### 11.3. Что не реализуется в первой версии
- карта в интерфейсе CRM (отображение поставщиков, покупателей, точек забора);
- расчёт расстояний от площадки заготовителя до точек забора;
- маршрутизация рейсов выездного забора;
- автоматический подбор оптимального покупателя по расстоянию;
- кластеризация поставщиков по географическим зонам.

Хранение координат как побочного продукта ввода адреса (раздел 11.1) сохраняет возможность подключить эти функции в последующих версиях без миграции данных.

## 12. Адрес партии — snapshot

### 12.1. Принцип snapshot
Адрес партии в объявлении хранится как **независимая копия** структурированного адреса, а не как ссылка на адрес компании. В момент создания объявления значения полей адреса компании копируются в адрес партии; после этого две сущности эволюционируют независимо.

Структурно объявление содержит собственный экземпляр адреса (раздел 2.1), не имеющий FK на адрес компании.

### 12.2. Поведение при изменении адреса компании
Изменение фактического адреса компании через профиль компании **не влияет** на адреса партий уже опубликованных или архивных объявлений. Объявления продолжают отображаться на карте с прежними координатами и кругами.

При создании нового объявления после изменения адреса компании по умолчанию подставляется новое значение адреса. Автор объявления может оставить его или изменить (раздел 3.2).

### 12.3. Поведение при переопубликации
Переопубликация объявления (`04-marketplace/listings.md`, раздел 7.7) создаёт новое объявление с собственным snapshot адреса партии. Перед публикацией автор может отредактировать любое поле, включая адрес партии. После публикации новое объявление имеет независимый snapshot и независимый отображаемый центр (раздел 7.4).

### 12.4. Обоснование решения
Альтернатива «адрес партии как FK на адрес компании» отклонена по причинам:
- изменение адреса компании задним числом меняло бы координаты исторических объявлений в архиве (`04-marketplace/archive.md`, раздел 1.4 — архив без ограничения срока), что искажает историческую запись;
- изменение адреса компании после публикации объявления, но до закрытия его сделки, меняло бы круг и координаты уже видимого покупателям объявления, нарушая консистентность UX;
- модель FK не покрывает кейс `04-marketplace/listings.md`, раздел 8.1 (адрес партии может изначально отличаться от адреса компании, если партия физически вне площадки).

## 13. Ограничения первой версии и открытые вопросы

### 13.1. Только территория РФ
Платформа в первой версии работает только на территории Российской Федерации. Адреса вне РФ не принимаются справочником регионов.

### 13.2. Точечные адреса
Каждый адрес — одна точка с координатами. Полигоны (например, территория крупной промзоны) и линии (адрес «вдоль улицы») не поддерживаются.

### 13.3. Один адрес на компанию
Компания в первой версии имеет один фактический адрес площадки. Сценарий «у компании несколько площадок, каждая со своим набором объявлений и сотрудников» в первой версии не реализуется. Если компания фактически работает с нескольких площадок, она:
- регистрирует одну как основной адрес компании;
- вторую при необходимости указывает как альтернативный адрес партии в объявлении (раздел 3.2).

### 13.5. Открытый вопрос: KLADR/ФИАС-идентификатор в справочнике регионов
В первой версии минимальный состав записи справочника `regions` — код, название, признак активности. Включение KLADR/ФИАС-идентификатора региона упростило бы сопоставление с ответом геокодера и сделало бы его независимым от расхождений в текстовом написании названия региона. Решение — отдельная задача при детализации `07-platform/admin-panel.md`, раздел 13.x.
