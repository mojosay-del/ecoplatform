## 1. Назначение документа

Документ фиксирует архитектурные решения по картографическому провайдеру ЭкоПлатформы:
- выбор провайдера для первой версии и его обоснование
- набор используемых API провайдера
- лимиты, ключи, режим хранения секретов
- контракт абстракции `MapsProvider`, обеспечивающий заменяемость
- поведение платформы при недоступности провайдера

Продуктовое поведение, опирающееся на картографический сервис (модель адреса, алгоритм круга, расчёт расстояний, фильтр по региону), фиксируется в `08-architecture/geo-logic.md`.

## 2. Картографический провайдер первой версии

### 2.1. Решение
Картографический провайдер первой версии — **Яндекс.Карты**.

### 2.2. Обоснование выбора
Ключевые критерии:
- покрытие РФ, в том числе малых городов и промзон. Это критично для рынка вторсырья: типичная площадка заготовителя расположена в складской или промышленной зоне на окраине малого или среднего города. Геокодер Яндекса в этой части стабильно превосходит альтернативы;
- российская юрисдикция, отсутствие санкционных рисков для платформы, обслуживающей B2B-сегмент в РФ;
- зрелый Suggest API для автодополнения адресов, что является обязательным условием формы регистрации (`07-platform/auth-and-onboarding.md`, раздел 4.6: ручной ввод без выбора подсказки не принимается);
- JavaScript API карт промышленного качества с коммерческой лицензией;
- прозрачные коммерческие условия, понятная процедура подключения.

Альтернативы и причины отказа:
- 2ГИС — слабее в малых городах и сёлах, что повышает риск отказа в регистрации компании из-за отсутствия подходящих подсказок;
- OpenStreetMap (Nominatim, MapLibre) — нестабильное качество геокодирования в РФ вне городов-миллионников.

## 3. Используемые API провайдера

### 3.1. JavaScript API карт
Применяется для:
- рендера карты на торговой площадке (`04-marketplace/public-ui.md`, раздел 1.2);
- отображения кругов активных объявлений (`04-marketplace/public-ui.md`, раздел 1.3);
- управления зумом, центром, синхронизации карты со списком и фильтрами.

### 3.2. Suggest API
Применяется для автодополнения адресов во всех формах ввода адреса:
- регистрация компании (`07-platform/auth-and-onboarding.md`, раздел 4.6);
- редактирование фактического адреса компании в профиле;
- ввод адреса партии при создании или редактировании объявления (`04-marketplace/listings.md`, раздел 8.1);
- ввод адресов контрагентов в CRM (`03-crm/contractors.md`, разделы 1.4 и 1.5);
- ввод точки забора в сделке закупки (`03-crm/funnel-purchase.md`, раздел 1.3).

Все формы ввода используют единый компонент, обращающийся к Suggest API через абстракцию (раздел 6).

### 3.3. Геокодер
Применяется для:
- получения координат и структурных полей адреса по выбранной подсказке Suggest API (адрес → координаты + регион + город);
- обратного геокодирования (координаты → адрес) — резервный сценарий, когда координаты сохранены, но текстовое представление утрачено или повреждено.

В штатной работе платформы основная нагрузка приходится на Suggest API; Геокодер используется для обогащения структурными полями адреса в момент сохранения.

### 3.4. API, не используемые в первой версии
- API маршрутов (Routing) — расчёт расстояний по дорогам не выполняется. Расстояние для сортировки маркетплейса считается по формуле haversine на стороне приложения (`08-architecture/geo-logic.md`, раздел 9);
- API поиска организаций;
- панорамы;
- кластеризация маркеров на стороне провайдера (если потребуется при росте числа объявлений — задача последующих версий).

## 4. Лимиты и тарификация

### 4.1. Бесплатные квоты
Бесплатные квоты Яндекс.Карт зависят от типа API и от условий лицензии в момент подключения. Конкретные численные значения квот фиксируются в техническом задании на интеграцию и проверяются актуально на момент подключения, а не принимаются из настоящего документа.

### 4.2. Коммерческие лимиты
При превышении бесплатной квоты подключается платная лицензия. Тарификация и расчёт фиксируются в договоре с провайдером. Решение по выбору тарифного плана и моменту перехода на платную лицензию принимается на основе мониторинга расхода (раздел 4.3).

### 4.3. Подход к мониторингу расхода
Платформа ведёт учёт количества запросов к каждому из используемых API (Suggest, Геокодер, JS API) с разбивкой по типам и периодам. Конкретный механизм учёта — техническое задание.

При приближении к лимиту администратор платформы получает системное уведомление (`07-platform/notifications.md`, категория «системные»).

## 5. Ключи API

### 5.1. Перечень ключей
В первой версии используются:
- ключ JavaScript API карт (frontend);
- ключ Suggest API (frontend; может совпадать с ключом JavaScript API в зависимости от тарифа);
- ключ Геокодера (backend).

### 5.2. Хранение секретов
Backend-ключ Геокодера хранится как секрет окружения, не попадает в клиентский код, не индексируется.

Frontend-ключи (JS API, Suggest) по своей природе доступны клиенту. Их безопасность обеспечивается ограничениями referer и доменов на стороне провайдера (раздел 5.3), а не сокрытием значения.

### 5.3. Привязка к домену и referer
Frontend-ключи Яндекс.Карт привязываются к официальным доменам платформы (production, staging, dev). Запросы с других доменов отклоняются провайдером. Перечень доменов фиксируется в эксплуатационной документации.

### 5.4. Разделение dev/staging/prod
Для каждого окружения используются отдельные ключи (или раздельные квоты в одном ключе с привязкой к разным доменам). Это исключает влияние тестовой нагрузки на продуктивные лимиты.

## 6. Контракт абстракции MapsProvider

### 6.1. Назначение абстракции
Платформа не обращается к API Яндекс.Карт из бизнес-логики напрямую. Все обращения проходят через единый внутренний контракт `MapsProvider`, реализуемый адаптером под конкретного провайдера. Требование вытекает из `08-architecture/tech-stack.md`, раздел 1.7, и обеспечивает заменяемость провайдера без переработки ядра системы.

### 6.2. Перечень методов
Контракт `MapsProvider` в первой версии:

- `suggestAddress(query)` — возвращает список адресных подсказок по введённой пользователем строке. Используется компонентом ввода адреса.
- `geocodeBySuggestion(suggestion)` — возвращает структурированный адрес по выбранной подсказке.
- `reverseGeocode(coordinates)` — возвращает структурированный адрес по координатам. Используется в резервных сценариях.

Контракт оперирует только продуктовыми типами (раздел 6.3) и не содержит вендор-специфичных идентификаторов в публичной части.

### 6.3. Перечень типов
- `Address` — структурированный адрес. Состав полей зафиксирован в `08-architecture/geo-logic.md`, раздел 2.1. Поля `provider` и `providerPlaceId` являются служебными и используются адаптером.
- `Coordinates` — пара (широта, долгота).
- `Suggestion` — подсказка автодополнения с полями: `displayText` (человекочитаемое представление), `providerSuggestionId` (внутренний идентификатор подсказки у провайдера, передаётся обратно в `geocodeBySuggestion`).

## 7. Заменяемость провайдера

### 7.1. Архитектурный принцип
Адаптеры реализуют единый контракт `MapsProvider`. Подключение нового провайдера сводится к написанию нового адаптера и переключению конфигурации, без изменений в бизнес-логике, формах ввода адресов, маркетплейсе и CRM.

### 7.2. Что меняется при замене провайдера
- адаптер провайдера (новая реализация контракта);
- ключи API в эксплуатационной конфигурации;
- frontend-загрузка JS API нового провайдера;
- параметры рендера карты, специфичные для нового провайдера.

### 7.3. Что не меняется при замене провайдера
- модель адреса (`08-architecture/geo-logic.md`, раздел 2.1);
- алгоритм круга и смещения (`08-architecture/geo-logic.md`, раздел 7) — работает на координатах и не зависит от провайдера;
- алгоритм расчёта расстояний (haversine, `08-architecture/geo-logic.md`, раздел 9);
- справочник регионов (`07-platform/admin-panel.md`, раздел 13.x);
- бизнес-логика маркетплейса, CRM, регистрации.

## 8. Поведение при недоступности провайдера

### 8.1. Suggest API недоступен
Форма ввода адреса блокируется. Пользователь видит сообщение о временных технических проблемах с предложением повторить позже. Регистрация компании, редактирование фактического адреса компании, создание объявления с альтернативным адресом партии и ввод адресов контрагентов в CRM в этот момент невозможны (`07-platform/auth-and-onboarding.md`, раздел 4.6: адрес без выбора подсказки не принимается).

### 8.2. Геокодер недоступен
Адрес сохраняется без обогащения структурными полями (`regionCode`, `regionName`, `city`). Запись помечается флагом, требующим повторной обработки. Фоновая задача периодически дообогащает такие записи. Влияние на пользовательский флоу отсутствует — адрес считается сохранённым.

### 8.3. JS API карт недоступен
Карта на маркетплейсе не отображается. Список объявлений (`04-marketplace/public-ui.md`, раздел 1.2) продолжает работать, фильтры и сортировка остаются доступны. Отображается информационное сообщение о временной недоступности карты.

## 9. Ограничения первой версии и открытые вопросы

### 9.1. Один активный провайдер
В первой версии в платформе работает только один активный картографический провайдер. Параллельная работа двух провайдеров (например, fallback при недоступности основного) не предусмотрена.

### 9.2. Маршруты
Маршрутный API не подключён. Расчёт расстояний по дорогам и времени в пути в первой версии не выполняется.

### 9.3. Кастомизация визуала и кластеризация
Используется стандартный визуал Яндекс.Карт. Кастомные стили карты и кластеризация маркеров — задачи последующих версий.

### 9.4. Конкретные численные лимиты
Точные численные значения квот и тарифов на момент создания документа являются предметом эксплуатационной конфигурации, а не продуктового описания. Документ фиксирует подход.
