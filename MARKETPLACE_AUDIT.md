# Торговая площадка — аудит и roadmap правок

> Полная проверка торговой площадки: логика, безопасность/права, расхождения фронт↔БД,
> мусор в схеме, UI/UX. Составлено 2026-06-11 по ветке `feat/marketplace`.
> Площадка строилась «за закрытыми дверьми»: `MARKETPLACE_ENABLED=1` + `MarketplaceFeatureGuard`.

## Как пользоваться этим файлом

- Каждая задача — чекбокс. Отмечай `[x]` по факту выполнения.
- **Исполнитель** проставляет владелец (`Claude` / `Codex`) — поле в каждой задаче.
- Приоритет: 🔴 высокий · 🟡 средний · ⚪ низкий/на вырост.
- Делать **шаг за шагом**, по одному логическому коммиту на задачу (как в `AGENTS.md`).
- После правок: typecheck + lint + relevant unit/integration; широкое влияние → root-проверки.
- Решения владельца по развилкам уже вшиты в формулировки (отмечено «✅ решение владельца»).

## A. Безопасность и приватность

### A1. 🔴 Деанонимизация покупателя через город до акцепта
**✅ Решение владельца:** до акцепта показывать только регион/расстояние, точный город — после принятия.

- [ ] **Исполнитель:** _(назначить)_

**Проблема.** Закрытый аукцион должен скрывать покупателя до акцепта. Но в карточке
предложения продавцу `city` отдаётся в корне DTO ещё до принятия — для условия «цена на
воротах» это точный город покупателя. Город + цена + рейтинг вместе частично
деанонимизируют покупателя до раскрытия контактов.

**Где.**
- [marketplace-offers.helpers.ts:73](apps/api/src/marketplace/services/marketplace-offers.helpers.ts#L73) — `toListingOfferItem`, `city: offer.city` в корне (а не только в `buyerContact`).
- [ListingOffersPanel.tsx:14](apps/web/src/views/marketplace/ListingOffersPanel.tsx#L14) — `offerConditionText` печатает «Доставка к покупателю: {город}».
- DTO `ListingOfferItem` в `packages/shared/src/dto.ts`.

**Действие.**
1. На `Offer` хранить `region` покупателя (или геокодить город ставки) — сейчас оффер хранит только `city: String?`. Минимально: добавить колонку `region` к `Offer`, заполнять из подсказки/геокода при создании ставки.
2. В `toListingOfferItem` до акцепта (`acceptedAt === null`) отдавать `region` (или «город скрыт»), а точный `city` — только внутри `buyerContact` (revealed-ветка).
3. Фронт: `offerConditionText` до акцепта показывает регион/«город скрыт», после — город.
4. Тест в `marketplace.integration.test.ts`: до акцепта в листинге офферов нет точного города покупателя.

**Затрагивает регрессию.** `MakeOfferForm` (ввод города), `MyOffersView` (там город свой — ок).

## B. Логика и баги

### B1. 🟡 Сортировка «Ближе ко мне» не работает — адреса компаний не геокодируются
**✅ Решение владельца:** геокодить адрес компании.

- [ ] **Исполнитель:** _(назначить)_

**Проблема.** Опция сортировки «Ближе ко мне» берёт координаты компании из её
фактического адреса (`factualAddress.latitude/longitude`). Но адреса компаний нигде не
прогоняются через геокодер — координаты всегда `null`. Проверено в dev-БД: 0 из адресов
компаний имеют координаты. Опция почти всегда скрыта/неактивна.

**Где.**
- [MarketplaceView.tsx:81](apps/web/src/views/marketplace/MarketplaceView.tsx#L81) — ждёт `factualAddress.latitude`.
- [billing-company.helpers.ts:40](apps/api/src/billing/billing-company.helpers.ts#L40) — `upsertCompanyAddress` сохраняет адрес без геокода.
- Геокодер: [marketplace-geocoder.service.ts](apps/api/src/marketplace/services/marketplace-geocoder.service.ts) (есть `geocode`).

**Действие.**
1. При сохранении `factualAddress` компании прогонять `formatted` через геокодер и писать `latitude/longitude` (graceful: нет ключа/ошибка → без координат, как в объявлениях).
2. По желанию вынести геокодер из `marketplace/` в общий модуль (его теперь использует и billing).
3. Бэкофилл-скрипт для уже существующих адресов компаний (как `backfill-video-renditions.ts`).
4. Тест: после сохранения адреса с ключом-моком у компании есть координаты.

### B2. 🟡 Причина архива не видна в «Мои объявления»
- [ ] **Исполнитель:** _(назначить)_

**Проблема.** Архивные объявления показывают только бейдж «Архив». Владелец не различает:
продано / истёк срок / снято автором / снято модератором / не договорились. Поле
`archiveReason` уже есть в DTO `MyMarketplaceListingItem`, но в UI не выводится.

**Где.**
- [MyListingsView.tsx:99](apps/web/src/views/marketplace/MyListingsView.tsx#L99) — рисуется только `ListingStatusBadge`.
- `archiveReason` маппится в [marketplace-listings.helpers.ts:87](apps/api/src/marketplace/services/marketplace-listings.helpers.ts#L87).

**Действие.** Под бейджем «Архив» показывать причину: `sold`→«Продано», `expired`→«Истёк
срок», `withdrawn`→«Снято вами», `removed_by_moderator`→«Снято модератором»,
`not_settled`→«Сделка не закрыта в срок». Добавить словарь меток в `listing-ui.tsx`.

---

### B3. ⚪ Переподача не геокодит адрес, если у исходного нет координат
- [ ] **Исполнитель:** _(назначить)_

**Проблема.** `republish` копирует `latitude/longitude` из исходного адреса. Если объявление
создавалось при недоступном геокодере (координат нет), копия тоже без круга на карте —
навсегда.

**Где.** [marketplace-listings.service.ts:365](apps/api/src/marketplace/services/marketplace-listings.service.ts#L365) — `republish`.

**Действие.** При переподаче, если у исходного адреса нет координат, попытаться геокодить
заново (graceful). Низкий приоритет — зависит от B1/общего геокодера.

### B4. 🟡 Возможная гонка при параллельных ставках одного покупателя
- [ ] **Исполнитель:** _(назначить)_

**Проблема.** Правило «один активный оффер на покупателя+объявление» проверяется через
`findFirst` + затем `create` без уникального ограничения/транзакции. Два одновременных
запроса от одного покупателя могут оба пройти проверку и создать две активные ставки.

**Где.** [marketplace-offers.service.ts:77](apps/api/src/marketplace/services/marketplace-offers.service.ts#L77) — `createOffer`, проверка `existing` вне транзакции с `create`.

**Действие.** Либо partial unique index (`buyerCompanyId, listingId` where status in active/accepted),
либо обернуть проверку+создание в транзакцию с сериализацией. Аналогично проверить
`acceptOffer` (там есть проверка `pending`, но тоже вне транзакции — [marketplace-offers.service.ts:236](apps/api/src/marketplace/services/marketplace-offers.service.ts#L236)).

## C. БД и схема (миграции)

### C1. 🟡 Дублирующаяся колонка `packaging`
**✅ Решение владельца:** чистить миграцией сейчас.

- [ ] **Исполнитель:** _(назначить)_

**Проблема.** Упаковка хранится дважды: `MarketplaceListing.packaging` (агрегат) и
`ListingPosition.packaging` (источник истины после миграции `20260611193000`). Сервис
агрегирует позиции обратно в листинг — двойное хранение, размытый источник истины.

**Где.**
- Схема: [schema.prisma:1259](apps/api/prisma/schema.prisma#L1259) (listing) и [schema.prisma:1301](apps/api/prisma/schema.prisma#L1301) (position).
- Агрегация: `listingPackagingForCreate/Update`, `aggregatePositionPackaging` в [marketplace-listings.service.ts:548](apps/api/src/marketplace/services/marketplace-listings.service.ts#L548).
- Чтение: `mapToDetail` использует `listing.packaging` ([helpers:137](apps/api/src/marketplace/services/marketplace-listings.helpers.ts#L137)); фронт `ListingModal` показывает `listing.packaging`.

**Действие.**
1. Перевести отображение упаковки на агрегат позиций (вычислять на лету в маппере), не читать `listing.packaging`.
2. Миграция: `DROP COLUMN "packaging"` у `MarketplaceListing` (данные уже на позициях).
3. Убрать `packaging` из `createListingDtoSchema`/`updateListingDtoSchema` и связанную логику в сервисе.
4. Фронт `ListingFormView`: убрать `existing.packaging` fallback в `parsePackaging`.
5. Тесты: упаковка корректно отображается из позиций; миграция аддитивно-безопасна.

**⚠️ Риск.** Затрагивает create/update/detail/form — проверить регрессию полного цикла.

### C2. 🟡 Мёртвые колонки `moisturePct` / `contaminationPct`
**✅ Решение владельца:** чистить миграцией сейчас.

- [ ] **Исполнитель:** _(назначить)_

**Проблема.** Форма перешла на текстовые состояния (`dry/slightly_wet/wet`,
`clean/may_have_inclusions/has_inclusions`) — `buildDto` шлёт `moisturePct: null`,
`contaminationPct: null`. Числовые колонки не заполняются новыми записями, читаются лишь
как fallback для старых.

**Где.**
- Схема: [schema.prisma:1305](apps/api/prisma/schema.prisma#L1305).
- DTO: `listingPositionInputSchema` ([dto.ts:221](packages/shared/src/dto.ts#L221)) ещё принимает оба pct.
- Fallback-чтение: `ListingModal` (`moistureLabel`/`contaminationLabel`), `ListingFormView` (`moistureConditionFromPct`/`contaminationConditionFromPct`).

**Действие.**
1. Миграция: бэкофилл оставшихся `*Pct` в текстовые `*Condition` (логика порогов уже есть в форме — перенести в SQL, как в `20260611193000`), затем `DROP COLUMN "moisturePct"`, `DROP COLUMN "contaminationPct"`.
2. Убрать оба поля из `listingPositionInputSchema` и из `buildDto`.
3. Убрать fallback-ветки на pct в `ListingModal` и `ListingFormView`.
4. Тесты: влажность/включения показываются из condition.

### C3. 🟡 Адреса-снимки объявлений накапливаются как сироты
- [ ] **Исполнитель:** _(назначить)_

**Проблема.** У объявления свой адрес-снимок (`addressId @unique`). При удалении
объявления (cascade от компании) `Address` остаётся; `republish` плодит новые `Address`.
Нет ретеншена осиротевших адресов — таблица растёт.

**Где.**
- Схема: [schema.prisma:1246](apps/api/prisma/schema.prisma#L1246) (listing→address), [schema.prisma:455](apps/api/prisma/schema.prisma#L455) (Address).
- Cleanup-кроны: [scheduler.service.ts](apps/api/src/scheduler/scheduler.service.ts) (есть `cleanup-orphan-files`, `cleanup-stale-records`).

**Действие.** Добавить в существующий ретеншен-крон удаление `Address`, не привязанных ни к
компании (`companyAsFactual`/`companyAsLegal`), ни к объявлению (`marketplaceListing`).
Связано с прошлым DB-аудитом (ретеншен растущих таблиц).

### D2. 🟡 Стартовый зум карты — объявления почти не видны

**Действие.** Поднять стартовый зум, либо активнее использовать `fitToViewport`/`setBounds`
по точкам при загрузке, либо понизить порог круга. Цель — чтобы при открытии ленты
объявления было видно без ручного зума.

## Что уже хорошо (не трогать без причины)

- **Раскрытие контактов** корректно: телефоны/город сторон отдаются только после акцепта
  (`revealed = acceptedAt !== null`); владелец/админ — через `canSeeContacts`. Разделение
  `isOwner` ≠ `canSeeContacts` сделано верно (админ не видит «Редактировать»).
- **IDOR-защита**: `findOwnedOr404`, `findOwnOfferOr404`, `loadSellerOfferOr404` проверяют
  принадлежность; чужие черновики/архив → 404.
- **Роли**: продавец = только `collector`; покупатель = `trader`/`processor`; нельзя
  предлагать на своё объявление; админ без компании не публикует.
- **Модерация** полиморфная (`marketplace_listing`/`marketplace_review`): снятие объявления
  блокирует переподачу, скрытие отзыва пересчитывает рейтинг; санкция
  `module_restriction` блокирует участие. Офферы намеренно не модерируются (приватные ставки).
- **Кроны** под advisory-lock: авто-архив истёкших, авто-разрешение принятых ставок за 24ч.
- **Файлы**: загрузка валидирует MIME по содержимому; удаление только своих файлов;
  draft-uploads чистятся при уходе с формы.
- **Геокодер** graceful: без ключа/при ошибке объявление сохраняется без круга, список
  остаётся доступен.
