# Ecoplatform Agent Playbook

## Главный принцип

Работай project-aware: перед правкой пойми текущую архитектуру, соседние компоненты, переиспользование и проверки. Цель — сохранять или улучшать качество проекта, а не просто менять видимый фрагмент.

## Старт задачи
- Читай свою память по проекту
- Начинай с текущего состояния проекта: `README.md`, ближайший README/документация по зоне правки, `git status --short` и свежий `git log`.
- Перед изменением ищи влияние через `rg`: импорты, соседние компоненты, API-клиенты, DTO, shared-типы, тесты.
- Считай актуальным то, как проект устроен сейчас: текущий код, текущие README и текущие команды.
- Если код, README и слова пользователя расходятся — остановись и уточни, что считать истиной.

## Архитектура

- Следуй текущим паттернам проекта, не вводи новые библиотеки без сильной причины.
- Frontend: `app/.../page.tsx` держать тонким; page-level UI — в `apps/web/src/views`; переиспользуемые компоненты — в `apps/web/src/components`; не-UI логика — в `apps/web/src/lib`.
- Backend: контроллеры тонкие, бизнес-логика в сервисах и соседних helpers; общие контракты — в `packages/shared`.
- Не дублируй уже существующие компоненты, хуки, форматтеры и API-методы.
- Если меняется контракт API или DTO — обнови обе стороны и релевантные тесты.

## UI и дизайн

- Перед UI-правкой посмотри соседние экраны и существующие компоненты.
- Сначала опирайся на стиль Ecoplatform, внешние референсы используй только точечно.
- После видимой UI-правки проверь страницу в браузере и приложи скриншот, если пользователь не снял это требование.

## Качество

- Валидация входных данных обязательна на границах: формы, HTTP-роуты, webhooks.
- Никаких хардкод-секретов, `console.log` в production-коде и опасного `dangerouslySetInnerHTML` без санитайзера.
- Запускай проверки по масштабу изменения: типы, lint, релевантные unit/integration-тесты; при широком влиянии — root-проверки.
- Не объявляй задачу готовой, если есть упавший typecheck, lint или тесты.

## Документация и git

- README обновляй только если изменились архитектура, запуск, команды, окружение, deploy или публичный workflow.
- Один логический таск — один логический коммит, но `git commit`, `push` и deploy делать только по прямой просьбе пользователя.
- Стейджить файлы явно по именам, не использовать `git add -A`.
