# Этап 3 — Логические ошибки и UX-баги

Покрыто: сквозные сценарии (admin login, demo login, доступ к чужим разделам, неверные ссылки), формы (login/register), состояния UI (loading/error/empty), навигация (404, deep-link), адаптивность (mobile).

Запущены реальные dev-серверы (api :4000, web :3000), проверка через preview-инструменты.

## 🔴 P0 — критичные

### 1. Битая ссылка «Забыли пароль?» в форме логина ✅ DONE 2026-05-24
> Создан `app/forgot-password/page.tsx` — заглушка с инструкцией написать в поддержку (`support@ecoplatform.local`) и ссылкой назад на `/login`. Сам сброс по email ещё не реализован, но пользователь не падает в 404.
- **Где**: [apps/web/src/components/AuthForms.tsx:420](apps/web/src/components/AuthForms.tsx#L420) — `<Link href="/forgot-password">`. Маршрута `/forgot-password` НЕТ ни в `app/`, ни в коде.
- **Что**: ключевой fallback на странице логина (когда пользователь забыл пароль) ведёт на голый Next-404 с английским текстом «404 — This page could not be found.» На прод-сайте это критично — пользователь не может восстановить доступ.
- **Чем чинить**: либо реализовать страницу + back-end эндпоинт восстановления пароля (через email), либо до запуска убрать/заглушить ссылку («Свяжитесь с поддержкой — admin@…»).

### 2. Default Next.js 404 без layout и на английском ✅ DONE 2026-05-24
> Созданы `app/not-found.tsx` и `app/error.tsx` с локализованным русским текстом, ссылками «К новостям» / «Войти» и (для error) кнопкой `reset()`. Проверено в браузере: `/nonexistent-route` показывает русский 404 с навигацией.
- **Где**: отсутствует [apps/web/app/not-found.tsx].
- **Что**: любой битый URL даёт системную страницу с «This page could not be found.» Без сайдбара, шапки, локализации, способа вернуться. Любой пользователь, попавший на старую ссылку или сделавший опечатку, теряется.
- **Чем чинить**: добавить `app/not-found.tsx` с AppShell, русским сообщением, кнопкой «На главную». То же сделать для `app/error.tsx` (рантайм-ошибки) и `app/loading.tsx` (Suspense fallback).

### 3. Реальный сетевой 4xx показывается как «не удалось войти» — серверная диагностика теряется ✅ DONE 2026-05-24
> `apps/web/src/lib/api.ts` теперь распаковывает NestJS-envelope (`{ message, error, statusCode }`) в `ApiError.message`. `AuthForms.tsx` показывает `err.message` (если это `ApiError`) — проверено: «Неверный email или пароль.» вместо JSON-портянки.
- **Где**: [apps/web/src/components/AuthForms.tsx:389](apps/web/src/components/AuthForms.tsx#L389) и [:458](apps/web/src/components/AuthForms.tsx#L458).
  ```ts
  try { await login(...); }
  catch { setError("Не удалось войти..."); }   // пустой catch
  ```
- **Что**: API возвращает осмысленные сообщения («Учётная запись заблокирована», «Доступ к компании закрыт»). Они выкидываются как `ApiError(message, status)` и в форме теряются — пользователь видит обобщённое «Не удалось войти». Заблокированный или истёкший аккаунт получит то же самое, что и человек с опечаткой в пароле.
- **Чем чинить**: `catch (err) { setError(err instanceof ApiError ? err.message : "Не удалось войти..."); }`. На register — то же.

## 🟡 P1 — серьёзные

### 4. Direct URL `/news/<bad-slug>` показывает «Не удалось загрузить данные»
- **Где**: [apps/web/app/news/[slug]/page.tsx], рендер через DataViews → состояние `error`.
- **Что**: при заходе на несуществующую новость пользователь видит общий error-state, а не понятное «Новость не найдена». Не различает «404 на ресурс» и «упал сервер». Это путает поддержку: одно — типичная опечатка, другое — реальный сбой.
- **Чем чинить**: в `useApiData` различать `status === 404` и показывать `NotFound`-вариант.

### 5. На admin-разделах под demo-пользователем нет редиректа
- **Что**: захожу demo-юзером по URL `/admin/users`, страница показывает «Раздел доступен только администратору.», но сам URL остаётся `/admin/users`. F5 — снова та же ошибка. Ничего страшного по безопасности (API возвращает 403), но UX тупиковый: нет кнопки/ссылки уйти.
- **Чем чинить**: в `AccessClosed` добавить кнопку «На главную». Опционально — `router.replace("/")` на стороне клиента после получения 403 в admin-разделах.

### 6. Каждый news-«card» — `<button>` вместо `<a>` ✅ DONE 2026-05-25
> Заменено на `<a href="/news?post=…">` с `onClick.preventDefault()` (модалка открывается без перезагрузки). Ctrl/Cmd/Shift/middle-клик пропускаются — браузер открывает в новой вкладке как обычная ссылка. SEO теперь видит shareable URL.
- **Где**: [DataViews.tsx:255–284](apps/web/src/components/DataViews.tsx#L255).
- **Что**: модалка с `?post=slug` — нормальный паттерн, но плиты сделаны через `<button onClick={openPost(slug)}>` без `<a href>`. Следствия:
  - Cmd-click / правая кнопка → «открыть в новой вкладке» не работают.
  - Скрин-ридер не объявляет элемент как ссылку.
  - Поисковики не индексируют структуру.
- **Чем чинить**: завернуть содержимое в `<a href={"/news?post=" + slug}>`, обработчик клика — превентить дефолт и звать `router.push`. То же для модалок в `knowledge-base`/`education` (надо проверить).

### 7. На странице логина — английский placeholder `name@example.ru` смешан с русским интерфейсом
- **Где**: [AuthForms.tsx:210](apps/web/src/components/AuthForms.tsx#L210).
- **Что**: мелочь, но видна сразу — единственное латинское вкрапление в форме. Лучше `почта@пример.ru` или просто оставить пустым.

### 8. Кнопки «Сменить email» / «Сменить телефон» в /account — `disabled`, но не подписано почему
- **Где**: [DataViews.tsx:2593–2596](apps/web/src/components/DataViews.tsx#L2593).
- **Что**: серая «обещающая» кнопка вызывает желание кликнуть. У disabled-кнопки нет ни tooltip, ни подписи «Скоро». Пользователь решит, что у него что-то не работает.
- **Чем чинить**: либо скрыть до реализации, либо добавить tooltip «Появится в ближайшем обновлении».

### 9. 6 пунктов сайдбара постоянно показаны как `disabled` (Торговая площадка / Форум / Магазин / Документация / Карты / Калькуляторы)
- **Где**: [apps/web/src/components/AppShell.tsx:48, 54, 59, 65, 71, 72](apps/web/src/components/AppShell.tsx#L48).
- **Что**: половина сайдбара — нерабочие пункты. Это нормально для дорожной карты, но для пользователя по умолчанию выглядит «полно битых ссылок».
- **Чем чинить**: либо повесить плашку «скоро» / tooltip; либо сложить в свернутую секцию «Скоро появится».

### 10. На мобильном — огромная высота заголовка «Последние обновления»
- **Что**: на 375×812 заголовок ленты разрастается ≈на 200px, отдавливая первую новость в середину второго экрана. Не баг, но мобильный пользователь видит почти пустую первую «папку».
- **Чем чинить**: media-query на `.news-feed-header h1` со сжатым шрифтом ≤ 28px.

### 11. UI не отображает реальную политику пароля при регистрации ✅ DONE 2026-05-25
> `AuthField` принимает `hint`. На странице регистрации под полем «Пароль» отображается «Не короче 10 символов, минимум одна буква и одна цифра» — текст синхронизирован с `MIN_PASSWORD_LENGTH` из shared, расхождения теперь невозможны.
- **Где**: [AuthForms.tsx:527](apps/web/src/components/AuthForms.tsx#L527) — `<PasswordInput minLength={8}>`.
- **Что**: сервер требует минимум 8 символов + хотя бы одну букву + одну цифру. На клиенте `minLength=8` валидирует только длину. Пользователь вводит `12345678`, жмёт submit → сервер вернёт 400 «Пароль должен содержать букву и цифру», который в форме теряется (см. P0 #3) → «Не удалось зарегистрироваться».
- **Чем чинить**: hint под полем + клиентская валидация pattern.

## 🟢 P2 — улучшения

### 12. Toast/alert-механизма нет — все сообщения inline
- Пароль сменили, сессию отозвали, тикет создан — пользователь видит inline-уведомление в карточке. На длинной странице легко пропустить. Toast в углу хотя бы для success-операций.

### 13. Submit-кнопка disabled на время запроса — defended, но...
- Если запрос «застрял» (плохой канал), кнопка остаётся disabled навсегда. Желательно тайм-аут + кнопка «Повторить».

### 14. `useEffect` на subscribe-токен в `auth.tsx:45–66`
- Два независимых `useEffect` стартуют в одном компоненте, оба читают localStorage и устанавливают токен. Сейчас работает, но при рефакторинге легко получить флешинг состояния.

### 15. Loading-state на каждой странице — белый «прыгающий» layout
- Нет `<Skeleton>`/`<Loading…>`. В переходах между разделами короткий кадр пустоты.

### 16. Иконки в loading нет — для медленного API длинная пустая страница без объяснений.

### 17. У `/news/[slug]` есть отдельная страница, но из ленты открывается модалка
- Двойной паттерн: deep-link → страница, клик из ленты → модалка. UX неконсистентен. Лучше выбрать одно.

### 18. На /education ссылки в админ-сайдбаре «Сырьё», но в основном — «Обучение». Терминологический рассинхрон, проверить с продуктом.

## ✅ Что работает хорошо

- **Логин / регистрация** работает сквозно. Сессия и refresh-cookie создаются корректно. Logout-кнопка реально завершает сессию.
- **Состояния** есть: `unauthenticated`, `forbidden`, `error`, `loading`, `empty`. Не во всех страницах одинаково красиво, но идея единого набора есть.
- **Адаптивность** базово работает: сайдбар сворачивается в гамбургер на mobile (375px).
- **RBAC на фронте**: demo-пользователь не видит «Панель управления», «Модерация» и т.д. в сайдбаре. API всё равно проверяет — это backup, как и должно быть.
- **Уведомления (10 у demo)** видны в шапке, есть popover (по коду).
- **Регистрация** разбита на 3 fieldset с легендами, нормальные labels (`<label>` + `<span>`) — accessibility лучше многих коммерческих форм.
- **Phone-input** правильно нормализует +7XXXXXXXXXX с маской, hidden input с raw-value для отправки.
- **Eye/EyeOff** на password input — стандартная и приятная мелочь.
- **AccountView** имеет 3 таба: профиль / безопасность / уведомления, плюс blocks для сессий, подписки, тикетов — структурно неплохо.
- **Breadcrumb-навигация** есть на каждой главной странице.
- **`isomorphic-dompurify`** реально режет XSS в paragraph-блоках (проверял через `<script>`, не вставляется).
- **dev console чистый** — ни одной React-ошибки или warning при базовой навигации.
- **Сетевые ошибки во время тестов** не дублируются — `apiFetch` корректно прерывает повторные 401-редиректы через `didRedirectOn401`.
