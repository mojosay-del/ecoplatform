# Деплой ЭкоПлатформы на Timeweb — пошагово

Инструкция для **одного** облачного сервера Ubuntu. Всё приложение (сайт + API +
HTTPS) запускается в Docker. База данных и S3 — внешние сервисы Timeweb.

Команды можно копировать как есть. Везде, где встречается `ecoplatform.pro`,
`89.23.96.245`, имя бакета и т.п. — подставьте свои значения.

```
Интернет
   │  https://ecoplatform.pro
   ▼
[ Сервер 89.23.96.245 ]
   ├─ caddy  (80/443) — выдаёт HTTPS, раздаёт трафик:
   │     /api/*  → api  (NestJS, :4000)
   │     всё остальное → web (Next.js, :3000)
   ├─ api    ─┐
   ├─ web     ├─ Docker-контейнеры
   └─ redis  ─┘
        │
        ▼
[ Timeweb PostgreSQL 185.185.143.220 ]   [ Timeweb S3 ]
```

---

## Что понадобится под рукой

- Доступ по SSH к серверу `ecoplatform` (логин/пароль или ключ — из карточки сервера).
- Данные подключения к БД `Fair Quail` (вкладка **«Подключение»** кластера).
- Ключи S3-бакета (раздел S3 → ключи доступа).
- Доступ к управлению доменом `ecoplatform.pro`.
- Доступ к приватному репозиторию на GitHub (Personal Access Token, см. шаг 4).

---

## Шаг 1. Направить домен на сервер (DNS)

В управлении доменом `ecoplatform.pro` добавьте две A-записи:

| Тип | Имя (хост) | Значение |
|-----|------------|----------|
| A   | `@`        | `89.23.96.245` |
| A   | `www`      | `89.23.96.245` |

`@` — это сам `ecoplatform.pro`. Записи «расходятся» по интернету от нескольких
минут до пары часов. Проверить можно так (с любого компьютера):

```bash
ping ecoplatform.pro      # должен показать 89.23.96.245
```

## Шаг 2. Открыть порты сервера

В настройках firewall сервера разрешите **входящие**: `22` (SSH), `80` (HTTP),
`443` (HTTPS). Убедитесь, что порты 80/443 не заняты другим веб-сервером.

## Шаг 3. Подключиться к серверу и поставить Docker

```bash
ssh root@89.23.96.245           # или ваш пользователь

# Docker + Compose (официальный установщик):
curl -fsSL https://get.docker.com | sh
docker --version                # проверка, что встал
```

## Шаг 4. Скачать код на сервер

Репозиторий приватный, поэтому нужен **Personal Access Token** GitHub:
GitHub → Settings → Developer settings → Personal access tokens → Tokens
(classic) → Generate, отметить права `repo`. Скопируйте токен.

```bash
cd ~
git clone https://github.com/mojosay-del/ecoplatform.git
# Логин: mojosay-del   Пароль: вставьте токен (не пароль от GitHub!)
cd ecoplatform
```

## Шаг 5. Закрыть базу от посторонних

Сейчас у БД публичный адрес — это значит достучаться может кто угодно. Во
вкладке **«Сеть»** кластера `Fair Quail` оставьте доступ **только** для IP
вашего сервера `89.23.96.245`, остальное запретите. (Если сервер и БД в одной
приватной сети Timeweb — используйте приватный адрес БД и закройте публичный.)

## Шаг 6. Заполнить секреты

```bash
cp deploy/.env.prod.example deploy/.env.prod
nano deploy/.env.prod           # откроется редактор
```

Заполните все `replace-with-...`. Особое внимание:

- `DATABASE_URL` — из вкладки «Подключение» БД, обязательно с `sslmode=require`.
- `JWT_ACCESS_SECRET` и `JWT_REFRESH_SECRET` — **два разных**, сгенерируйте:
  ```bash
  openssl rand -hex 32          # выполните дважды, вставьте два разных значения
  ```
  ⚠️ Если секреты короче 32 символов — **API не запустится** (это защита).
- `S3_BUCKET` / ключи — из раздела S3.

Сохранить в `nano`: `Ctrl+O`, `Enter`, затем `Ctrl+X`.

## Шаг 7. Запустить

```bash
docker compose -f docker-compose.prod.yml --env-file deploy/.env.prod up -d --build
```

Первая сборка идёт несколько минут (собираются сайт и API). При старте контейнер
`api` **сам применит миграции** базы (`prisma migrate deploy`).

Проверка, что всё поднялось:

```bash
docker compose -f docker-compose.prod.yml ps        # все контейнеры Up
curl -s localhost:4000/api/ready                     # должно быть "ok"/200
```

`/api/ready` проверяет связь с Postgres, Redis и S3 — отличная первая диагностика.

## Шаг 8. Создать обязательные юр-документы (seed)

Без этого регистрация и кабинет блокируются (нет документов для согласий).

```bash
docker compose -f docker-compose.prod.yml exec api pnpm seed
```

Создаёт 5 юр-документов (политика, оферта и т.д.) и временные учётки
`admin@ecoplatform.local` / `demo@ecoplatform.local` (их удалим на шаге 9).

## Шаг 9. Стать первым админом (ваша почта + пароль)

1. Откройте `https://ecoplatform.pro`, **зарегистрируйтесь** обычным способом —
   почту и пароль вы вводите сами (рекомендуется `mojosay@icloud.com`).
2. Назначьте себе права админа и очистите временные учётки. Сначала «вхолостую»
   (ничего не меняет, только покажет, что будет):
   ```bash
   docker compose -f docker-compose.prod.yml exec api \
     pnpm exec ts-node prisma/scripts/promote-first-admin.ts
   ```
   Если в отчёте всё верно — примените:
   ```bash
   docker compose -f docker-compose.prod.yml exec \
     -e PROMOTE_FIRST_ADMIN_WRITE=1 api \
     pnpm exec ts-node prisma/scripts/promote-first-admin.ts
   ```
   ⚠️ Эта команда оставляет в системе **только владельца** и удаляет остальных —
   запускайте её только сейчас, на чистой базе. После появления реальных
   пользователей **не запускать**.

## Шаг 10. Проверить

Откройте `https://ecoplatform.pro` — должен быть зелёный замочек (HTTPS), вход
вашим аккаунтом, админ-раздел доступен.

## Шаг 11. Бэкапы

В кластере `Fair Quail` → вкладка **«Бэкапы»**: у «Физических бэкапов» нажмите
**«Настроить расписание»** и поставьте ежедневное создание. По желанию включите
«Логические бэкапы». Перед каждым крупным обновлением делайте бэкап вручную.

## Как выкатывать обновления в будущем

```bash
cd ~/ecoplatform
git pull
docker compose -f docker-compose.prod.yml --env-file deploy/.env.prod up -d --build
```

Новые миграции БД применятся автоматически при перезапуске `api`.
**Перед обновлениями, меняющими базу, сделайте бэкап** (шаг 11).

## Если что-то не так — смотрим логи

```bash
docker compose -f docker-compose.prod.yml logs -f api     # лог бэкенда
docker compose -f docker-compose.prod.yml logs -f caddy   # лог HTTPS/прокси
docker compose -f docker-compose.prod.yml logs -f web     # лог сайта
```

Частые причины:
- **API не стартует** → почти всегда короткие/пустые `JWT_*` секреты или
  неверный `DATABASE_URL` в `deploy/.env.prod`.
- **Нет HTTPS** → DNS ещё не «доехал» (шаг 1) или закрыты порты 80/443 (шаг 2).
- **Сайт не видит API** → неверный `NEXT_PUBLIC_API_URL` (после его правки нужно
  пересобрать: `... up -d --build`).
- **Пустые согласия / не зарегистрироваться** → не выполнен `seed` (шаг 8).
