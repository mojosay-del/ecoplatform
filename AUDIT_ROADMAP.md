# Roadmap аудита перед бета-тестом

Сквозной аудит проекта от 2026-07-10 (Claude, полный проход: схема БД + живая БД,
бэкенд, фронтенд, CI, деплой-конфигурация, зависимости). Это рабочий файл:
задачи выполняются по одной, исполнитель ставит `[x]` и добавляет короткую
заметку «как сделано» рядом с задачей. Исполнители: **Claude** или **Codex**
(указан рекомендуемый — можно меняться, если владелец решит иначе).

Правила работы по этому файлу:

- Один пункт = один логический коммит (bugfix-workflow проекта).
- После правки прогонять проверки по масштабу: `pnpm typecheck`, `pnpm lint`,
  релевантные тесты; для API-правок — `pnpm test:integration`.
- `push` и deploy — только по прямой просьбе владельца.

## Что проверено и в порядке (не перепроверять без причины)

- **Схема БД**: 66 миграций без дрейфа (`prisma migrate status` чистый);
  все FK покрыты индексами (проверено SQL-запросом по `pg_constraint`);
  имена таблиц/полей читаемые, у неочевидных моделей русские док-комментарии;
  ретеншен-кроны на копящихся таблицах; полнотекстовый поиск форума/документации
  на tsvector+trigram с partial GIN-индексами — образцово.
- **Бэкенд**: валидация `parseBody(zod)` на всех мутациях (инвариант-тест);
  rate-limit трёхуровневый (short/long/auth) на Redis; session-cache в Redis
  с fallback и метриками; санитайзер DOMPurify на сервере; лимит upload 100 МБ;
  advisory-lock на всех cron; метрики за Basic Auth; graceful shutdown;
  идемпотентность ручной активации; `console.log` в prod-коде отсутствует.
- **Фронтенд**: access-токен только в памяти (не localStorage); авто-refresh
  синглтоном; CSRF double-submit; поллинг уведомлений 60с с паузой в фоне;
  robots/sitemap/manifest/error/not-found на месте; security-заголовки + CSP.
- **Тесты локально**: unit 232 (api) + web + shared зелёные; integration
  259/260 (1 падение — задача A1); lint/typecheck чистые.
- **Зависимости**: свежие (Next 16, React 19, Nest 11, Prisma 6.19, Zod 4);
  `pnpm audit --prod` — единственная moderate (js-yaml, транзитивная — задача D2).
- **Нагрузка 1000–2000 пользователей**: архитектура выдерживает с запасом —
  Redis session-cache срезает БД-lookup на каждый запрос, connection pool 20,
  все горячие запросы под составными индексами, списки пагинированы с потолком,
  счётчики батчатся (нет N+1 на лентах), поллинг щадящий. Подтвердить цифрами —
  задача C3 (нагрузочный смоук).

- [x] **C6. Исследование: CSP без `'unsafe-inline'` для script-src** — *Claude* ✅ 2026-07-10
  **Вывод: осознанно откладываем nonce-CSP до post-беты.** Исследовано (context7,
  доки Next 16 CSP). Технически путь есть: `middleware.ts` генерит per-request
  nonce, кладёт в заголовок `x-nonce` + собирает CSP-заголовок с
  `script-src 'nonce-…' 'strict-dynamic'`; Next 16 сам вешает nonce на свои
  hydration/framework-скрипты (читает его из CSP-заголовка запроса). НО цена:
  (1) страницы с nonce ОБЯЗАНЫ рендериться динамически (nonce per-request) —
  теряется статическая оптимизация почти по всему App Router; (2) CSP надо
  выдавать из middleware, а не из `next.config` headers(); (3) плюс сложность
  middleware. Выгода мала: текущая CSP уже режет XSS-векторы (`object-src none`,
  `base-uri self`, `form-action self`, `frame-ancestors none`), а весь
  пользовательский HTML санитизируется server-side (DOMPurify) и нигде не
  отражается сырым — практический риск `unsafe-inline` в script-src низкий.
  Итого: делать ПОСЛЕ беты, когда можно спокойно проверить перф-эффект
  all-dynamic. Путь реализации записан выше — бери как есть.
