## Статус покрытия этого прохода (честно)

Кодовая база большая (~70k LOC: `apps/api` ~29k, `apps/web` ~36k, `packages/shared` ~4k;
70+ Prisma-моделей, 50 миграций, 24 контроллера, 45 web-роутов). Этот первый
проход = **глубоко проверенный фундамент** (auth/RBAC, схема БД, контроллеры, навигация,
god-файлы) + **карта зон**, где нужен системный проход. Поэтому часть пунктов —
конкретные подтверждённые правки, а часть — мета-задачи 🔵 на полный аудит зоны
(их и будем исполнять «шаг за шагом»).

**Хорошая новость:** фундамент зрелый. JWT-гард с кэшем сессий + проверкой
статуса юзера/компании, double-submit CSRF, `RolesGuard`, basic-auth на `/metrics`,
sealed-bid авторизация площадки, ограничения загрузки файлов — всё на месте и
сделано грамотно. Код хорошо прокомментирован. Находки — это в основном полировка,
гигиена БД, декомпозиция god-файлов, UX и продуктовые нюансы прав, а не зияющие дыры.

### Deep-dive проход 2 (2026-06-15) — все зоны 🔵 пройдены

Прогнаны системно: SEC-1 (матрица прав), SEC-5 (файлы), DB-2/3/4 (мусор/индексы/
ретеншен), LOG-1/2/3 (сделка/биллинг/рейтинг), FE-1 (контракт DTO), CQ-1/2 (гигиена/
сборка), UX-1 (живой прогон в браузере collector+admin, desktop+mobile). Сборка
зелёная (typecheck/lint/137 api unit + web unit). **Главное, что НАШЛОСЬ:**

- 🔴 **LOG-6** — `POST /billing/subscriptions` выдаёт платную подписку БЕЗ оплаты
  (план из тела запроса) → весь пейволл обходится. Осознанная заглушка до Тинькофф,
  но **блокер монетизации**.
- 🟡 **SEC-6** — биллинг/профиль компании не ограничены ролью `owner` (member сможет
  менять подписку/реквизиты, когда включат мультиюзер).
- 🟡 **LOG-4** — гонка двойного оффера: «race guard» ловит P2002, которого у `Offer` нет
  (нет `@@unique`) → мёртвая ветка + возможны 2 активных оффера при гонке.
- 🟡 **DB-4** — `AdminActionLog`/`InAppNotification` растут без ретеншена.
- 🟡 **DB-5** — удаление файла сканирует ВСЕ content-блоки (O(таблиц)).
- 🟢 мелочи: мёртвые колонки (`seatsPurchased`, `Payment.provider*`, `logoFileId`),
  `gender` без потребителя (PII), `canAccessBasicContent` дубль-хелпер, FK-индекс
  `DocumentationArticle.fileAssetId`, карта 2ГИС в dev без подложки (UX-3).

**Отлично подтверждено:** пейволл enforced на ВСЕХ чтениях контента + площадке +
модерации (баг past_due закрыт); sealed-bid + accept атомарны; рейтинг пересчитывается
во всех путях вкл. модерацию; контракт фронт↔API типобезопасен; консоль чистая везде.

### [SEC-5] 🔵 Проверка приватности файлов и signed-URL — регресс-проход [ПРОЙДЕНО 2026-06-15]
- [x] **Исполнитель:** Claude (проход выполнен)
- **Где:** `apps/api/src/files/files.controller.ts`, `files.service.ts`.
- **РЕЗУЛЬТАТ:**
  - ✅ `GET /files?ids=` — deny-by-default: не-стафф получает только `public`-файлы
    (`files.service.ts:408-413`); `authenticated`/`platform_private`/`conversation_private`
    не утекают. Применяется ко ВСЕМ id (фильтр в `where`, не «по первому»).
  - ✅ `DELETE /files/:id` — `canDeleteAsset` (`:505`): только `admin` или загрузивший
    (`uploadedById === actor.id`); каждый id проверяется индивидуально.
  - ✅ Загрузка: magic-byte sniffing (`file-type`) + сверка declared/detected MIME +
    блок-лист расширений + лимит размера + дневная квота + `media_only` для не-стаффа.
  - ✅ Обложки: `assertCoverImageAllowed` — только публичная картинка, загруженная
    самим юзером (или admin).
  - ✅ Signed-URL приватных файлов: TTL 1 час, перезапрос урока перепроверяет доступ.
- **ОСТАЁТСЯ (вынесено):** enum доступа имеет 4 уровня (`public/authenticated/
  platform_private/conversation_private`), но не-стафф через общий эндпоинт видит
  только `public`. Значит `authenticated`-файлы (вложения уроков и т.п.) отдаются
  ТОЛЬКО доменными эндпоинтами → **их и надо проверить** (см. LOG-2/FE-1): что урок/
  новость/документация подписывают приватные ссылки лишь после проверки доступа/подписки.
- **Перф-замечание → DB-5:** `hasBlockReference` (`:495`) грузит payload ВСЕХ
  content-блоков (news+lesson+kb) при каждом удалении файла — O(размер таблиц).

### [SEC-6] 🟡 Биллинг и профиль компании НЕ ограничены ролью owner [проверено]
- [x] **Исполнитель:** Codex
- **Где:** `apps/api/src/billing/billing.controller.ts:31-51` + `billing.service.ts`.
- **Проблема:** `PATCH /billing/company` (правка профиля компании) и
  `POST /billing/subscriptions` (активация/смена подписки = деньги) защищены только
  проверкой `user.companyId` — БЕЗ `companyRole === owner`. По модели прав
  (`schema.prisma`: «owner управляет биллингом», «member не управляет») это должны
  мочь только владельцы. `companyRole` в биллинге не используется вообще
  (проверено: используется лишь в auth-deletion, атрибуции отзыва, регистрации).
- **Почему latent:** сейчас компании де-факто одно-пользовательские (все existing
  юзеры = owner, мультиюзер-UI не достроен), поэтому в проде member'ов нет. Но API
  открыт, и при запуске приглашений (CompanyInvitation) это станет дырой:
  рядовой сотрудник сменит подписку/реквизиты компании.
- **Что сделать:** в обоих хендлерах добавить проверку `user.companyRole === "owner"`
  (или общий helper `assertCompanyOwner`). Покрыть тестом.
- **Выполнено (2026-06-15):** `RequestUser` теперь несёт `companyRole`, старый
  session-cache без роли перечитывается из БД; `PATCH /billing/company` и
  `POST /billing/subscriptions` доступны только `CompanyRole.owner`. Добавлены
  integration-тесты `member → 403` для профиля компании и самоактивации подписки.

  без явной команды владельца.

  задача НЕ готова.
- Поведение менять только там, где это и есть задача. Рефактор (волна 4) — без
  изменения поведения.
